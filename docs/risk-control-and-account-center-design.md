# 风控中心与账号中心设计文档

## 1. 设计目标

风控中心与账号中心共同解决一个问题：平台如何安全、稳定、可解释地使用 TG 账号执行运营任务。

账号中心负责账号资产生命周期，风控中心负责账号参与运营时的资格、强度、策略和异常处置。

核心原则：

```text
账号中心管账号本体
风控中心管账号能不能用、怎么用、出问题后怎么处理
任务中心执行前必须询问风控策略
所有拦截、降频、转派、暂停、恢复都必须留痕
```

## 2. 模块定位

### 2.1 账号中心

账号中心是 TG 账号资产中心，负责账号从接入、登录、同步、分组、详情查看到基础状态维护的完整生命周期。

账号中心回答：

- 账号有没有接入？
- 账号是否登录完成？
- 账号 Session 是否可用？
- 账号资料、联系人、群和频道是否已同步？
- 账号属于哪个分组？
- 账号能看到哪些 TG 资产？
- 账号详情和历史记录在哪里查看？

### 2.2 风控中心

风控中心是平台的发送安全和账号运营控制中心，负责全局发送节奏、账号评分、策略分层、任务执行准入、异常账号处置和风控留痕。

风控中心回答：

- 当前系统整体发送是否安全？
- 哪些账号可以参与任务？
- 哪些账号需要降频、冷却、暂停或禁用？
- 达到限额、FloodWait、账号受限、Session 失效时怎么处理？
- 哪些任务、账号、目标被风控拦截或转派？
- 待验证账号和异常账号应该由谁处理？

## 3. 职责边界

| 能力 | 账号中心 | 风控中心 |
| --- | --- | --- |
| 新增 TG 账号 | 负责 | 不负责 |
| 验证码 / 二维码 / 2FA 登录 | 负责 | 只提示阻塞原因和跳转入口 |
| Session 失效处理 | 负责重新登录 | 负责识别风险、暂停参与任务、生成处置项 |
| 账号资料同步 | 负责 | 可把同步异常计入评分 |
| 联系人 / 群 / 频道同步 | 负责 | 可把权限异常计入评分 |
| 账号分组 | 负责 | 可按评分建议移动或标记分组 |
| 单账号代理绑定 | 负责维护 `proxy_id` 与本地代理地址 | 将代理健康、代理复用和代理异常纳入评分 |
| 账号健康分 | 展示基础值 | 负责评分规则、评分原因和策略分层 |
| 发送节奏 | 不负责 | 负责全局策略 |
| 账号小时 / 日限额 | 展示结果 | 负责策略与执行判定 |
| 冷却、降频、转派 | 展示状态 | 负责策略与处置 |
| 已风控账号队列 | 可显示账号状态 | 负责主工作台 |
| 审计留痕 | 展示账号相关记录 | 记录风控命中、动作和策略变更 |

## 4. 总体业务流

```text
新增账号
  ↓
账号中心完成登录
  ↓
账号中心同步资料、联系人、群和频道
  ↓
运营目标中心确认可运营目标
  ↓
任务中心创建任务并选择账号 / 账号组
  ↓
风控中心计算账号评分、策略和可用性
  ↓
任务中心生成执行项
  ↓
执行前再次校验全局风控、账号风控、目标风控和规则中心
  ↓
通过：发送 / 评论 / 浏览 / 点赞
不通过：拦截 / 延后 / 转派 / 跳过 / 暂停
  ↓
风控中心记录命中和处置结果
```

## 5. 账号中心设计

### 5.1 页面结构

```text
TG账号管理
  ├─ 账号列表
  ├─ 添加账号
  ├─ 登录 / 完成登录
  ├─ 账号详情
  ├─ 账号分组
  ├─ 官方验证码
  ├─ 同步记录
  └─ 账号操作记录
```

### 5.2 账号列表

账号列表主要展示账号资产状态，而不是承担风控处置主工作台。

建议字段：

| 字段 | 说明 |
| --- | --- |
| 账号名称 | 展示 display_name、TG 姓名、用户名和手机号 |
| 登录状态 | 未登录、等待验证码、等待扫码、等待 2FA、在线、需重新登录、Session 失效、禁用 |
| 所属分组 | 默认账号池、监听账号组、发送账号组、备用账号组等 |
| 开发者应用 | 账号绑定的 api_id / 应用名 / 应用健康状态 |
| 代理绑定 | 当前绑定的 `proxy_id`、代理类型、本地地址和健康状态 |
| 同步状态 | 资料、联系人、群频道最近同步结果 |
| 可运营资产 | 可发送目标数、可监听目标数 |
| 风控摘要 | 当前评分、风险等级、是否被暂停参与任务 |
| 最近活跃 | 最近成功执行时间 |

### 5.3 添加账号与完成登录

未完成登录的账号不展示复杂操作，只保留一个主动作：完成登录。

登录流程：

```text
新增账号
  ↓
选择验证码登录或二维码登录
  ↓
等待验证码 / 等待扫码
  ↓
如需 2FA，继续输入密码
  ↓
登录成功
  ↓
立即触发资料、联系人、群频道同步
  ↓
账号进入可用或待验证状态
```

### 5.4 账号详情

账号详情用于看清单个账号的资产和历史，不做复杂发送任务编排。

建议 Tab：

- 基础信息：姓名、用户名、手机号、头像、开发者应用、Session 状态。
- 登录与验证：当前登录流、官方验证码、2FA 状态、重新登录入口。
- 代理绑定：当前账号绑定的 `proxy_id`、代理协议、本地地址、端口、最近检测结果。
- 同步记录：资料同步、联系人同步、群频道同步记录和失败原因。
- 资产覆盖：联系人、群、频道、运营目标覆盖情况。
- 风控摘要：评分、等级、最近风险、当前策略、是否可参与任务。
- 执行记录：该账号最近发送、评论、浏览、点赞、监听记录。
- 操作记录：移动分组、资料修改、重新登录、禁用等审计记录。

### 5.5 账号状态

账号中心状态用于描述账号生命周期和基础可用性。

建议状态：

| 状态 | 含义 | 是否可参与任务 |
| --- | --- | --- |
| 待登录 | 已创建但未开始登录 | 否 |
| 等待验证码 | 登录流程等待验证码 | 否 |
| 等待扫码 | 二维码登录等待扫码 | 否 |
| 等待 2FA | 登录流程等待二步验证 | 否 |
| 在线 | Session 可用 | 需要再过风控 |
| 需重新登录 | Session 可能不可用 | 否 |
| Session 失效 | Session 已不可用 | 否 |
| 受限 | TG 返回账号受限 | 否，除非人工恢复 |
| 疑似封禁 | 多次失败或异常 | 否 |
| 已封禁 | 明确不可用 | 否 |
| 禁用 | 平台内人工禁用 | 否 |
| 异常 | 未归类错误 | 否，需处置 |

## 6. 风控中心设计

### 6.1 页面结构

```text
风控中心
  ├─ 风控总览
  ├─ 全局策略
  ├─ 账号评分
  ├─ 处置队列
  ├─ 策略命中记录
  └─ 策略变更审计
```

### 6.2 风控总览

风控总览回答“现在系统能不能继续跑”。

建议指标：

| 指标 | 说明 |
| --- | --- |
| 当前风控等级 | 正常、注意、收紧、暂停 |
| 今日发送 / 互动总量 | 覆盖消息、评论、浏览、点赞 |
| 全局静默状态 | 当前是否处于静默期 |
| 可用账号数 | 在线且通过风控的账号 |
| 降频账号数 | B / C 级账号数量 |
| 阻塞账号数 | 待验证、Session 失效、受限、疑似封禁 |
| 最近 FloodWait | 最近命中次数和账号 |
| 最近转派 | 因风控换号的执行项数量 |
| 最近拦截 | 内容、账号、目标、限速拦截 |
| 待处理处置项 | 需要人工处理的风险事件 |

### 6.3 全局策略

全局策略是所有任务的默认运行边界。

建议配置：

| 策略 | 说明 |
| --- | --- |
| 最小 / 最大发送抖动 | 防止固定间隔发送 |
| 批次间隔 | 批量执行之间的基础间隔 |
| 静默时段 | 全局不主动发送或降低频率的时间段 |
| 是否遵守发送窗口 | 任务和目标是否必须落在允许时间内 |
| 默认重试次数 | 执行失败后的默认重试次数 |
| 默认重试间隔 | 重试基础间隔 |
| 重试退避策略 | 固定、线性、指数 |
| 账号每小时上限 | 单账号跨任务小时上限 |
| 账号每日上限 | 单账号跨任务日上限 |
| 账号全局冷却 | 单账号跨目标冷却 |
| 账号异常处理 | 跳过账号、暂停任务、停止任务 |
| API 限流处理 | 等待重试、跳过、暂停 |
| 内容拦截处理 | 跳过、改写重试、暂停 |

### 6.4 账号评分

账号评分用于把账号从“在线 / 不在线”的粗状态，升级为“可用程度和适合策略”。

评分建议采用 0 到 100 分：

| 评分项 | 影响 |
| --- | --- |
| 登录状态 | 未登录、Session 失效、需重新登录直接进入不可用 |
| 最近成功执行 | 长时间无成功执行降低可信度 |
| 最近失败率 | 连续失败降低评分 |
| FloodWait 次数 | 命中越多，降频越明显 |
| 账号受限 / 疑似封禁 | 直接进入阻塞状态 |
| 目标权限失败 | 降低对应目标能力，不一定影响全局 |
| 同步异常 | 资料或群频道同步失败降低评分 |
| 发送密度 | 接近小时 / 日限额时降低可调度优先级 |
| 代理健康 | 代理不可达、超时、频繁失败或多个高风险账号共用同一出口时降低评分 |
| 内容拦截关联 | 多次发送被拦截内容时降低评分 |
| 角色一致性 | AI 活跃和转发任务中，角色漂移或重复表达降低评分 |

初版评分可以先采用“基础分 + 扣分 / 加分”的可解释模型，不追求复杂机器学习。

建议初版规则：

| 项目 | 分值影响 | 说明 |
| --- | --- | --- |
| 基础分 | 100 | 新账号完成登录、绑定代理、同步成功后进入初始评分 |
| 未登录 / 等待验证码 / 等待扫码 / 等待 2FA | 直接 E | 账号不能参与任务 |
| Session 失效 / 需重新登录 | 直接 E | 生成重新登录处置 |
| 账号受限 / 疑似封禁 / 已封禁 | 直接 E | 暂停参与任务 |
| 未绑定代理 | -40，且正式任务阻塞 | 账号可保留但不能参与发送 |
| 代理不可达 / 认证失败 / disabled | 直接 E | 绑定账号阻塞新执行项 |
| 代理 observing | -10 到 -20 | 只降频，不一定阻塞 |
| 代理复用超限 | -5 到 -15 | 按超限比例扣分 |
| 最近 1 小时 FloodWait 1 次 | -15 | 同时触发账号冷却 |
| 最近 1 小时 FloodWait 多次 | -30 到直接 D/E | 视等待时间和次数决定 |
| 最近 20 次执行失败率超过 30% | -15 | 失败原因需要区分账号、目标、内容和代理 |
| 最近 20 次执行失败率超过 60% | -30 | 进入保守或冷却 |
| 目标权限失败 | 目标维度阻塞，账号全局 -5 | 优先影响目标能力，不直接全局封死 |
| 内容拦截连续 3 次 | -10 | 同时回写规则命中 |
| 角色一致性异常 | -5 到 -15 | 影响 AI 活跃和转发任务选号 |
| 连续成功 10 次 | +5 | 每天最多恢复 10 分 |
| 连续 24 小时无风险 | +5 | 不超过 A/B 等级上限 |

评分恢复规则：

- 硬阻塞解除后不直接恢复到 A，先进入 B 或 C。
- 代理恢复只解除代理阻塞，账号仍按最近执行表现逐步恢复。
- 人工“标记已处理”不直接加分，必须有成功登录、同步、检查或发送证据。
- 分数变更必须记录 `score_reasons`，前端展示最近主要扣分原因。

### 6.5 风控等级与策略

| 等级 | 分数 | 策略 |
| --- | --- | --- |
| A 健康 | 85-100 | 正常参与任务，标准节奏 |
| B 观察 | 70-84 | 可参与任务，轻微降频 |
| C 保守 | 55-69 | 只参与低频任务，降低并发 |
| D 冷却 | 30-54 | 不参与新任务，只保留恢复检查 |
| E 阻塞 | 0-29 | 暂停参与任务，需要人工处置 |

任务选号时默认优先选择 A / B 账号；C 账号只在任务允许保守账号时参与；D / E 不参与新执行项。

### 6.6 处置队列

处置队列是风控中心最重要的运营工作台。

队列类型：

| 类型 | 来源 | 处理动作 |
| --- | --- | --- |
| 待完成登录 | 账号处于待登录、等待验证码、等待扫码、等待 2FA | 去账号中心完成登录 |
| Session 失效 | 发送、同步或健康检查发现 Session 不可用 | 去账号中心重新登录 |
| 账号受限 | TG 返回账号受限 | 暂停账号、设置冷却、替换任务账号 |
| FloodWait 频繁 | 短期多次触发 FloodWait | 降频、冷却、暂停任务 |
| 代理异常 | 本地代理端口不可达、超时、认证失败、出口异常 | 暂停账号、切换代理绑定、触发代理检查 |
| 疑似封禁 | 多次失败且符合封禁特征 | 禁用账号、转人工确认 |
| 目标权限异常 | 账号不能向目标发送或监听 | 处理目标权限验证、替换账号 |
| 待验证任务 | 目标或账号需要验证辅助处理 | 打开验证任务处理 |
| 连续内容拦截 | 账号发送内容多次被规则拦截 | 检查任务内容、账号角色和规则 |

处置动作：

- 立即暂停账号参与任务
- 设置冷却时长
- 降低账号等级
- 触发健康检查
- 触发代理检查
- 修改账号代理绑定
- 重新同步资料或群频道
- 前往账号中心重新登录
- 前往验证任务处理
- 替换运行中任务账号
- 移动到观察分组
- 禁用账号
- 标记已处理

## 7. 待验证处理关系

待验证不是一个独立中心的主线，而是账号中心和风控中心之间的协作状态。

### 7.1 账号登录类验证

包括验证码、二维码、2FA。

归属：

```text
账号中心负责处理
风控中心负责提示不可参与任务
```

表现：

- 账号中心显示完成登录流程。
- 风控中心处置队列显示该账号阻塞原因。
- 任务中心不能选择该账号，或执行前自动转派。

### 7.2 目标权限类验证

包括入群验证、发言权限、频道权限、讨论组权限等。

归属：

```text
账号中心 / 运营目标中心处理具体权限
风控中心汇总风险和处置入口
```

表现：

- 风控中心显示目标权限异常。
- 处置动作可以跳转到验证任务、账号详情或目标详情。
- 处理完成后恢复对应账号在对应目标下的可发送或可监听能力。

### 7.3 Session 与风控事件

Session 失效、账号受限、FloodWait 频繁属于风控事件，不等同于登录验证码。

归属：

```text
风控中心负责判定、暂停和降频
账号中心负责重新登录和恢复 Session
```

## 8. 与任务中心的关系

任务中心不直接决定账号是否安全，只负责创建任务、生成执行项和展示执行结果。

任务中心使用风控中心能力的时机：

```text
创建任务时：提示可用账号数量和容量风险
启动任务时：检查账号池是否满足风控要求
生成执行项时：按账号评分和策略选号
执行前：再次检查账号限额、冷却、状态和目标权限
执行失败后：把失败原因回写给风控中心
恢复任务时：重新计算账号可用性
```

执行策略：

- 账号超过全局限额：优先转派，无法转派则延后。
- 账号处于冷却：优先转派，无法转派则延后。
- 账号 Session 失效：标记账号不可用，生成处置项，任务换号或暂停。
- 账号受限：按全局策略跳过账号、暂停任务或停止任务。
- FloodWait：按策略等待重试、跳过或暂停。
- 内容拦截：交给规则中心结果，按风控策略跳过、改写重试或暂停。

## 9. 与规则中心的关系

规则中心管理内容、路由、账号策略、限速和重试规则的配置版本；风控中心管理这些规则命中后的系统风险状态和处置。

```text
规则中心：定义规则
风控中心：执行风险归因、统计命中、触发处置
任务中心：按规则和风控结果执行
```

举例：

- 敏感词命中：规则中心给出命中原因，风控中心记录内容拦截并决定跳过或改写重试。
- 账号权重策略：规则中心定义账号选择倾向，风控中心根据评分过滤不可用账号。
- 限速策略：规则中心可定义任务或目标级限速，风控中心负责统一展示当前限速压力和处置记录。

## 10. 与运营目标中心的关系

运营目标中心负责目标资产和授权关系，风控中心负责目标相关风险。

目标相关风险包括：

- 目标未授权运营
- 账号无发言权限
- 账号无监听权限
- 群慢速模式
- 目标日上限已达
- 目标冷却中
- 链接白名单不通过
- 最近失败率过高

风控中心可以展示目标风险，但不替代目标中心维护目标资料和授权关系。

主导航建议调整为：

## 12. 单账号代理设计

单账号代理是账号中心和风控中心之间的重要基础能力。它解决的是“每个 TG 账号通过哪个稳定出口访问 Telegram”的问题。

### 12.1 设计边界

服务器上可以运行 Clash、sing-box、v2ray、xray 或其他代理软件。机场订阅、节点列表、节点筛选、节点切换、分流规则都配置在代理软件里。

平台不直接理解机场订阅和机场节点。

平台只认识本地代理地址：

```text
socks5://127.0.0.1:11001
http://127.0.0.1:12001
```

核心边界：

```text
代理软件
  负责订阅、节点、分流、固定出口

TG 运营平台
  只维护 proxy_id、代理协议、本地地址、本地端口、账号绑定和健康状态
```

也就是：

```text
账号 -> proxy_id -> 本地代理端口 -> 代理软件固定路由 -> 机场节点
```

平台不保存机场订阅链接，不展示机场节点名称，不直接切换机场节点，也不把节点作为业务概念暴露给运营人员。

### 12.2 代理配置对象

代理配置建议作为本地代理资源维护。

```text
account_proxies
  id
  name
  protocol
  host
  port
  username
  password_ciphertext
  status
  alert_status
  check_interval_seconds
  timeout_ms
  max_bound_accounts
  max_concurrent_sessions
  last_check_at
  last_error
  disabled_reason
  notes
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| id | 系统内部 `proxy_id` |
| name | 运营可识别名称，例如“账号A出口”“频道互动出口1” |
| protocol | socks5 或 http |
| host | 通常是 127.0.0.1 |
| port | 本地代理端口 |
| username / password | 本地代理如需认证时使用 |
| status | 健康、异常、禁用 |
| alert_status | 代理告警状态，不等同于健康状态 |
| check_interval_seconds | 健康检查间隔 |
| timeout_ms | 健康检查和连接超时阈值 |
| max_bound_accounts | 建议绑定账号上限，用于复用风险提示 |
| max_concurrent_sessions | 建议并发连接上限，用于任务派发保护 |
| notes | 只写本地用途，不写机场节点敏感信息 |

### 12.3 账号绑定方式

账号表或账号扩展表只需要保存 `proxy_id`。

```text
tg_accounts.proxy_id -> account_proxies.id
```

一个账号最多绑定一个当前生效代理；一个代理可以绑定多个账号，但风控中心需要展示代理复用风险。

账号登录、同步、发送、监听、评论、点赞、浏览都应使用账号绑定的代理。

未绑定代理账号的边界：

| 状态 | 允许动作 | 不允许动作 |
| --- | --- | --- |
| 新建草稿账号 | 保存基础资料、分组、备注 | 登录、同步、监听、发送 |
| 已导入但未绑定代理 | 绑定代理、删除、编辑备注 | 登录、同步、监听、发送、加入正式任务 |
| 已登录后代理被解绑 | 查看历史、重新绑定代理、进入处置队列 | 新发送、新监听、新同步、Session 刷新 |

默认原则是：正式连接 Telegram 的动作都必须先有可用 `proxy_id`。如果后续为了迁移旧账号需要短期兼容，可以只允许管理员在迁移模式下完成一次 Session 检查，但不能作为常规路径，也不能裸连。

账号代理绑定的安全规则：

- 代理密码只保存密文，前端不回显明文密码。
- 账号详情页可以展示 `proxy_id`、代理名称、本地端口、健康状态和最近错误，但不展示机场节点或订阅信息。
- 修改账号绑定代理必须写入审计，记录修改前 `proxy_id`、修改后 `proxy_id`、操作者、原因和影响账号数。
- 批量切换代理需要先做预检查：新代理是否可达、绑定账号数是否超出建议上限、是否已有严重告警。
- 正在执行中的任务不直接热切换连接；已创建但未执行的执行项在下一次派发前重新读取账号当前 `proxy_id`。
- 代理被禁用时，绑定账号进入“代理不可用”风险原因，任务中心按策略换号、延后、跳过或暂停。

运行中任务遇到代理切换时：

| 执行项状态 | 处理 |
| --- | --- |
| pending / scheduled | 下一次派发前重新读取账号当前 `proxy_id`，并重新执行预检查 |
| queued / leased 但未创建 Telegram 客户端 | 释放旧租约或继续派发前重新读取 `proxy_id` |
| executing 且已创建 Telegram 客户端 | 不热切换连接；本次按原连接完成或失败回写 |
| failed 且原因是代理异常 | 重试前重新读取账号当前 `proxy_id` |
| paused / delayed | 恢复前重新执行账号、代理、目标和规则检查 |

如果代理切换影响正在运行的任务，系统需要在任务详情展示“代理绑定已变更，待执行项将使用新代理，执行中的连接不热切换”。

代理切换审计要记录：

```text
old_proxy_id
new_proxy_id
change_reason
affected_account_ids
affected_pending_action_count
affected_running_action_count
operator
trace_id
```

### 12.3.1 代理和账号告警主因

同一个账号可能同时出现 Session 失效、代理异常、目标权限失败和内容拦截。处置队列需要有主因，避免同一账号刷出多条重复待办。

主因优先级：

```text
账号封禁 / 账号受限
  ↓
Session 失效 / 需重新登录
  ↓
代理严重告警 / 代理 disabled
  ↓
目标权限异常
  ↓
FloodWait / 冷却 / 限额
  ↓
内容规则拦截
```

合并规则：

- 同一账号同一时间只展示一个主处置项，其他原因作为关联原因展示。
- 同一 `proxy_id` 影响多个账号时，优先生成代理聚合处置项；账号详情显示“受代理告警影响”。
- 如果账号已经 Session 失效，代理异常作为次要原因，不要求运营先处理代理。
- 如果代理严重告警影响多个在线账号，代理告警作为主处置项，绑定账号不再各自重复提醒。
- 处置主因变化时写入风控事件，保留旧主因和新主因。

```text
加载账号
  ↓
读取 proxy_id
  ↓
读取本地代理地址
  ↓
创建 Telegram 客户端
  ↓
通过该代理执行登录 / 同步 / 发送 / 监听 / 互动
```

### 12.4 前端页面

账号中心增加代理相关展示和操作：

- 账号列表展示代理名称、协议、本地端口、健康状态。
- 账号详情展示代理绑定、最近检测、最近错误。
- 添加账号时可以先保存基础资料，但未绑定代理前只能处于草稿 / 待配置状态，不能登录、同步、监听或发送。
- 批量移动代理：支持给多个账号绑定同一个本地代理或分配不同代理。

系统设置或风控中心可以提供“本地代理资源”页面，但它不是日常任务创建入口。没有告警时，运营人员仍然按账号分组 / 账号池选择账号，不需要按代理来选择账号。

```text
本地代理资源
  ├─ 代理列表
  ├─ 新增代理
  ├─ 编辑本地地址
  ├─ 健康检查
  ├─ 绑定账号数
  └─ 最近失败账号
```

页面文案必须避免让运营人员以为系统在管理机场节点。推荐使用“本地代理地址”“本地代理端口”“出口绑定”这类口径。

代理提醒原则：

```text
无告警：按账号分组 / 账号池正常使用
有告警：风控中心提醒处理代理
```

也就是说，代理是账号运行环境，不是运营任务的主要选择维度。只有当本地代理不可达、超时、认证失败、同代理账号异常聚集、同出口 FloodWait 异常增加时，系统才把代理问题推到风控中心处置队列。

### 12.5 代理告警与提醒

代理告警要避免两个极端：普通波动频繁打扰运营，真实出口问题又被淹没在账号异常里。因此代理采用“健康观察 + 告警驱动 + 人工处置闭环”的模式。

建议告警状态：

| 状态 | 含义 | 前端表现 |
| --- | --- | --- |
| normal | 最近检查正常 | 只在账号详情和代理列表展示 |
| observing | 出现轻微失败或延迟上升 | 展示黄色健康提示，不进入处置队列 |
| alerting | 达到告警条件 | 进入风控中心处置队列，并在总览显示待处理 |
| acknowledged | 人工已确认正在处理 | 保留风险，但不重复提醒 |
| disabled | 人工或策略禁用该代理 | 阻塞绑定账号参与新任务 |
| recovered | 连续检查恢复正常 | 从处置队列移到已恢复记录 |
| ignored | 人工确认暂不处理 | 按过期时间自动重新观察 |

建议告警触发条件：

| 类型 | 触发条件 | 默认动作 |
| --- | --- | --- |
| 不可达 | 连续 3 次健康检查连接失败 | 代理进入 `alerting`，绑定账号阻塞新执行项 |
| 超时 | 连续 5 次超过超时阈值，或 10 分钟内超时率超过 50% | 代理进入 `observing`，账号评分下调 |
| 认证失败 | 任意一次明确认证失败 | 代理进入 `alerting`，绑定账号阻塞 |
| 同代理异常聚集 | 15 分钟内同代理下 2 个及以上账号发生连接失败、登录失败或发送失败 | 代理进入 `alerting`，生成聚合处置项 |
| FloodWait 聚集 | 30 分钟内同代理下多个账号出现异常 FloodWait 增长 | 降低该代理绑定账号发送强度，进入观察或告警 |
| 复用超限 | 绑定账号数或并发连接数超过建议上限 | 不一定阻塞，但提示拆分代理或降低并发 |

告警去重规则：

- 同一个 `proxy_id` 在未恢复前只保留一个活跃代理处置项。
- 同代理多个账号异常合并到同一代理处置项中，账号详情显示各自风险原因。
- 代理告警确认后，在确认有效期内不重复弹出提醒，但仍记录新增命中。
- 代理恢复需要连续 3 次健康检查通过，并且最近 10 分钟没有新的连接类错误。
- 代理恢复后，绑定账号不自动恢复到最高评分，只解除代理阻塞，账号评分按最近执行表现逐步恢复。

提醒入口：

- 风控中心总览显示代理告警数量、严重告警、最近恢复。
- 处置队列提供“检查代理”“切换 proxy_id”“禁用代理”“标记处理中”“标记已恢复”动作。
- 账号中心账号列表只显示健康徽标和跳转，不弹出复杂处置。
- 任务中心创建任务时不要求按代理选择账号；只有预检发现代理告警时才提示处理入口。
- 如系统支持管理员通知，可按严重级别发送站内通知或管理员 TG 通知。

### 12.6 风控关系

代理健康进入账号评分，但不要和账号状态混成一个字段。

代理相关风险：

| 风险 | 处理 |
| --- | --- |
| 未绑定代理 | 新账号可提示，正式任务中按策略阻塞或降级 |
| 代理不可达 | 暂停账号参与任务，进入处置队列 |
| 代理超时严重 | 降低账号评分，降低任务参与优先级 |
| 代理认证失败 | 标记代理异常，阻塞绑定账号 |
| 多个高风险账号共用代理 | 提示出口复用风险，建议拆分代理 |
| 同代理下多个账号同时 FloodWait | 提示代理或出口风险，降低该代理绑定账号的发送强度 |

风控中心需要按两个维度看风险：

```text
账号维度：某个账号是否安全
代理维度：某个本地代理出口是否稳定
```

代理异常时，处置动作包括：

- 触发代理健康检查。
- 暂停绑定账号参与任务。
- 给绑定账号设置冷却。
- 切换账号绑定的 `proxy_id`。
- 提示 admin 到服务器代理软件中检查订阅、节点和分流。

没有代理告警时：

- 任务中心按账号分组、账号池或手动账号选择来创建任务。
- 消息发送中心按已选账号过滤可发送目标。
- 风控中心只在账号评分详情里展示代理绑定和健康状态，不主动打断任务。
- 代理维度只做后台健康观察和统计，不作为普通运营人员的日常操作入口。
- 代理只能作为健康徽标、预检原因和后台约束，不能变成任务中心的主选择入口。

### 12.7 发送调用链中的代理

发送调用链需要把代理作为账号运行环境的一部分。

```text
Worker / 派发器加载执行项
  ↓
读取账号
  ↓
读取账号 proxy_id
  ↓
读取本地代理 socks5/http 地址
  ↓
读取账号 session 和开发者应用凭据
  ↓
创建 Telegram 客户端
  ↓
通过本地代理端口连接 Telegram
  ↓
发送消息 / 评论 / 点赞 / 浏览
```

如果代理不可用，不能继续尝试裸连，也不能临时随机换节点。系统只能按策略换绑定的 `proxy_id`、延后、跳过或进入处置队列。

登录和验证流程也必须遵守同一条链路。账号首次登录、验证码验证、扫码、2FA、Session 刷新、账号同步、监听启动和目标权限探测，都必须通过账号绑定代理创建 Telegram 客户端。未绑定代理或代理不可用时，前端应先引导绑定或修复代理，再进入登录 / 验证流程。

### 12.8 代理健康检查任务

代理健康检查由后台定时任务执行，不依赖运营人员手工打开页面。

检查分层：

| 层级 | 检查内容 | 失败含义 |
| --- | --- | --- |
| 本地端口 | `host:port` 是否可连接 | 代理软件未启动、端口错误或本机网络异常 |
| 协议握手 | socks5/http 握手是否成功 | 协议配置错误或认证失败 |
| 认证 | 用户名密码是否可用 | 本地代理认证信息错误 |
| Telegram 连接 | 通过该代理建立 Telegram 连接探测 | 出口不可用、分流异常或代理软件路由异常 |
| 业务回写 | 结合最近发送、同步、监听失败 | 判断代理问题是否影响真实业务 |

检查策略：

- 默认每 1 到 5 分钟检查一次，严重告警期间可以临时提高频率。
- 手动点击“检查代理”应立即触发一次检查，并把结果写入代理告警和审计。
- 检查任务只验证本地代理地址和 Telegram 可达性，不读取机场订阅，不解析节点列表。
- 同一代理健康检查要加互斥锁，避免多个 worker 同时检查同一个 `proxy_id`。
- 检查失败不能立刻删除代理，只更新状态、评分和告警。

健康检查结果：

```text
proxy_health_checks
  id
  proxy_id
  check_type
  status
  latency_ms
  error_code
  error_detail
  checked_at
  trace_id
```

### 12.9 代理容量和分配原则

代理不是业务分组，但需要容量约束，避免多个账号挤在同一个出口上。

容量规则：

- 每个代理可配置建议绑定账号数上限和并发连接上限。
- 超过绑定账号数上限时先提示，不一定阻塞任务。
- 超过并发连接上限时，执行中心应排队、换账号或延后，不能继续压同一代理。
- 同一代理下多个账号短时间集中发送时，风控中心应降低该代理绑定账号的发送强度。
- 代理维度的容量只影响派发和评分，不改变运营创建任务时的账号分组方式。

推荐默认：

| 场景 | 默认处理 |
| --- | --- |
| 新账号未绑定代理 | 可以保存账号，但不能参与正式发送 |
| 代理绑定账号过多 | 风控提示拆分，不直接阻塞 |
| 代理并发超限 | 执行项排队或延后 |
| 代理严重告警 | 绑定账号阻塞新执行项 |
| 代理恢复 | 解除阻塞，账号评分逐步恢复 |

### 12.10 代理告警通知规则

代理告警需要有提醒，但不能刷屏。

通知渠道：

| 渠道 | 用途 |
| --- | --- |
| 风控中心总览 | 值班人员查看当前风险 |
| 主导航角标 | 提醒有待处理严重告警 |
| 站内通知 | 告知代理不可达、认证失败、严重聚集异常 |
| 管理员 TG 通知 | 可选，仅用于 critical 告警 |
| 审计记录 | 保留确认、忽略、恢复和禁用过程 |

通知规则：

- `observing` 只在页面展示，不主动推送。
- `alerting` 生成站内通知，并进入风控中心待处理。
- `critical` 可触发管理员 TG 通知，但同一 `proxy_id` 在 30 分钟内只通知一次。
- `acknowledged` 后不再重复提醒，但新增失败仍然写入命中记录。
- `ignored` 必须设置过期时间，到期后重新进入观察。
- `recovered` 只做恢复通知和审计，不要求人工再次处理。

通知文案必须使用本地代理口径，例如：

```text
本地代理 proxy_12 连续不可达，已影响 4 个账号。请检查服务器代理软件或切换账号绑定代理。
```

不要出现机场订阅、机场节点、地区节点等平台不理解的概念。

### 12.11 同级 admin 与安全

系统后续只保留少量同级 admin，不设计 RBAC，不区分管理员、运维、高级运维等角色。所有 admin 登录后拥有相同操作能力，风险控制依靠高风险动作保护、原因填写和审计追踪完成。

admin 操作原则：

- 所有 admin 权限一致，不做角色分层。
- 高风险动作必须二次确认。
- 高风险动作必须填写原因。
- 高风险动作必须写入审计。
- 前端不展示“角色”“权限组”“运维角色”等概念。

高风险动作建议：

| 操作 | 保护要求 |
| --- | --- |
| 新增 / 编辑代理 | 写入审计 |
| 查看代理密码 | 不允许查看明文 |
| 重置代理密码 | 二次确认，必须填写原因，写入审计 |
| 批量切换账号代理 | 二次确认，展示影响账号和执行项，必须填写原因 |
| 禁用代理 | 二次确认，展示影响账号、运行中任务和待执行项 |
| 忽略代理告警 | 必须设置过期时间和原因 |
| 标记代理恢复 | 建议先触发一次健康检查 |
| 清理风控数据 | 二次确认，写入审计，不能删除审计记录 |

安全要求：

- 代理密码只存密文，只允许重置，不允许回显。
- 导出账号或代理数据时不导出代理密码密文。
- 代理绑定变更要记录变更前后值、影响账号、操作者和原因。
- 高风险批量操作前要展示影响范围，例如影响账号数、运行中任务数和待执行项数。
- 被禁用代理不能被新账号绑定，也不能被任务执行器选用。

## 14. 功能流程设计

### 14.1 账号接入流程

```text
账号中心点击新增账号
  ↓
录入手机号、名称、分组
  ↓
选择登录方式
  ↓
验证码 / 二维码 / 2FA 完成
  ↓
登录成功后立即同步资料
  ↓
同步联系人、群、频道
  ↓
生成账号基础健康状态
  ↓
风控中心生成初始评分
  ↓
绑定或确认账号代理
  ↓
账号可被任务选择
```

失败分支：

| 失败点 | 前端提示 | 后续入口 |
| --- | --- | --- |
| 验证码等待 | 账号未完成登录 | 继续完成登录 |
| 2FA 等待 | 账号未完成登录 | 输入二步验证密码 |
| Session 写入失败 | 登录失败 | 重新登录 |
| 同步失败 | 账号已登录但资料同步异常 | 重新同步 |
| 代理未配置 | 账号缺少本地代理绑定 | 绑定 `proxy_id` |
| 代理不可达 | 本地代理端口异常 | 检查服务器代理软件或切换 `proxy_id` |
| 无可用开发者应用 | 不能新增账号 | 去系统设置配置开发者应用 |

### 14.2 账号进入任务流程

账号不是直接拿来发送，而是必须经过筛选。

```text
任务创建页选择任务类型
  ↓
选择账号来源
  - 全部可用账号
  - 指定账号分组
  - 手动选择账号
  ↓
前端请求账号候选列表
  ↓
后端按账号中心状态过滤
  ↓
风控中心按评分和策略过滤
  ↓
运营目标中心按目标权限过滤
  ↓
前端展示可用账号、降频账号、不可用账号及原因
  ↓
用户确认账号池
  ↓
任务保存或启动
```

账号选择页面必须把“不可用原因”提前显示出来，不能等到发送失败才提示。

账号展示分组：

```text
推荐可用
  - A / B 等级
  - 在线
  - 目标权限通过
  - 未超限

谨慎使用
  - C 等级
  - 可用但需要降频
  - 接近小时 / 日上限

不可用
  - 未登录
  - Session 失效
  - 等待验证
  - 冷却中
  - 受限 / 疑似封禁
  - 无目标权限
```

### 14.3 普通消息发送流程

消息发送中心用于人工直接发送。它也必须走风控，不允许绕过。

```text
消息发送中心
  ↓
选择发送类型：私聊 / 群 / 频道 / 批量
  ↓
选择发送账号
  ↓
前端按账号过滤可发送目标
  ↓
选择目标
  ↓
输入内容和素材
  ↓
点击发送 / 定时发送
  ↓
发送前预检查
  - 账号状态
  - 代理绑定与代理健康
  - 风控评分
  - 账号限额
  - 账号冷却
  - 目标权限
  - 目标风控
  - 内容规则
  ↓
通过：创建发送任务或直接进入队列
不通过：显示阻塞原因和处理入口
```

前端交互重点：

- 先选账号，再按账号过滤目标。
- A 账号只能看到 A 账号可发送的群、频道、联系人。
- 多账号批量发送时，目标列表要显示每个账号是否可达。
- 内容输入后可以提供“发送前检查”按钮。
- 检查不通过时，不允许提交真实发送。

### 14.4 自动任务发送流程

任务中心用于自动化任务，发送动作不应该由前端直接调用 TG API，而是生成执行项交给 Worker。

```text
任务中心创建任务
  ↓
选择任务类型
  ↓
选择目标范围
  ↓

  ↓
配置节奏、规则、内容、时间窗
  ↓
风控预估容量
  ↓
启动任务
  ↓
后端生成 actions / 执行项
  ↓
Worker 获取到期执行项
  ↓
执行前风控校验
  ↓
通过后调用 Telegram 网关发送
  ↓
写入执行结果
  ↓
回写账号评分、任务统计和风控事件
```

自动任务的账号选择由后端最终决定，前端只负责配置账号范围和展示预估。

## 15. 账号使用与发送调用设计

### 15.1 账号使用总原则

账号使用不能只看“在线”，必须同时满足账号状态、代理健康、风控策略、目标权限和规则内容检查。

```text
账号中心状态通过
  - 登录状态
  - Session
  - 开发者应用
  - 代理绑定与代理健康
  ↓
风控中心策略通过
  ↓
运营目标权限通过
  ↓
规则中心内容和任务规则通过
  ↓

| 层级 | 检查内容 | 不通过动作 |

| 风控中心 | 评分、等级、限额、冷却、受限、FloodWait | 降频、延后、转派、暂停 |
| 运营目标 | 账号是否能向目标发送、监听或互动 | 过滤目标、换号、生成验证任务 |
| 规则中心 | 内容、链接、敏感词、路由、限速、重试 | 拦截、改写、跳过、暂停 |

```

### 15.3 发送前预检查接口

建议前端在真正提交前调用预检查能力。预检查接口由风控中心提供，消息发送中心和任务中心调用；它不创建任务，也不直接发送，只返回当前请求是否具备发送资格和需要处理的原因。

```text
POST /api/risk-control/preflight
```

请求：

```text
scenario: manual_send | task_start | action_dispatch
account_ids
proxy_ids
target_ids
content_preview
task_type
scheduled_at
```

返回：

```text
decision: allow | warn | block
available_accounts
limited_accounts
blocked_accounts
proxy_decisions
proxy_warnings
proxy_alerts
target_warnings
content_warnings
suggested_actions
decision_reasons
expires_at
trace_id
```

建议返回字段说明：

| 字段 | 说明 |
| --- | --- |
| decision | 本次请求整体结论，`block` 时前端不能继续提交 |
| available_accounts | 可参与发送的账号，以及对应评分、代理、目标权限摘要 |
| limited_accounts | 可谨慎使用的账号，以及限额、冷却、代理观察原因 |
| blocked_accounts | 不可使用账号，以及阻塞原因和处理入口 |
| proxy_decisions | 每个 `proxy_id` 的健康、告警状态、是否阻塞、建议动作 |
| proxy_warnings | 代理观察类提示，不一定阻塞发送 |
| proxy_alerts | 已进入处置队列的代理告警 ID 和严重级别 |
| target_warnings | 目标权限、目标风控、目标冷却提示 |
| content_warnings | 规则中心返回的内容过滤、改写、阻断提示 |
| suggested_actions | 前端可展示的下一步动作，例如换账号、检查代理、改为定时发送 |
| decision_reasons | 机器可读原因码，用于审计和执行记录 |
| expires_at | 预检查结论有效期，过期后执行前必须重新检查 |
| trace_id | 串联前端预检、任务创建、执行校验和审计记录 |

前端处理：

| decision | 前端动作 |
| --- | --- |
| allow | 允许提交 |
| warn | 显示风险提示，允许继续或调整 |
| block | 禁止提交，显示处理入口 |

代理相关原因码建议：

```text
proxy_missing
proxy_unreachable
proxy_timeout
proxy_auth_failed
proxy_alert_active
proxy_reuse_over_limit
proxy_concurrency_over_limit
proxy_floodwait_cluster
```

预检查和执行前校验的关系：

- 前端预检查用于减少无效提交和提前给运营处理入口。
- Worker / 派发器执行前必须重新检查一次，不能只信任前端预检结果。
- 预检通过不代表一定立即发送成功；执行前账号、代理、目标和规则状态变化时，以执行前结果为准。
- 预检被代理阻塞时，前端进入代理处置入口，不允许绕过代理直接连接 Telegram。

### 15.4 策略优先级和接口清单

多个中心同时给出结论时，执行中心按以下优先级处理：

```text
硬阻塞事实
  账号未登录 / Session 失效 / 代理严重告警 / 目标无权限 / 账号禁用
  ↓
全局风控硬策略
  静默期 / 全局暂停 / 账号 E 级阻塞 / 代理 disabled
  ↓
规则中心内容硬拦截
  内容禁止发送 / 链接不允许 / 输出校验失败
  ↓
限额与冷却
  账号小时日上限 / 目标冷却 / 代理并发上限
  ↓
任务级偏好
  账号池 / 权重 / 目标粘性 / 任务重试策略
```

原则：

- 硬阻塞不能被任务级配置覆盖。
- 规则中心可以建议账号权重，但不能让被风控阻塞的账号发送。
- 任务级重试策略不能绕过代理、账号状态和目标权限。
- 人工确认告警只减少提醒，不代表解除发送阻塞；是否解除由告警状态和策略决定。
- 手动发送和自动任务走同一套执行前校验，只是前端入口不同。

建议接口：

| 接口 | 用途 |
| --- | --- |
| `GET /api/risk-control/overview` | 风控总览、待处理数量、代理告警数量 |
| `GET /api/risk-control/accounts` | 账号评分列表 |
| `GET /api/risk-control/accounts/{account_id}` | 账号风控详情 |
| `POST /api/risk-control/preflight` | 发送或任务启动前预检查 |
| `GET /api/risk-control/events` | 风控命中记录 |
| `GET /api/risk-control/dispositions` | 处置队列 |
| `POST /api/risk-control/dispositions/{id}/handle` | 处置动作提交 |
| `GET /api/account-proxies` | 本地代理资源列表 |
| `POST /api/account-proxies` | 新增本地代理资源 |
| `PATCH /api/account-proxies/{proxy_id}` | 编辑本地代理资源 |
| `POST /api/account-proxies/{proxy_id}/check` | 手动触发健康检查 |
| `POST /api/account-proxies/{proxy_id}/disable` | 禁用代理 |
| `POST /api/accounts/{account_id}/proxy-binding` | 单账号绑定或切换代理 |
| `POST /api/accounts/proxy-bindings/batch` | 批量切换账号代理 |

### 15.7 账号转派设计

账号转派用于避免一个账号被限额、冷却或异常拖死整个任务。

```text
执行项原账号 A
  ↓
风控检查不通过
  ↓
在同一任务账号池里找替换账号
  ↓
候选账号必须满足：
  - 在线
  - 未禁用
  - 风控等级允许
  - 未超小时 / 日上限
  - 不在冷却中
  - 对目标有发送权限
  - 绑定代理健康
  - 角色和行为一致性允许
  ↓
找到 B：执行项改派给 B
找不到：执行项延后或任务暂停
```

前端展示：

```text
原账号：A
系统动作：已转派
转派账号：B
转派原因：A 账号全局冷却中
```

### 15.8 账号角色一致性

AI 活跃群和转发监听任务里，账号不是随便替换的。

转派时需要检查：

- 账号角色是否一致。
- 账号历史语气是否接近。
- 同一轮对话中是否已经发过言。
- 是否会造成前后人设冲突。
- 是否会让同一账号短时间重复表达。

如果找不到角色一致的替换账号，优先延后该执行项，而不是随便换一个账号发送。

### 15.9 发送结果回写

每次调用发送后都必须回写四类结果：

```text
执行项结果
  - success / failed / skipped / pending
  - remote_message_id
  - failure_type
  - failure_detail

账号状态
  - last_active_at
  - health_score
  - status
  - cooldown_until

风控事件
  - 是否命中限额
  - 是否 FloodWait
  - 是否账号受限
  - 是否目标权限失败
  - 是否内容拦截

任务统计
  - 完成数
  - 失败数
  - 跳过数
  - 转派数
```

前端任务详情和风控中心都必须能看到这些结果。

### 16.11 数据保留和清理

风控和代理数据会快速增长，需要在设计阶段明确保留周期。

建议保留策略：

| 数据 | 明细保留 | 聚合保留 | 说明 |
| --- | --- | --- | --- |
| `risk_events` | 180 天 | 2 年 | 风控命中和执行原因，支持运营复盘 |
| `risk_disposition_tasks` | 2 年 | 2 年 | 人工处置记录，保留较长时间 |
| `proxy_alerts` | 1 年 | 2 年 | 代理异常和恢复历史 |
| `proxy_health_checks` | 30 天 | 1 年 | 高频检查明细只短期保留，按小时 / 天聚合 |
| `risk_notifications` | 90 天 | 1 年 | 通知明细主要用于已读、去重和追踪 |
| `account_proxy_bindings` | 2 年 | 2 年 | 代理绑定历史用于追溯发送出口 |
| 审计记录 | 长期保留 | 长期保留 | 敏感操作审计不随风控明细清理 |

清理规则：

- 清理任务只能删除明细，不能删除审计记录。
- 清理前先生成日级聚合，保留风险趋势、告警次数、影响账号数和恢复耗时。
- 正在处理中的处置任务、未恢复代理告警和最近 7 天内关联执行失败的记录不能清理。
- 清理任务本身也要写入审计，记录清理范围、数量和执行人 / 系统任务。
- 导出数据时默认不包含代理密码密文和敏感凭据。

## 17. 状态联动规则

| 触发事件 | 账号中心状态 | 风控中心动作 | 任务中心动作 |
| --- | --- | --- | --- |
| 登录未完成 | 等待验证码 / 等待扫码 / 等待 2FA | 生成阻塞项 | 不允许选用 |
| Session 失效 | Session 失效 / 需重新登录 | 评分降为 E，生成重新登录处置 | 换号或暂停 |
| FloodWait | 在线 | 降频、冷却、记录事件 | 延后、重试或换号 |
| 账号受限 | 受限 | 暂停账号参与任务 | 换号、暂停或停止 |
| 疑似封禁 | 疑似封禁 | 进入人工处置 | 不再分配执行项 |
| 目标无权限 | 在线 | 生成目标权限异常 | 换号或跳过目标 |
| 代理不可达 | 在线 / 异常 | 标记代理异常，绑定账号降级或阻塞 | 换代理、延后或暂停 |
| 代理观察告警 | 在线 | 降低评分，展示观察原因 | 低频执行或换号 |
| 代理严重告警 | 在线 / 异常 | 生成代理处置项，阻塞绑定账号新执行项 | 换账号、延后或暂停 |
| 代理告警已确认 | 在线 / 异常 | 保留风险，不重复提醒 | 按确认后的策略执行 |
| 代理恢复 | 在线 | 解除代理阻塞，评分逐步恢复 | 重新纳入候选账号 |
| 内容规则拦截 | 在线 | 记录内容风险 | 跳过、改写重试或暂停 |
| 连续成功恢复 | 在线 | 提升评分或解除冷却 | 可重新参与任务 |

## 18. 实施分期

### P0：先把中心做清楚

- 新增风控中心一级导航。
- 从运营数据中拆出全局风控指标和策略编辑。
- 账号列表增加风控摘要，但不把账号中心变成风控工作台。
- 处置队列先汇总现有状态：Session 失效、受限、疑似封禁、待验证、连续失败。
- 消息发送中心先接入发送前预检查，确保账号和目标提前过滤。
- 账号中心补充代理绑定展示和 `proxy_id` 维护口径。
- 补齐代理告警状态、告警去重和代理处置入口，先以人工处理为主。
- 预检查接口返回账号、代理、目标、内容四类决策结果和原因码。
- 补齐本地代理资源、代理告警、处置队列和账号绑定代理的基础接口。
- 代理密码密文存储，前端不回显明文。

### P1：账号评分可解释

- 建立账号评分计算。
- 展示评分原因。
- 按 A / B / C / D / E 分层选号。
- 任务创建和启动时提示账号容量和风险。
- 任务中心账号选择展示推荐可用、谨慎使用、不可用三组。
- 风控评分纳入代理健康、代理未绑定、代理复用和代理异常。
- 建立代理健康检查任务，支持不可达、超时、认证失败、复用超限和 FloodWait 聚集识别。
- 账号登录、验证、同步、监听和发送统一通过账号绑定代理创建 Telegram 客户端。
- 建立站内通知和主导航角标，只对严重告警主动提醒。

### P2：风控闭环

- 建立风控事件和处置任务。
- 支持暂停、冷却、降频、换号、恢复。
- 风控事件关联任务、账号、目标、规则命中和执行项。
- 支持策略版本和审计回溯。
- 执行链路回写转派、延后、拦截、暂停的完整原因。
- 执行链路回写代理异常、代理切换和代理维度风险。
- 代理新增、修改、禁用、绑定变更、告警确认和恢复都进入审计。
- 代理健康检查、代理告警和发送失败可以通过 `trace_id` 串起来。

### P3：策略自动化

- 根据近期失败、FloodWait 和发送压力自动收紧全局策略。
- 根据连续恢复和成功执行逐步恢复账号等级。
- 对任务提供容量预测和风险预警。
- 支持账号角色一致性驱动的自动转派和自动延后。
- 支持按代理出口维度做容量预测和风险预警。

## 19. 验收清单

风控中心和账号中心的设计落地，至少要满足以下检查：

| 类别 | 验收点 |
| --- | --- |
| 模块边界 | 账号中心只管账号事实、登录、同步、分组和代理绑定；风控中心管评分、告警、准入和处置 |
| 代理边界 | 系统只认识本地 socks5/http 地址和 `proxy_id`，不保存机场订阅，不展示节点名称 |
| admin 模型 | 系统只有少量同级 admin，不设计 RBAC，不展示角色和权限组 |
| 任务入口 | 没有代理告警时，任务仍按账号分组 / 账号池创建，不要求按代理选择 |
| 代理告警 | 不可达、超时、认证失败、异常聚集、FloodWait 聚集、复用超限都有明确状态和处置 |
| 通知 | 只有 `alerting` 或 critical 级别主动提醒，观察状态不刷屏 |
| 预检查 | 返回账号、代理、目标、内容四类决策，且执行前必须重新校验 |
| 发送链路 | 登录、验证、同步、监听、发送都通过账号绑定代理，不能裸连 |
| 策略优先级 | 硬阻塞事实和全局风控硬策略不能被任务级配置覆盖 |
| 审计 | 代理新增、编辑、禁用、绑定变更、告警确认、忽略、恢复和凭据重置都可追溯 |
| 安全 | 代理密码密文保存，不回显，不导出 |
| 前端页面 | 风控中心有总览、全局策略、账号评分、代理告警、处置队列、命中记录和策略审计 |
| 可追踪 | 预检、任务创建、执行校验、失败回写、代理告警和审计可通过 `trace_id` 串联 |

## 20. 结论

账号中心和风控中心不要混成一个模块。

账号中心是资产管理和登录同步入口；风控中心是运行期安全控制和处置入口。账号中心提供账号事实，风控中心基于事实生成评分、策略和处置，任务中心按风控结论执行。

这样拆分后，运营人员看到的是清晰链路：

```text
账号在哪里接入和修复？
  -> TG账号管理

账号现在能不能参与任务、需要怎么降频或处置？
  -> 风控中心

任务为什么没发、为什么换号、为什么暂停？
  -> 风控中心 + 任务详情 + 审计记录
```
