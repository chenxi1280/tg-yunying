# TG 运营管理平台功能介绍与改进分析

本文基于当前代码与新版架构改造结果整理，说明平台现有能力、已收敛的产品口径，以及后续仍需要继续推进的事项。

## 1. 平台定位

新版 TG 运营管理平台不再以多租户 SaaS、卡密订阅、旧 Campaign 或 AI 草稿人工审核作为主线。当前主线是：

```text
账号接入
  ↓
资产同步
  ↓
运营目标确认
  ↓
规则配置
  ↓
任务创建
  ↓
监听 / AI / Worker 执行
  ↓
任务详情追踪
  ↓
数据复盘
  ↓
审计留痕
```

平台围绕自有、授权、可运营的 TG 账号和目标工作。AI 活跃群与转发监听群不走人工审核，而是通过自动校验、系统级规则、风控限速、上下文检查、失败重试和审计记录保证可控。

## 2. 当前主导航

当前前端主导航已经按新版架构收敛为：

| 导航 | 核心用途 |
| --- | --- |
| 运营概览 | 展示账号、目标、任务、监听、发送、AI、失败和风险状态 |
| TG账号管理 | 开发者应用绑定、账号接入、登录、健康、同步、验证码、分组 |
| 运营目标 | 统一维护群 / 频道目标，展示能力、账号覆盖、最近消息和快捷任务入口 |
| 消息发送 | 面向运营目标、联系人和手动目标的直接发送、批量发送、定时发送和记录追踪 |
| 任务中心 | 频道互动、AI 活跃群、转发监听群任务创建、配置、启动、暂停、重试、重置、停止和详情 |
| 监听中心 | 按频道 / 源群聚合监听状态、订阅任务、事件积压和错误状态 |
| 规则中心 | 系统级规则集、规则版本、过滤 / 转换 / 路由 / 账号策略和规则测试 |
| 归档中心 | 从运营目标创建归档，查看消息库、成员库、初始化方案、导出和重跑 |

## 3. 已完成的核心收敛

### 3.1 P0 口径收敛

- README、产品文档和技术架构文档已改为新版 TG 运营管理平台口径。
- 前端主导航已移除旧卡密、套餐、订阅、多租户商业后台入口。
- 系统设置只展示当前真实可用的开发者应用和 AI 相关配置。
- 旧 `/api/campaigns`、`/api/ai-drafts`、`/api/operation-tasks` 与 `/api/review-*` 已隔离为兼容路由，开发/生产默认不注册，仅测试环境或显式 `ENABLE_LEGACY_CAMPAIGN_ROUTES=true`、`ENABLE_LEGACY_OPERATION_TASK_ROUTES=true`、`ENABLE_LEGACY_REVIEW_ROUTES=true` 时开启。新功能入口默认走任务中心、运营目标、监听中心和规则中心。旧 Campaign 连续任务、旧 OperationTask worker 调度和旧 ReviewQueue 发送阻塞默认关闭，只能通过环境变量显式启用。

### 3.2 运营目标统一

运营目标列表现在会返回：

- 关联群资产 ID
- 是否可发送
- 是否可监听
- 是否可归档
- 可发账号数
- 监听账号数

消息发送、任务中心和归档中心都已优先围绕运营目标工作。群目标需要回查关联 `tg_groups` 和账号权限，避免只靠标题或 peer 模糊匹配。

### 3.3 频道任务子任务化

频道浏览、点赞、评论任务详情按“频道消息 + 动作类型”聚合展示，每条消息能看到目标数、完成数、失败数、重复数、运行中数量、容量缺口和子任务状态。

频道动作执行继续使用去重规则，避免同账号对同频道、同消息、同动作重复执行。

### 3.4 监听中心化

任务执行层增加监听聚合缓存，频道和群上下文采集会按频道 / 源群在短窗口内去重，避免多个任务重复触发同一监听对象。监听中心页面通过后端汇总接口展示监听对象、订阅任务、最后事件、积压和错误。

### 3.5 AI 活跃群循环化

AI 活跃群已经按 Cycle / Turn 记录本轮计划、账号角色、上下文快照、自动校验结果和发送记录。新增静默期、每日爬坡、上下文过期检查：

- 静默期降低参与账号和每轮发言数。
- 每日启动期按比例逐步爬坡。
- 群上下文变化超过阈值时，未发送 Turn 会跳过并记录上下文过期原因。

AI 活跃群不进入人工审核队列，生成后由自动校验和执行前检查决定通过、跳过、拦截、延迟或失败。

### 3.6 转发监听规则化

转发监听群任务支持绑定系统级规则集和版本，执行流程为：

```text
源群消息事件
  ↓
过滤规则
  ↓
转换规则
  ↓
路由规则
  ↓
账号分配
  ↓
目标群自动发送
```

规则中心支持规则集创建、版本创建、发布和规则测试。转发任务详情按源事件、转发批次、目标群发送项展示处理结果。

### 3.7 运营数据与审计

运营数据页已接入真实汇总接口，覆盖账号、目标、消息发送、频道互动、AI 活跃群、转发监听、归档、AI 用量和失败风险。

审计接口和页面已支持操作者、动作、对象、账号、运营目标、任务、状态、关键词和时间范围筛选。

### 3.8 任务终态语义

任务中心现在区分自然完成、人工停止和删除：

- 到达结束时间：`completed`
- 管理员停止：`stopped`
- 管理员删除：`deleted`，同时软删除隐藏任务

停止和删除都会把未完成执行项标记为跳过，并写入明确原因，避免把人工终止误判为自然完成。

## 4. 仍需继续推进的事项

### P1：旧兼容层清理策略

旧 Campaign、AI Draft、旧 OperationTask 和 MessageTask 仍作为数据兼容层存在。当前策略是新入口不再使用旧 Campaign / AI Draft / OperationTask，相关路由在开发和生产默认不注册；`/api/message-tasks` 继续支撑消息发送中心的发送记录、派发、重试和取消。旧 Campaign 连续任务和旧 OperationTask worker 调度默认不运行。

后续需要明确：

- 哪些旧接口只保留只读兼容，哪些可以在迁移完成后完全删除。
- 哪些旧 worker 逻辑可以彻底移除，而不是仅靠 `ENABLE_LEGACY_CAMPAIGN_ROUTES`、`ENABLE_LEGACY_OPERATION_TASK_ROUTES`、`ENABLE_LEGACY_REVIEW_ROUTES`、`ENABLE_LEGACY_CAMPAIGN_WORKER` / `ENABLE_LEGACY_OPERATION_TASK_WORKER` 临时启用。
- 哪些旧状态字段需要迁移到任务中心。

### P1：监听聚合持久化

当前监听聚合已有短窗口运行时去重和状态汇总，但还不是完整持久化订阅模型。后续可补：

- 监听对象表
- 任务订阅表
- 事件表
- 事件分发记录
- 监听账号自动切换记录

### P1：频道动态新消息任务

频道任务已支持 `dynamic_new` 消息范围。Worker 每轮按聚合监听窗口采集频道新消息，任务只为尚未规划过的频道消息生成动作子任务，因此同一频道可以由多个任务共享采集结果，同时避免重复执行已成功动作。后续如果需要更强实时性，可以继续把 Telethon 频道新消息事件持久化为订阅事件表。

- 频道新消息事件持久化。
- 显式任务订阅表。
- 监听账号自动切换记录。

### P2：旧前端上下文瘦身

前端主入口已不展示旧卡密、套餐、多租户管理页面，订阅 / 卡密 / token ledger 状态也已从主上下文隔离。`context.tsx` 仍保留少量旧 Campaign 兼容函数，用于把历史入口引导到新版任务中心；后续应在不破坏当前构建的前提下继续拆分：

- 删除旧 Campaign 创建入口的兼容状态。
- 把兼容层函数移动到 legacy 模块，避免主上下文继续膨胀。

### P2：规则能力继续细化

当前 P0 规则能力覆盖过滤、转换、路由和规则测试。后续可继续补：

- 规则冲突检测
- 规则回滚入口
- 发送账号粘性策略可视化
- AI 改写自动校验记录
- 源消息编辑 / 删除同步策略

## 6. 总结

当前项目已经从旧的 Campaign / 卡密订阅 / 多租户 SaaS 原型，收敛为以账号、运营目标、任务中心、监听中心、规则中心、AI 内容、执行、数据和审计为核心的 TG 运营管理后台。下一阶段重点不是继续堆入口，而是清理旧兼容层、持久化监听订阅、强化动态频道任务，并继续降低前端主上下文的历史负担。
