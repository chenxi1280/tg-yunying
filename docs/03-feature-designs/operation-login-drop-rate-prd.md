# 运营数据账号分组登录掉号比例 PRD

> 日期口径：2026-06-20（Asia/Shanghai）
> 适用范围：运营数据、账号分组、账号中心登录状态、登录失败追踪。

## 1. 背景

运营数据当前能看到账号总数、在线账号、异常账号和异常账号明细，但无法判断问题集中在哪个账号分组。实际排障时，运营更关心“哪个分组登录类掉号比例高”，尤其是无需额外刷新就能从账号状态、主授权状态或最新登录流水看出的登录失败、验证码没收到、Session 失效和主授权不可用问题。

现有账号中心已经定义了“登录有问题”的搜索口径，运营数据应复用该口径，避免把账号受限、禁用、疑似封禁等非登录问题混入“掉号比例”。

## 2. 目标

1. 在运营数据中展示每个账号分组的登录类掉号比例。
2. 比例口径清晰可解释：分组内登录有问题账号数 / 分组内未删除账号总数。
3. 登录问题口径与账号中心保持一致，覆盖登录失败、验证码没收到、Session 失效、需重新登录和主授权不可用。
4. 展示每个分组的掉号数、总数、百分比和主要原因分布，方便运营定位优先处理分组。
5. 不新增静默兜底、模拟成功或新的状态转换规则。

## 3. 非目标

- 不修改登录、健康检查、验证码读取或授权修复流程。
- 不把受限、禁用、疑似封禁、已封禁单独计入登录掉号，除非账号同时命中登录流水失败或主授权不可用。
- 不改变账号中心已有搜索行为。
- 不新增分组管理、迁移账号或自动修复功能。

## 4. 统计口径

账号分组使用 `AccountPool`，账号通过 `TgAccount.pool_id` 归属；只统计 `deleted_at is null` 的账号。

一个账号命中以下任一条件，即计入“登录有问题”：

- 账号状态为 `待登录`、`等待验证码`、`等待扫码`、`等待2FA`、`需重新登录`、`Session失效`、`异常`。
- 主授权状态不是 `active`，例如主授权缺失或不可用。
- 最新一条 `TgLoginFlow` 存在 `failure_type` 或 `failure_detail`，例如登录失败、验证码没收到。

每个分组输出：

- `pool_id`
- `pool_name`
- `total_accounts`
- `login_issue_accounts`
- `login_issue_rate`
- `status_counts`
- `failure_count`
- `primary_unavailable_count`

`login_issue_rate` 按百分比保留 1 位小数。空分组显示 `0/0，占比 0%`。

## 5. 页面需求

运营数据页面在“账号数据”后新增“账号分组登录掉号比例”区域。

每个分组显示为一个指标卡：

- 标题：账号分组名。
- 数值：`登录问题数/账号总数`。
- 详情：`登录掉号 Z%；状态分布 ...；登录失败 X；主授权不可用 Y`。

展示顺序：

1. 登录掉号比例高的分组优先。
2. 比例相同时，登录问题账号数多的优先。
3. 再按分组 ID 升序。

## 6. 验收标准

1. 后端 `/api/operation-metrics/summary` 返回 `account_pool_login_drop_rates`。
2. 账号分组中有 4 个账号、2 个命中登录问题时，返回比例 `50.0`，详情包含 `2/4`。
3. 最新登录流水有失败详情的账号，即使账号状态为在线，也计入登录问题。
4. 主授权不可用的账号计入登录问题。
5. 受限账号如果没有登录流水失败且主授权正常，不计入登录问题。
6. 前端运营数据展示“账号分组登录掉号比例”区块。
