# 素材库与图片发送设计

## 1. 设计目标

素材库用于把文字、图片、表情包、文件和组合消息沉淀成平台可复用素材，让消息发送、转发监听和 AI 活跃群不再只能发送纯文字。

核心目标：

- 后台可以实时上传和维护素材。
- 素材中心支持上传 zip 压缩包批量导入图片、表情包分组和头像包，减少运营逐张上传成本。
- 图片和表情包优先缓存到 Telegram 私有缓存会话，作为后续下载重发的稳定来源。
- 普通消息发送可以直接选择图片、表情包或组合消息。
- 转发监听可以保留源图、替换图片、追加图片或按规则选择素材。
- AI 活跃群可以按规则选择表情包或图片作为自然互动的一部分。
- 所有素材选择、发送、失败、回退和命中规则都能追踪到素材、任务、账号、目标和规则版本。

素材库不是旧 Campaign 的素材附属品，也不是人工审核队列，更不是系统设置里的附属 Tab。它应作为独立一级“素材中心”，被规则中心、任务中心、消息发送、AI 内容和账号资料初始化共同引用；系统设置只保留缓存账号、缓存会话、上传限制和临时文件 TTL 等运行配置。

## 2. 核心原则

```text
素材统一入库
图片只临时中转
媒体持久化交给 Telegram 缓存
发送只消费可用素材
规则决定何时用图
AI 只提出意图和标签
执行层负责发送与回退
全链路审计留痕
```

设计边界：

- 素材必须属于当前系统范围和授权运营目标，不做公共素材市场。
- 平台默认不持久化图片、表情包、视频或文件原始二进制，避免占用服务器磁盘和备份空间。
- 图片、表情包、文件等媒体素材必须有可下载和可重新发送的来源：Telegram 缓存消息或外部可恢复 URL；本地文件只作为上传、下载、预览和重发前的临时中转。
- 执行层不是“媒体引用转发”。发送图片、文件、sticker、源媒体和表情包时，默认从已缓存来源临时下载文件，再用发送账号重新上传到目标；不能把 TG file ref 当作跨账号、跨网关、跨目标都可复用的通用引用。
- 素材资产发布后要按版本引用，运行中的规则版本和执行项不能因为后台编辑素材而静默改变。
- 自动化任务不能绕过规则中心和风控中心直接发送素材。
- AI 活跃群里的表情包选择以规则为主，AI 可以输出情绪、话题、意图或标签，但最终素材命中必须由规则引擎解释和记录。
- 转发监听保留源媒体时必须记录源消息、源媒体指纹和目标发送结果；替换或追加平台素材时必须记录平台素材 ID。

## 3. 素材类型

素材库支持的类型：

| 类型 | 用途 | 发送形态 |
| --- | --- | --- |
| 文本 | 常用话术、固定回复、尾注 | `send_message` |
| 图片 | 活动图、宣传图、截图、产品图 | `send_file` + caption |
| 表情包 | AI 活跃群自然互动、欢迎、回应、气氛补充 | 图片伪表情、静态 sticker、animated sticker、video sticker 或 custom emoji |
| 文件 | 文档、压缩包、资料包 | `send_file` |
| 链接 | 活动链接、落地页链接 | 文本 + URL |
| 组合消息 | 多段文本、图片、表情包、链接的顺序组合 | 分段发送 |

素材状态建议：

```text
草稿
上传中
待缓存
可用
缓存失效
停用
发送失败过多
已删除
```

当前代码已有 `materials` 基础表、素材类型、审核状态、使用次数、最后使用时间，以及消息发送中心的图片素材选择能力。后续实现应在此基础上增强媒体缓存和规则使用关系，而不是另起一套旧 Campaign 素材模型。

表情包资产需要单独记录子类型：

| `emoji_asset_kind` | 含义 | 默认处理 |
| --- | --- | --- |
| `image_meme` | 普通图片伪表情包 | 作为图片下载后重新上传发送。 |
| `static_sticker` | Telegram 静态 sticker | 保留 sticker 文件形态，目标不支持时可降级为图片。 |
| `animated_sticker` | Telegram animated sticker | 发送前校验账号、目标和网关能力，不支持时跳过或降级。 |
| `video_sticker` | Telegram video sticker | 默认低频，按规则显式允许后发送。 |
| `custom_emoji` | Telegram custom emoji | 受账号、客户端和目标环境限制，必须做能力校验，不可用时降级或跳过。 |

表情包能力边界：

- 平台设计上支持普通图片伪表情包、真实 Telegram sticker、animated sticker、video sticker 和 custom emoji。
- AI 只能输出表情意图，不直接决定具体 `emoji_asset_kind`。
- 规则中心可以限制任务类型、目标群和账号允许使用的表情包子类型。
- custom emoji 不是稳定通用文件能力，执行前必须校验目标和发送账号是否可用。

## 4. 上传与 Telegram 缓存

后台新增素材时应支持四种来源：

- 本地上传文件。
- zip 压缩包批量导入图片包。
- 输入图片 / 文件 URL。
- 从历史源消息或归档消息中收藏为素材。

推荐上传流程：

```text
后台选择文件 / zip / URL / 历史消息
  ↓
生成素材草稿和基础元数据
  ↓
计算内容指纹
  ↓
临时落地到上传缓存目录
  ↓
选择素材缓存账号
  ↓
上传到 Telegram 素材缓存会话 / 私有频道 / 收藏夹
  ↓
保存 Telegram 缓存消息、引用摘要和可重发状态
  ↓
删除本地临时文件
  ↓
标记素材可用
```

默认存储策略：

```text
平台数据库：只存元数据、标签、指纹、规则引用、TG 缓存位置和状态
平台磁盘：只允许最长 10 天 TTL 临时文件，不持久化图片原文件
Telegram 缓存会话：承载后续下载和重新发送的媒体实体
外部 URL：仅作为可选恢复来源，不默认依赖
```

平台仍应保存最小可追踪信息：

- 素材 ID、类型、标题、标签。
- 原始文件名、MIME、大小、宽高、时长。
- 内容指纹。
- 外部 URL 或原始来源说明；不保存本地原图路径作为长期依赖。
- Telegram 缓存会话、缓存消息 ID、媒体类型、远端引用摘要和可重发状态。
- 最近一次缓存时间、最近一次发送成功时间、缓存失效次数。

执行发送任务时只消费已经缓存可用的素材，不在发送现场上传素材、不等待缓存刷新、不处理素材缓存账号 FloodWait。

压缩包批量导入规则：

- zip 导入只在素材中心作为批量导入容器，不把 zip 自身保存为可发送文件素材，也不允许自动化任务直接发送 zip 包。
- 导入前必须选择导入类型：表情包分组、头像包或普通图片分组；默认可用 zip 文件名作为分组 / 素材包名称，运营人员可在导入前修改。
- 包内默认只接受 `.png`、`.jpg`、`.jpeg`，同时做 MIME 二次校验，不能只信任扩展名。
- 单张图片上限 500KB；超过 500KB 的文件必须跳过并记录 `file_too_large`，不能默认压缩、裁剪或静默降质。
- 非图片文件、目录、隐藏文件、`__MACOSX`、空文件、重复文件和校验失败文件必须跳过，并在导入结果中展示原因。
- zip 解析、校验、去重和上传缓存进入异步导入任务；页面展示包名、解析总数、成功数、失败数、跳过数、重复数、超过 500KB 数、生成的素材 / 素材包和逐文件失败原因。
- 成功导入的每张图片仍要生成素材记录、资产版本、指纹和 TG 缓存引用，后续发送只消费缓存可用素材。

发送时优先顺序：

```text
Telegram 缓存消息可用：临时下载到发送工作目录
  ↓
发送账号重新上传到目标并发送
  ↓
缓存消息不可用或素材未缓存：标记 cache_not_ready，进入素材库刷新队列
  ↓
刷新队列尝试从外部可恢复 URL 重新导入 TG 缓存；没有来源则放弃
  ↓
本次执行按规则回退为文本、换素材或跳过
```

需要注意：Telegram 用户客户端、Bot API 和 Telethon 对媒体引用的复用能力不完全相同。实现时应把 `tg_file_ref` 设计成网关内部结构，不要把它当作所有场景都稳定通用的公开 file_id。页面只展示“已缓存 / 需刷新 / 缓存失败 / 可重发”等运营状态；业务执行口径统一为 `delivery_mode=download_reupload`。

### 4.1 不持久化图片原文件

为控制服务器空间，素材库默认不保存图片、表情包、视频、文件原件。

允许保存：

- 文件名、大小、MIME、宽高、时长。
- 内容指纹。
- 标签、caption、来源类型。
- Telegram 缓存账号、缓存 peer、缓存消息 ID。
- Telegram 缓存位置、引用摘要和刷新状态。
- 外部 URL，前提是该 URL 被标记为可恢复且通过安全校验。

不允许默认保存：

- 图片原文件。
- 表情包原文件。
- 视频原文件。
- 大文件原件。
- 源群转发图片的长期本地副本。

仅允许最长 10 天 TTL 临时文件：

```text
上传 / 下载开始
  ↓
写入临时目录
  ↓
完成安全校验和 TG 缓存上传
  ↓
删除临时文件
  ↓
只保留 TG 缓存位置、引用摘要和元数据
```

临时文件清理要求：

- 正常上传成功后立即删除。
- 上传失败、下载失败、预览失败、缓存刷新失败或重新发送异常后，临时文件最长保留 10 天，超过后自动清理。
- 后台启动时扫描并清理过期临时文件。
- 临时目录不进入备份。
- 临时文件路径不作为业务可恢复字段。
- 10 天是异常残留上限，不是成功文件保留期；成功流程仍应立即删除本地临时文件。

### 4.2 缓存账号异常与刷新队列

素材缓存账号本身也可能离线、Session 失效、FloodWait、代理不可用或无法访问素材缓存会话。素材库不能在发送现场临时阻塞等待复杂恢复，应该把缓存刷新设计成独立队列。

关键边界：

- 素材上传、缓存刷新、源媒体收藏和外部 URL 恢复都属于素材库队列，不属于消息发送任务现场。
- 执行发送任务只能使用 `cache_ready_status=ready` 的素材。
- 如果素材缓存账号处于 FloodWait，记录 `last_cache_flood_wait_until`，刷新队列延后处理。
- 缓存账号 FloodWait 时，本次发送任务不等待、不现场上传、不阻塞整个批次，直接按规则换已就绪素材、降级文本或跳过。
- 发送账号自己的 FloodWait 仍归风控中心和账号执行策略处理，不能和素材缓存账号 FloodWait 混在一起。
- 同一源消息或同一素材在 FloodWait 期间只能有一条待刷新记录，后续命中同一去重键时复用原记录并更新重试时间，不能反复创建“新图片”。

缓存刷新状态：

```text
待刷新
刷新中
刷新成功
缓存账号不可用
缓存会话不可访问
TG 限流等待
素材未缓存
无可恢复媒体来源
刷新失败
```

处理策略：

- 缓存账号不可用时，刷新队列延后或改用系统配置中的可用缓存账号；发送现场不等待。
- Telegram 限流时记录等待秒数，按队列延后刷新。
- 缓存会话不可访问时，提示重新选择缓存位置。
- Telegram 缓存消息不可用且没有外部可恢复 URL 时，素材标记为不可恢复，禁止继续自动发送。
- 刷新失败达到阈值后，素材状态转为“缓存失效”，运行中任务按规则降级、换素材或跳过。

刷新队列要记录素材 ID、资产版本、缓存账号、缓存 peer、失败原因、重试次数、下一次重试时间、`last_cache_flood_wait_until` 和最后一次成功引用。

刷新队列去重键建议：

```text
平台素材：material_id + asset_version_id
源媒体：source_peer_id + source_message_id + source_media_group_id + media_group_index
可计算文件指纹时：asset_fingerprint
无法下载文件时：TG media 摘要 + 源消息 ID
```

去重命中时只更新状态、重试次数和下一次重试时间，不生成新的素材记录。

### 4.3 TG 缓存会话与失效边界

平台不持久化媒体原文件后，Telegram 缓存会话就是媒体素材的核心资产。但它不是平台灾备系统，也不承诺能恢复已经被 Telegram 删除或账号失权的媒体本体。素材库只维护当前可用缓存引用、健康状态、清理记录和失效后的降级边界。

缓存 peer 类型：

| 类型 | 用途 | 说明 |
| --- | --- | --- |
| 素材缓存私有频道 | 平台素材缓存 | 推荐默认方式，便于统一管理、分片和健康检查。 |
| 源媒体缓存私有频道 | 转发监听源媒体缓存 | 单独存放源群临时保留媒体，便于按授权和保留期清理。 |
| 缓存账号收藏夹 | 小规模兜底 | 仅适合早期或测试，不建议长期作为唯一缓存。 |

缓存会话管理规则：

- 缓存频道由系统初始化流程创建或绑定。
- 缓存频道名称应带环境和用途，例如 `tg_ops_material_cache_prod_001`、`tg_ops_source_media_cache_prod_001`。
- 缓存账号必须是已接入、在线、健康、代理可用的账号。
- 缓存账号必须保持对缓存频道的访问权限。
- 缓存频道不能暴露给普通运营目标，不作为业务群使用。
- 缓存频道按用途、时间或素材量分片，避免单个频道消息过多。

缓存失效处理：

```text
TG 缓存消息可用：执行期临时下载并由发送账号重新上传
  ↓
TG 缓存消息不可访问：标记 cache_not_ready / material_unrecoverable
  ↓
素材库刷新队列可选尝试外部可恢复 URL 重新导入
  ↓
本次执行不现场上传、不等待、不做灾备切换，按规则换已就绪素材、降级文本或跳过
```

每个素材只要求维护当前有效 TG 缓存引用。平台不设计“备用缓存引用”作为必备能力；如果未来允许多引用，也只能作为多个可用来源记录，不能宣传为灾备或恢复保证。

缓存健康检查：

- 定期验证缓存账号 Session 是否有效。
- 定期抽检缓存 peer 是否可访问。
- 定期抽检缓存消息是否存在。
- 定期验证最近 N 个热门素材是否能临时下载和重新发送前校验。
- 缓存异常进入素材库健康汇总和风控 / 系统运行提醒。

缓存清理策略：

- 平台素材被规则版本、运行中任务、发送记录或审计引用时，不能直接清理其当前缓存引用。
- 停用且无运行中引用的素材，可在保留期后清理 TG 缓存消息。
- 源媒体缓存应按源任务授权和保留期直接清理，默认保留期短于平台素材。
- 清理前必须做影响检查并写审计。
- 清理后素材状态应改为“缓存已清理”或“不可自动发送”，不能继续显示为可用。

### 4.4 缓存引用版本

素材内容版本和 Telegram 缓存引用版本要分开。

```text
asset_version_id
  表示素材内容版本，例如图片内容、外部 URL、组合消息结构。

tg_ref_version_id
  表示 Telegram 缓存引用版本，例如缓存账号、缓存 peer、缓存消息 ID 或媒体引用刷新。
```

需要生成新 `asset_version_id` 的情况：

- 替换图片、表情包、文件。
- 修改外部 URL。
- 修改组合消息结构。
- 重新裁剪、压缩、转码。

只生成新 `tg_ref_version_id` 的情况：

- 同一素材重新上传到缓存频道。
- 缓存消息被刷新。
- 缓存账号替换。

执行项发送时应固化：

```text
material_id
asset_version_id
asset_fingerprint
tg_ref_version_id
tg_cache_peer_id
tg_cache_message_id
resolved_rule_set_version_id
```

这样可以区分“素材内容变了”和“同一素材换了一个 Telegram 缓存引用”。

### 4.5 上传安全校验

素材入库必须先做基础安全和可用性校验。

默认校验项：

- 文件大小限制。
- MIME 白名单。
- 图片宽高和像素上限。
- 视频、压缩包、可执行文件默认禁止进入自动化任务；zip 仅可通过素材中心图片包导入入口作为批量导入容器。
- URL 必须可访问，禁止内网地址、临时鉴权地址和明显不可恢复地址。
- 图片 EXIF 等非必要元数据可按策略清理。
- 图片带二维码、联系方式、外链水印时，进入高风险标签或要求规则显式允许。
- 相同文件指纹默认去重，允许复用既有素材，不重复上传。

外部 URL 安全要求：

- 禁止访问 `localhost`、`127.0.0.0/8`、`::1`、内网网段、链路本地地址和云厂商元数据地址。
- DNS 解析后也要校验目标 IP，防止域名指向内网。
- 限制重定向次数，重定向后的 URL 重新做安全校验。
- 限制下载大小、连接超时和总耗时。
- 先探测响应头，再按流式下载，超过大小立即中断。
- 下载后做 MIME 二次校验，不只相信 `Content-Type`。
- 临时签名 URL、鉴权 URL、一次性 URL 默认不作为长期恢复来源，只能用于一次性导入。
- 外部 URL 失败达到阈值后，清除可恢复标记或提示运营重新上传到 TG 缓存。

校验结果：

```text
通过
需要人工确认风险标签
禁止入库
入库但禁止自动化任务使用
```

## 5. 数据模型建议

在现有 `materials` 基础上增强，推荐拆成三层：

```text
materials
  素材基础信息：标题、类型、标签、状态、说明、使用统计

material_assets
  媒体元数据：外部 URL、指纹、MIME、大小、宽高、预览状态

material_tg_refs
  TG 缓存引用：缓存账号、缓存 peer、缓存消息 ID、引用摘要、状态、刷新时间

material_tg_ref_versions
  TG 引用版本：每次刷新缓存账号、缓存 peer 或缓存消息时生成新版本

material_asset_versions
  素材资产版本：每次替换文件、变更 URL 或重建组合消息时生成新版本

material_import_jobs
  批量导入任务：来源文件、导入类型、目标分组 / 素材包、统计结果、逐文件失败原因
```

关键字段建议：

| 字段 | 说明 |
| --- | --- |
| `material_type` | 文本、图片、表情包、文件、链接、组合消息 |
| `content` | 文本内容、默认 caption、链接或组合消息 JSON |
| `tags` | 用于规则命中和 AI 意图匹配 |
| `status` | 草稿、上传中、可用、缓存失效、停用等 |
| `asset_fingerprint` | 文件指纹，用于去重、归因和重复发送控制 |
| `source_kind` | upload、url、telegram_message、archive |
| `external_source_url` | 可选外部恢复 URL；平台不默认保存本地原图路径 |
| `tg_cache_account_id` | 用于上传和刷新 Telegram 缓存的账号 |
| `tg_cache_peer_id` | 素材缓存会话、私有频道或收藏夹 |
| `tg_cache_message_id` | Telegram 缓存消息 ID |
| `tg_ref_status` | 可用、需刷新、失败 |
| `tg_ref_version_id` | 当前 Telegram 缓存引用版本 |
| `asset_version_id` | 当前可用资产版本 |
| `referenced_by_count` | 被规则版本、任务或发送记录引用次数 |
| `delivery_mode` | 默认 `download_reupload`，表示执行期下载缓存文件后重新上传发送 |
| `emoji_asset_kind` | 表情包子类型：图片伪表情、静态 sticker、animated sticker、video sticker、custom emoji |
| `gateway_type` | 缓存引用来源网关类型，例如 Telethon、Bot API，用于能力校验和兼容边界 |
| `cache_ready_status` | not_cached、ready、refreshing、flood_wait、unrecoverable |
| `last_cache_flood_wait_until` | 素材缓存账号最近一次 FloodWait 结束时间 |

`material_import_jobs` 至少记录 `source_filename`、`import_type`、`target_group_id` / `target_pack_id`、`allowed_extensions`、`max_item_size_kb`、`total_count`、`success_count`、`failed_count`、`skipped_count`、`duplicate_count`、`oversize_count`、`status`、`error_summary` 和逐文件明细。zip 图片导入默认 `allowed_extensions=["png","jpg","jpeg"]`、`max_item_size_kb=500`。

组合消息建议使用结构化 JSON：

```json
[
  { "type": "文本", "content": "今晚活动开始了" },
  { "type": "图片", "material_id": 12, "caption": "活动图" },
  { "type": "表情包", "material_id": 18 }
]
```

### 5.1 素材版本与不可变引用

素材基础信息可以编辑，例如标题、标签、说明和默认 caption；但会影响发送内容的资产不能直接覆盖历史引用。

需要生成新资产版本的操作：

- 替换图片、表情包或文件。
- 修改外部 URL。
- 修改组合消息结构。
- 重新裁剪、压缩、转码素材。
- 从源消息重新收藏并生成新的 TG 缓存消息。

运行时固化引用：

```text
执行项生成时
  ↓
固化 material_id
  ↓
固化 asset_version_id
  ↓
固化 asset_fingerprint
  ↓
固化 tg_ref_version_id
  ↓
固化 resolved_rule_set_version_id
```

规则版本发布时只固化素材选择策略和可选素材范围；具体执行项生成时再固化本次最终素材 ID 和资产版本。这样既能让规则版本可追溯，又不会让运行中任务因为后台编辑素材而静默改变。

### 5.2 删除、停用与引用保护

素材被任务、规则版本、发送记录、审计记录或归因报表引用后，不允许物理删除。

删除策略：

- 未被引用的草稿素材可以删除。
- 已被引用的素材只能停用或软删除。
- 运行中任务正在引用的素材，停用时必须提示影响范围。
- 已发布规则版本引用的素材，停用后不修改历史版本，只影响后续执行项的素材可用性检查。
- 自动任务遇到停用素材时，按规则执行换素材、降级文本或跳过。

页面需要展示引用范围：

- 绑定的规则集和规则版本。
- 正在运行的任务。
- 最近使用的发送记录。
- 最近失败或降级记录。

### 5.3 素材库权限

素材库要接入后台账号与权限体系。前端隐藏按钮只是体验，后端接口必须校验权限。

建议权限点：

| 权限 | 动作 |
| --- | --- |
| `materials.view` | 查看素材列表、临时预览和基础信息；预览来自 TG 缓存或外部 URL，不依赖平台本地原图。 |
| `materials.create` | 新增素材、上传文件、从 URL 入库。 |
| `materials.edit` | 编辑标题、标签、caption 和说明。 |
| `materials.asset.replace` | 替换文件、重建资产版本。 |
| `materials.cache.refresh` | 刷新 Telegram 缓存引用。 |
| `materials.disable` | 停用或恢复素材。 |
| `materials.delete` | 删除未引用草稿或软删除素材。 |
| `materials.audit.view` | 查看素材使用记录、失败详情和归因报表。 |

高风险操作需要二次确认并写审计：

- 替换资产版本。
- 停用被规则或任务引用的素材。
- 批量刷新 TG 缓存。
- 删除或软删除素材。

## 6. 消息发送中心集成

消息发送中心保持“直接发送”定位，但发送内容不再只有文本。

新建发送支持：

- 纯文本。
- 图片 + caption。
- 表情包。
- 文件 + caption。
- 链接。
- 组合消息。

发送前预检查增加：

- 素材是否可用、未停用、未删除。
- 图片 / 表情包 / 文件是否存在已经可用的 TG 缓存消息。
- Telegram 缓存是否可下载；不可用时不在发送现场刷新，只记录 `cache_not_ready` 并按规则降级。
- 表情包子类型是否被目标、账号、网关和规则允许。
- 素材类型是否和消息类型匹配。
- 素材标签、链接、caption 是否通过规则中心输出校验。
- 账号和目标是否允许发送媒体。
- 目标是否处于只允许文本、禁图、慢速或风控冻结状态。

发送记录面向运营侧只展示媒体发送结论，不暴露缓存内部细节：

- 基础消息字段：发送账号、目标、发送时间、状态、远端消息 ID。
- 媒体结果：是否包含图片 / 表情包 / 文件，以及是否成功发送。
- 失败原因：内容失败、素材失败、缓存失败、账号失败或目标失败。

素材 ID、素材指纹、缓存版本、下载重发链路、等待队列、唤醒来源和临时文件清理等内部字段只进入审计和诊断，不作为发送记录主展示字段，避免运营人员把缓存过程误认为可人工补救流程。

素材相关失败原因码建议：

| 原因码 | 含义 |
| --- | --- |
| `tg_ref_expired` | Telegram 媒体引用过期或不可复用。 |
| `cache_message_deleted` | 缓存消息不存在或已被删除。 |
| `cache_peer_unavailable` | 缓存频道 / 会话不可访问。 |
| `cache_account_unavailable` | 缓存账号离线、Session 失效或代理不可用。 |
| `cache_account_flood_wait` | 素材缓存账号处于 FloodWait，本次发送不等待刷新。 |
| `cache_not_ready` | 素材尚未缓存完成或缓存刷新中。 |
| `material_cache_wait_timeout` | 等待源媒体缓存完成超时。 |
| `material_cache_wakeup_missed` | 缓存完成事件丢失后由兜底扫描重新唤醒。 |
| `material_cache_event_stale` | 缓存完成事件版本过旧，不能唤醒当前执行项。 |
| `material_cache_wait_queue_full` | 等待缓存执行项超过队列容量。 |
| `source_media_cache_failed` | 源媒体缓存失败；按规则丢弃媒体、降级文本或跳过，不提供人工缓存。 |
| `source_media_unrecoverable` | 源媒体已不可恢复，无法继续等待或补发，直接放弃对应媒体缓存。 |
| `external_url_unavailable` | 外部恢复 URL 不可访问。 |
| `external_url_unsafe` | 外部 URL 命中安全限制。 |
| `media_type_blocked` | 目标或规则禁止该媒体类型。 |
| `media_size_exceeded` | 媒体大小超过规则或网关限制。 |
| `emoji_type_unsupported` | 目标、账号或网关不支持该表情包子类型。 |
| `custom_emoji_unavailable` | custom emoji 当前账号或目标不可用。 |
| `album_segment_failed` | 相册中的某个分段发送失败。 |
| `preview_unavailable` | 预览不可用，但不一定影响发送。 |
| `material_disabled` | 素材被停用。 |
| `material_unrecoverable` | 无可用 TG 缓存消息且无可恢复外部来源。 |

### 6.1 组合消息执行结果

组合消息不是一个不可拆的发送动作。执行层需要按分段记录结果，避免“整体失败”掩盖真实状态。

分段结果建议：

```json
[
  { "index": 1, "type": "文本", "status": "sent", "remote_message_id": "101" },
  { "index": 2, "type": "图片", "material_id": 12, "asset_version_id": 3, "status": "failed", "failure_type": "cache_invalid" },
  { "index": 3, "type": "表情包", "material_id": 18, "status": "skipped", "reason": "target_media_cooldown" }
]
```

整体状态规则：

- 全部分段成功：成功。
- 部分分段成功：部分成功。
- 全部分段失败或跳过：失败或跳过。
- 文本成功、图片失败时，按规则决定是否重试失败分段、换已就绪素材或保持部分成功。

重试时应该能选择：

- 重试失败分段。
- 重新发送整个组合消息。
- 只重试缺失素材分段。
- 放弃素材并保留文本发送结果。

## 7. 转发监听集成

转发监听需要支持媒体消息，而不是把所有源消息降级为纯文本。

源消息事件应记录：

- `source_message_type`：text、photo、sticker、document、video、mixed 等。
- 文本内容或 caption。
- 源媒体指纹。
- 源媒体缓存状态和引用摘要。
- 源消息 ID、发送人、源群、采集账号。
- 相册消息的 `source_media_group_id`、组内顺序、总数和每条媒体的 caption。

### 7.1 源媒体落地时机

转发监听如果要支持图片、表情包或文件，不能只在执行发送时再回源群取媒体。源消息可能被删除、编辑，采集账号可能掉线，Telegram 媒体引用也可能失效。

推荐落地策略：

```text
监听中心捕获源消息
  ↓
立即记录源消息元数据
  ↓
计算可得的文本 / caption 指纹
  ↓
如果带媒体且规则可能需要媒体
  ↓
最长 10 天 TTL 临时下载并上传到 TG 源媒体缓存
  ↓
保存 source_media_asset_id、media_fingerprint、media_ref_status
  ↓
删除本地临时文件
```

源媒体记录至少包含：

- 源群、源消息 ID、采集账号。
- 消息类型、caption、媒体类型。
- 媒体指纹、大小、MIME、宽高。
- TG 源媒体缓存 peer、缓存消息 ID、媒体引用状态；不保存长期本地原图。
- 下载状态、失败原因、重试次数。
- 相册记录需要额外保存 `source_media_group_id`、`media_group_index`、`media_group_total` 和 `album_caption_policy`。

如果源媒体未能落地，转发任务只能执行“只保留文字”“降级文本”或“跳过”，不能声称已经保留源图。

源媒体缓存遇到 FloodWait 时的状态要可追踪：

- 首次捕获源图时先创建 `source_media_asset_id`，状态为 `pending_cache` 或 `cache_flood_wait`。
- 使用源群、源消息 ID、相册 ID、组内序号和可得媒体摘要作为去重键。
- FloodWait 结束后继续重试同一个 `source_media_asset_id`，不能把同一张源图当成新图反复入队。
- 后续转发执行如果发现源媒体仍是 `pending_cache` / `cache_flood_wait`，只能消费当前已有状态：等待缓存完成、丢弃媒体并降级文本、换已就绪平台素材或跳过；不能在发送现场上传、刷新或人工缓存。
- 只有源消息本身是另一条新消息、相册另一段、或指纹不同，才创建新的源媒体资产。

如果规则选择“等待源媒体缓存完成后继续发送”，执行链路应是事件驱动，而不是 PhotoWorker 直接发送：

```text
转发执行项命中源媒体待缓存
  ↓
执行项进入 waiting_material_cache
  ↓
PhotoWorker / 源媒体缓存 Worker 完成缓存
  ↓
发布 source_media_cached 事件
  ↓
任务中心唤醒同一 source_media_asset_id 关联的等待执行项
  ↓
执行中心重新检查账号、目标、规则、风控、相册顺序和幂等键
  ↓
通过后由发送账号下载缓存文件并重新上传发送
```

约束：

- PhotoWorker 只负责缓存和发出完成事件，不直接调用账号发送。
- 唤醒必须按 `source_media_asset_id`、执行项 ID 和幂等键去重，避免缓存完成事件重复导致重复发送。
- `source_media_cached` 事件必须带 `source_media_asset_id`、`cache_version` / `tg_ref_version_id`、缓存 peer、缓存消息 ID 和完成时间；执行中心只接受与等待执行项当前版本一致的事件，旧事件迟到不能唤醒已经换版本或已结束的执行项。
- 等待必须有超时上限，不能无限停留在 `waiting_material_cache`；默认等待到任务本轮超时，且建议不超过 30 分钟。
- 等待超时后按规则丢弃媒体并降级文本、换已就绪平台素材或跳过，并记录 `material_cache_wait_timeout`。
- 如果执行项已经因为等待超时完成了降级、换素材或跳过，后续缓存才完成也不能再补发；迟到的缓存完成事件只记录为过期事件，不创建新的补发执行项。
- 如果等待期间任务暂停，不补发；任务恢复后再重新检查等待执行项。
- 如果等待期间任务停止、任务结束、源群取消授权、目标不可用、任务时间窗关闭、目标静默期生效、群上下文已过期或规则版本失效，唤醒后只能跳过并记录原因。
- `source_media_cached` 事件可能丢失，因此需要定时兜底扫描：周期性查找 `waiting_material_cache` 且 `source_media_asset_id` 已 ready 的执行项，重新进入唤醒流程。
- 同一源媒体可能对应多个目标执行项；缓存完成事件应唤醒所有关联等待项，但每个目标必须使用自己的执行项幂等键，避免 A 目标成功后影响 B 目标。
- 相册等待按分段状态处理：ready 分段按原 `media_group_index` 顺序发送，失败、超时或不可恢复分段可以直接剔除；不能因为剔除失败图而打乱后续图片顺序。
- 等待队列必须有容量保护：按任务、源群和全局设置最大等待执行项数量；超过上限时直接放弃受影响素材缓存，清理对应等待项，并记录 `material_cache_wait_queue_full`。
- 等待中的执行项不提供人工缓存、人工重新缓存或人工刷新入口；失败就丢弃对应媒体缓存。页面最多展示取消等待、跳过执行项、按规则降级等运行控制动作。
- 如果源消息在成功下载前已被删除，且没有可用 TG media 摘要或已下载临时文件，源媒体资产应标记为 `unrecoverable` 并放弃缓存；如果已经拿到可用摘要或临时文件，则继续尝试缓存同一个 `source_media_asset_id`，直到成功、超时或达到失败阈值。

源媒体缓存终态：

```text
pending_cache
cache_flood_wait
ready
cache_failed
unrecoverable
```

终态规则：

- `cache_failed` 表示本轮缓存失败，需记录失败原因、重试次数和下一次自动队列时间；不提供人工缓存入口。
- `unrecoverable` 表示源消息、外部来源和缓存来源均不可用，只保留元数据、指纹和历史追踪，不再自动等待补发。
- 连续失败达到阈值、等待队列爆满、不可恢复素材暴增或兜底扫描频繁唤醒时，直接把受影响源媒体缓存标记为放弃 / 不可恢复并清理等待项；阈值由系统设置配置。

### 7.2 相册与 media group

Telegram 相册 / media group 必须作为一组有序源消息处理，不能简单拆散后丢失顺序。

监听记录要求：

- 同一组相册记录统一的 `source_media_group_id`。
- 每个媒体段记录 `media_group_index`，从源消息顺序递增。
- 记录 `media_group_total`，用于判断是否采集完整。
- 分别记录每段的媒体类型、caption、指纹、缓存状态和下载状态。
- 相册 caption 策略通过 `album_caption_policy` 固化，例如使用第一条 caption、每段保留各自 caption、或合并 caption。

转发执行默认策略：

```text
按 source_media_group_id 聚合
  ↓
按 media_group_index 排序
  ↓
逐段从 TG 源媒体缓存临时下载
  ↓
由发送账号按原顺序重新上传发送
  ↓
逐段记录远端消息 ID 和失败原因
```

失败处理：

- 默认保持相册顺序，某一段失败后剔除失败图并继续处理后续段，把整体结果标记为“部分失败”。
- 规则可以改为“任一段失败则整体跳过”或“失败段降级为文本说明”，但默认允许去掉失败图继续发。
- 相册源媒体处于待缓存时，ready 分段按原顺序发送；待缓存分段超时、失败或不可恢复时直接丢弃该分段，剩余图片继续按原顺序发送。如果全部分段都失败，再按规则降级文本或跳过。
- 相册分段失败使用 `album_segment_failed`，同时记录失败段 index、媒体类型和原因。
- 相册不能因为中间一张图失败而把后续图片乱序发送。

### 7.3 源媒体授权与保留边界

源媒体缓存只服务已授权源群、已授权任务和对应规则版本。

授权边界：

- 只有已授权监听的源群消息才能写入 TG 源媒体缓存。
- 任务停止不一定立即删除源媒体缓存，但不能继续产生新的源媒体缓存。
- 源群取消授权后，应停止新增缓存，并把历史源媒体缓存标记为待清理。
- 源媒体缓存是转发监听的临时执行资产，可以按 TTL、授权状态、任务状态和队列保护策略直接清理；清理前只保留元数据、指纹、失败原因和发送结果，不需要保留可重新下载的源文件。
- 源媒体缓存默认有保留期，保留期结束且没有执行记录、审计或归因需要时可以清理。
- 被发送记录、审计记录或规则归因引用的源媒体缓存，清理前必须保留元数据、指纹和发送结果。

源媒体缓存清理后，历史记录仍能回答：

```text
源消息来自哪个群
源消息 ID 是多少
媒体类型是什么
媒体指纹是什么
为什么选择保留 / 替换 / 降级
最终发到了哪些目标
```

但不承诺还能重新打开原图。

规则中心对转发监听增加媒体处理策略：

| 策略 | 含义 |
| --- | --- |
| 保留源媒体 | 源消息带图时从 TG 源媒体缓存下载后重新发送。 |
| 只保留文字 | 丢弃源媒体，只发送转换后的文本。 |
| 替换为素材库图片 | 根据源群、关键词、消息类型、标签选择平台素材。 |
| 追加素材 | 原文本后追加一张图片或表情包。 |
| 媒体不通过则跳过 | 源媒体无法获取、类型不允许或缓存失败时跳过。 |
| 媒体失败回退文本 | 图片发送失败时发送文本并记录降级。 |

推荐执行流程：

```text
源群新消息
  ↓
采集文本、caption、媒体类型、媒体指纹
  ↓
输入过滤：文本条件 + 媒体类型条件
  ↓
内容转换：文字清洗 + 媒体策略
  ↓
素材选择：源媒体 / 平台素材 / 无媒体
  ↓
输出校验：caption、链接、素材标签、媒体类型
  ↓
路由和账号策略
  ↓
发送文本、图片、表情包或组合消息
  ↓
记录源事件、素材指纹、素材 ID、规则版本和发送结果
```

转发监听的规则测试器应能模拟：

- 纯文本源消息。
- 图片 + caption 源消息。
- 相册 / media group 源消息，并验证顺序保持。
- 源媒体待缓存，执行项进入 `waiting_material_cache`。
- 等待超时后缓存才完成，验证迟到成功不会补发。
- `source_media_cached` 旧版本事件迟到，验证不会唤醒当前执行项。
- 相册中一张图片缓存或发送失败，验证剔除失败图后继续按原顺序发送剩余图片。
- 等待队列超过容量，验证直接放弃受影响素材缓存并记录 `material_cache_wait_queue_full`。
- 表情包源消息。
- 不允许媒体类型。
- 源媒体保留失败后的回退。
- 关键词命中后替换为指定素材。

## 8. AI 活跃群集成

AI 活跃群发送图片和表情包时，不能让模型自由决定具体文件。推荐做法是：

```text
AI 生成本轮文本、情绪、意图、话题标签
  ↓
规则引擎根据标签、上下文、账号角色和频控选择素材
  ↓
输出校验检查文本、caption、素材标签和目标限制
  ↓
执行层按节奏发送文本、表情包或图片
```

AI 输出结构建议增加：

```json
{
  "content": "这个点有点意思，先蹲一下后续",
  "role": "补充型",
  "intent": "附和",
  "mood": "轻松",
  "material_intent": "表情包:围观",
  "allow_material": true
}
```

规则引擎再根据 `material_intent` 和素材标签选择具体素材，例如：

```text
任务类型 = AI 活跃群
并且 mood = 轻松
并且 material_intent 包含 围观
并且 目标群 30 分钟内未发送表情包
则从标签 表情包,围观 中随机选择 1 个可用素材
```

AI 活跃群素材策略建议：

- 控制每轮素材比例，例如每轮最多 1 条表情包。
- 控制目标群冷却，例如同一群 20 分钟内最多 1 张图。
- 控制账号一致性，例如某账号常用表情包风格不应突变。
- 控制表情包子类型，例如普通群默认允许图片伪表情和静态 sticker，animated sticker、video sticker、custom emoji 需规则显式允许。
- 避免连续图片刷屏。
- 图片和表情包必须可被自动校验跳过、延迟或替换。
- 素材发送失败不应导致整轮 Cycle 失败，默认只跳过该 Turn 或回退文本。

### 8.1 AI 素材意图映射

AI 不直接选择素材 ID，而是输出素材意图。运营人员需要能维护“意图到素材标签”的映射表。

映射表示例：

| 意图 | 匹配标签 | 默认动作 |
| --- | --- | --- |
| 表情包:欢迎 | 表情包,欢迎 | 从可用素材随机 1 个 |
| 表情包:围观 | 表情包,围观 | 目标群冷却通过后发送 |
| 图片:活动 | 图片,活动 | 仅任务显式允许时发送 |
| 无素材 | 无 | 只发送文本 |

映射未命中时：

- 默认不发送素材。
- 可以配置回退到文本。
- 不能让 AI 直接构造文件 URL 或素材 ID。

每次 AI 素材选择需要记录：

- AI 输出的 `material_intent`。
- 命中的映射规则。
- 候选素材标签。
- 候选素材数量。
- 最终素材 ID 和资产版本。
- 未选择素材的原因。

## 9. 规则中心增强

规则中心需要把素材作为规则动作和规则条件的一部分。

新增规则条件：

- 消息类型。
- 媒体类型。
- 是否带图片。
- 是否带表情包。
- 源媒体指纹是否重复。
- 素材标签。
- 素材使用次数。
- 目标群近期素材发送频率。
- 账号近期素材发送频率。
- 同标签素材近期发送频率。
- 同素材跨目标重复发送频率。
- 相册是否完整。
- 相册分段顺序是否完整。
- 表情包子类型。
- 素材缓存是否已就绪。
- 素材缓存账号是否处于 FloodWait。

新增规则动作：

- 选择素材。
- 替换素材。
- 追加素材。
- 丢弃源媒体。
- 保留源媒体。
- 保持相册顺序发送。
- 相册失败图剔除后继续发送剩余图片。
- 降级为文本。
- 素材失败时丢弃媒体 / 换已就绪素材 / 跳过。
- 相册部分失败时继续后续段 / 整体跳过 / 失败段降级文本。
- 源媒体待缓存时等待缓存完成；超时后不补发。
- 等待源媒体缓存超时后丢弃媒体 / 降级文本 / 换已就绪素材 / 跳过。
- 等待队列超量时直接放弃受影响素材缓存。
- 表情包类型不支持时降级为普通图片 / 跳过。

规则版本必须固化素材选择策略。执行项不只记录“命中了规则”，还要记录：

- 规则集版本。
- 素材选择规则。
- 候选素材范围。
- 最终素材 ID。
- 素材指纹。
- 是否使用 Telegram 缓存并完成下载重发。
- 表情包子类型和能力校验结果。
- 相册组 ID、分段顺序和分段结果。
- 等待缓存开始时间、超时时间、唤醒来源和迟到事件处理结果。
- 缓存事件版本、等待队列处理结果和放弃缓存原因。
- 失败和回退原因。

## 10. 风控与频控

素材发送会比纯文本更容易触发目标群管理规则和账号行为异常，因此需要单独控制：

- 账号媒体发送小时 / 日上限。
- 目标群媒体发送小时 / 日上限。
- 同一素材重复发送冷却。
- 同一素材跨群发送冷却。
- 同一素材标签冷却。
- 同一目标连续图片间隔。
- 同一账号连续图片间隔。
- 源媒体重复转发去重。
- 相册批量媒体发送间隔和组内顺序保护。
- animated sticker、video sticker、custom emoji 的独立频控。
- 图片带链接或二维码时的额外输出校验。
- 大文件、视频、压缩包等高风险类型默认关闭，按规则显式允许。
- 素材缓存账号 FloodWait 不进入发送现场等待，只影响素材库刷新队列。

默认策略建议：

```text
普通消息发送：允许人工选择图片，但发送前强校验。
转发监听：默认只保留文字，需显式开启保留源媒体或素材替换。
AI 活跃群：默认允许低频表情包，不默认连续发图片。
文件 / 视频：默认不进入自动化任务，除非规则集显式允许。
```

## 11. 页面设计

素材库页面必须作为一级“素材中心”，不能放在系统设置或 AI 内容中心下的附属 Tab。表情包和头像包属于高频运营资产，需要批量上传、分组、缓存健康和引用关系的独立工作台。

页面结构：

```text
素材库
  ├─ 素材总览
  ├─ 表情包库
  ├─ 头像包
  ├─ 图片 / 文件 / 组合消息
  ├─ 实时上传 / 批量上传
  ├─ TG 缓存状态
  ├─ 标签 / 分组管理
  ├─ 使用记录
  └─ 失败与回退记录
```

素材列表字段：

- 标题。
- 类型。
- 标签。
- 状态。
- 预览状态；默认从 TG 缓存临时取预览，不保存原图。
- 文件大小。
- TG 缓存状态。
- 使用次数。
- 最近使用时间。
- 最近失败原因。
- 操作：预览、编辑、刷新缓存、停用、查看使用记录。
- 引用数量和运行中影响提示。

### 11.1 预览机制

素材预览不能把图片长期落到平台磁盘。

预览来源优先级：

```text
TG 缓存消息临时拉取
  ↓
外部 URL 临时代理
  ↓
无预览，仅展示类型、尺寸、指纹和缓存状态
```

预览方式：

- 后端校验 `materials.view` 权限。
- 后端通过缓存账号从 TG 缓存消息临时读取媒体。
- 后端以流式响应或短期签名地址返回给前端。
- 预览临时文件如必须生成，也只能进入最长 10 天 TTL 临时目录，成功响应后优先立即删除。
- 预览响应不进入长期存储，不进入备份，不作为业务恢复来源。
- 预览失败只影响页面查看，不直接代表素材不可发送；发送仍以 TG 缓存发送校验为准。

预览审计建议：

- 普通缩略预览可只记访问统计。
- 查看原尺寸、下载、批量导出等高风险动作需要单独权限和审计。

新增素材弹窗：

- 素材类型。
- 文件上传 / URL / 从历史消息收藏。
- 标题。
- 标签。
- 默认 caption。
- 是否立即上传到 TG 缓存。
- 缓存账号。
- 素材缓存位置。
- 上传安全校验结果。
- 是否允许自动化任务使用。
- 是否仅保存 TG 缓存位置和引用摘要。

任务配置中的素材选择不应只给一个下拉框。应支持：

- 固定选择指定素材。
- 按标签随机。
- 按规则选择。
- 按源消息保留。
- 禁用素材。

### 12.1 备份与缓存失效边界

平台数据库备份只能保证恢复：

- 素材基础信息。
- 标签、caption、规则绑定关系。
- 内容指纹和媒体元数据。
- 资产版本和 TG 引用版本。
- TG 缓存 peer、缓存消息 ID、引用状态和可重发状态。
- 发送记录、审计记录和归因记录。

平台数据库备份不保证恢复：

- 图片原文件。
- 表情包原文件。
- 视频或文件原件。
- 已被 Telegram 删除的缓存消息。
- 已失效且没有外部可恢复 URL 的媒体本体。

数据库恢复或缓存失效后的处理：

```text
恢复数据库
  ↓
检查缓存账号和缓存 peer 是否可访问
  ↓
抽检 TG 缓存消息
  ↓
可访问：素材恢复为可用或需刷新，并验证可下载重发
  ↓
不可访问：如有外部可恢复 URL，则重新导入到 TG 缓存
  ↓
仍不可用：标记不可恢复，只保留历史追踪
```

因此，平台不承诺 TG 缓存灾备。重要素材如果必须长期可恢复，应由业务侧提供外部可恢复 URL 或重新上传；否则平台只能保证“可追踪”，不能保证“可重新发送”。

### 12.2 缓存健康监控

素材库需要一个缓存健康汇总入口。

核心指标：

- 可用素材数。
- 需刷新素材数。
- 不可恢复素材数。
- 缓存账号异常数。
- 缓存 peer 不可访问数。
- 缓存消息疑似删除数。
- 待刷新队列长度。
- 最近刷新失败原因分布。
- 热门素材缓存就绪率。
- 执行期下载重发成功率。
- 缓存账号 FloodWait 累计时长。
- 等待源媒体缓存执行项数量。
- 缓存完成事件唤醒数量和兜底扫描唤醒数量。
- 等待缓存超时数量。
- 等待队列爆满放弃素材缓存数量。
- 不可恢复素材暴增触发保护次数。
- 兜底扫描频繁唤醒触发保护次数。

页面入口：

```text
素材库
  ├─ 素材列表
  ├─ TG 缓存健康
  ├─ 刷新队列
  ├─ 不可恢复素材
  └─ 缓存清理记录
```

健康检查不能直接下载全部图片，只做轻量抽检和必要的下载重发前校验，避免把“不存图”设计变成批量拉图。

## 13. 实施顺序

建议按低风险顺序落地：

1. 素材模型增强：补状态、媒体元数据、指纹、TG 缓存引用、`delivery_mode=download_reupload`、`emoji_asset_kind`、`cache_ready_status`，并明确平台不持久化原图。
2. 资产版本和引用保护：固化 `asset_version_id`，禁止被引用素材物理删除。
3. 素材上传接口：本地上传临时中转、zip 图片包异步导入、URL 入库、安全校验、去重和临时文件清理。
4. TG 缓存上传：用指定账号上传到素材缓存会话，建立刷新队列并保存引用。
5. TG 缓存健康与清理：补缓存健康检查、清理策略和失效降级边界；不做 TG 缓存灾备和备用缓存引用承诺。
6. 消息发送中心：支持图片、表情包、组合消息的下载重发和分段记录。
7. 规则中心：补媒体类型条件、素材选择动作、相册顺序策略、表情包类型策略、AI 意图映射和规则测试器。
8. 转发监听：源媒体捕获时上传到 TG 源媒体缓存，支持源媒体保留、相册顺序保持、失败图剔除、源媒体缓存直接清理、素材替换和媒体失败回退。
9. AI 活跃群：支持 `material_intent`，由规则选择低频表情包，并按 `emoji_asset_kind` 做能力校验。
10. 数据与审计：补素材归因、资产版本归因、TG 引用版本归因、失败回退、缓存刷新记录。

不建议第一步就做复杂 AI 选图。先把素材可上传、可缓存、可发送、可追踪打牢，再把规则引擎接入 AI 活跃群和转发监听。
