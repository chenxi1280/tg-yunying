# TG 运营管理平台新版功能架构设计与实现对齐检查

> 本文为新版项目功能架构设计与当前实现对齐稿。<br>
> 重点描述系统模块、功能边界、任务模型、页面规划、执行流程、当前落地状态和后续优化优先级。<br>
> 本文不包含数据库表结构设计。

---

## 1. 项目定位

TG 运营管理平台是一个面向 Telegram 运营团队的统一后台，用于集中管理：

- TG 开发者应用
- TG 账号与账号分组
- 群、频道、联系人等 TG 资产
- 运营目标
- 消息发送
- 自动化任务
- 频道互动
- AI 群活跃
- 转发监听
- 监听事件
- 系统级规则
- AI 内容生成
- 归档
- 运营数据
- 执行记录与审计

平台的核心目标是：

> 把 TG 账号、群频道目标、消息发送、频道互动、AI 群活跃、转发监听、归档、数据统计和审计统一到一个可运营、可追踪、可控制的后台系统里。

新版系统不再以旧的多租户 SaaS、卡密、订阅套餐、旧 Campaign、AI 草稿审核流作为主线。

新版主线是：

```text
账号接入
  ↓
资产同步
  ↓
目标确认
  ↓
规则配置
  ↓
任务创建
  ↓
监听 / AI / Worker 执行
  ↓
任务详情追踪
  ↓
数据复盘
  ↓
审计留痕
```

## 0. 当前实现对齐结论（2026-05-12）

本次检查以当前代码为准，重点核对了前端主导航、页面视图、后端路由、任务中心模型、监听中心、规则中心和运营数据接口。结论是：新版主线已经基本落到当前页面与接口里，但仍有几处需要继续收敛，文档后续章节按这个口径理解。

### 0.1 已经对齐的部分

- 前端主导航已经对齐为：运营概览、TG账号管理、运营目标、消息发送、任务中心、监听中心、规则中心、归档中心、运营数据、系统设置、审计记录。
- 后端默认主路由已经覆盖：账号、账号池、开发者应用、AI 配置、运营目标、消息发送、任务中心、监听与规则汇总、运营数据、归档和审计。
- 旧 Campaign、旧 operation-task、旧 review 路由已经从主路径隔离，只有显式打开兼容配置时才进入旧口径。
- 任务中心当前主任务类型为 5 类：AI 活跃群、转发监听群、频道消息浏览、频道消息点赞、频道消息评论/回复。
- 任务中心已经支持创建、创建并启动、启动、暂停、继续、停止、重试、重置、编辑配置、删除、详情查看和执行记录追踪。
- 频道浏览、点赞、评论/回复任务默认按 `dynamic_new` 持续采集新消息执行，支持容量检查、每条频道消息执行明细、账号执行项和容量不足提示。
- AI 活跃群已经按 Cycle / Turn 记录发言计划，支持账号角色映射、近期账号记忆、长期账号画像和 AI 生成记录，并在发送前做内容过滤、重复过滤、上下文采集和账号冷却控制。
- 转发监听群已经接入源群上下文采集、规则集版本、过滤、转换、路由、去重、目标群发送项和转发批次追踪。
- 监听中心已有独立页面和 `/api/listeners/summary` 汇总接口，按频道 / 群聚合展示关联任务、监听账号、事件积压、最后事件和错误。
- 监听中心已经补充备用监听账号、切换建议、最近事件、订阅事件类型、待分发执行项、去重键统计，以及群监听和频道监听的备用账号切换入口。
- 规则中心已有独立页面和规则集版本管理，支持常用规则模板生成、过滤、转换、运营目标路由、账号策略、限速、重试策略 JSON 配置，以及按规则版本进行过滤、转换、路由、账号和限速预览。
- 规则中心已经补充规则命中统计，按转发监听执行项聚合规则集版本、关联任务数、执行项数、成功、失败、跳过、待执行和最近命中时间。
- 规则中心已经补充规则冲突与绑定缺口提示，可发现未发布规则集、转发任务绑定缺失或绑定版本不一致等问题。
- 运营数据已有 `/api/operation-metrics/summary` 真实汇总接口，不再只是空壳入口，并已补充账号、目标、任务和失败执行项明细定位。
- 审计记录已经补充详情抽屉和 CSV 导出，可查看并导出动作、对象、操作人、来源 IP、详情和追溯口径。
- 运营目标列表和详情已经展示可发、可监听、可归档、可创建任务能力；群目标可区分 AI 活跃、转发监听源群、转发目标群和群归档，频道目标可区分浏览、点赞、评论/回复。
- 本地浏览器冒烟验证已覆盖运营概览、运营目标、任务中心、监听中心、规则中心、运营数据和审计记录；归档列表接口已通过旧库字段补齐迁移修复。
- 任务中心群类任务创建和编辑已经以运营目标 ID 作为主引用，前端新版创建/编辑 payload 只提交 `target_operation_target_id`、`source_operation_target_ids` 和 `target_operation_target_ids`，后端会解析为执行层所需的群资产 ID，并保留运营目标 ID 用于追溯；AI 活跃群、转发监听群和监听运行层也能在只有运营目标 ID、没有预归一化群 ID 的情况下直接解析到授权群资产，任务摘要会优先展示运营目标名称。执行动作 payload 会带 `operation_target_id` / `source_operation_target_id` 便于详情、审计和运营数据追溯，转发批次明细也会展示规则命中摘要和重试次数，旧 `target_group_id` / `group_id` 兼容路径继续保留。

### 0.2 仍需继续优化的部分

- 运营目标仍然处在 `groups` 和 `operation-targets` 双轨过渡期：频道任务、群类任务新建/编辑入口、群管理“用此群创建任务”、归档和消息发送里的群/频道目标已经优先使用运营目标，群类任务前端创建/编辑已经只提交运营目标 ID，任务执行动作也会保留运营目标 ID 追溯字段，消息发送的手动输入已收窄为个人目标；AI 活跃群、转发监听群和监听运行层已经支持运行时从运营目标 ID 直接解析授权群资产。执行层仍需要群资产承载 TG 授权、账号权限和上下文采集关系，后续主要是继续减少旧群 ID 兼容字段暴露。
- 监听中心当前已经有聚合状态、采集节流、订阅事件类型、待分发统计、去重键统计、备用账号建议、群监听切换和频道任务账号切换；任务中心 drain 已接入独立监听运行层，会按源群聚合订阅任务、统一采集上下文，并在采集到新事件后唤醒订阅任务，同时可在存在可用账号时自动恢复监听账号。投递守护已能给执行项写入 `lease_owner` / `lease_expires_at` 执行租约，把过期租约、持有者 worker 心跳过期或老式长时间 `executing` 执行项标记为超时失败，记录恢复阶段、恢复原因、恢复时间、原始结果快照、最近恢复执行项、恢复次数和上一个租约持有者，并交给既有重试策略；任务中心 drain 也会写入 `worker_heartbeats` 进程心跳，运营数据风控指标可看到 2 分钟内活跃和过期 worker。发送失败时也会按默认失败策略处理账号受限、API 限流和内容拦截。
- 规则中心当前已经有常用规则模板、多条件过滤、组合条件表达式、默认运营目标路由、源群映射、关键词路由、源群+关键词路由、账号权重、冲突提示、规则效果钻取、14 天趋势、近 7 天 / 前 7 天转化对比、规则版本 × 目标群 × 发送账号跨维归因，以及转发执行项里的过滤/转换/路由/账号策略命中摘要；转发批次层也会统计源事件数、素材指纹数和规则版本数，单条执行项保留源事件归因键与素材指纹，并可从规则跨维归因区域导出转发归因 CSV。规则中心页面已经补充素材归因报表，按素材指纹聚合源事件、目标、账号、执行项、成功率、失败/跳过和重试。复杂嵌套表达式已经支持通过 JSON 递归配置，拖拽式规则编排仍可继续增强。
- 频道评论和频道回复当前仍在一个 `channel_comment` 类型里承载，前端显示为“频道消息评论/回复”；任务配置已支持选择“评论频道消息 / 回复指定评论 / 评论+回复”。运营目标详情可从频道消息行同步讨论区评论树，系统会缓存一级评论和二级回复对象，任务配置可从已采集评论中可视化选择回复对象，也保留手动输入消息 ID 兼容；执行项 payload 会区分 `comment_mode`、`reply_to_message_id` 和 `reply_target_label`。独立频道回复任务类型仍可作为后续增强，但当前主流程已经能完成二级回复对象采集、选择和执行。
- 系统仍保留少量兼容字段和兼容模型，例如 tenant、ReviewQueue、旧 Campaign 服务、旧用量字段等。它们不应再进入新版主流程，但可以作为迁移兼容层逐步清理。
- 系统设置当前包含 TG 开发者应用、AI 与规则基础配置；发送节奏、全局静默时段和默认失败处理策略已经中心化到发送节奏配置，并已接入任务中心调度。运营数据页已新增全局风控中心和轻量策略抽屉，可查看并编辑发送抖动、批次间隔、发送窗口、静默时段、默认失败处理、账号跨目标小时/日限额、账号全局冷却和敏感词启停；运营目标详情已经补充群目标级风控策略和目标账号覆盖编辑，可维护发送时间窗、群日上限、账号/群冷却、敏感词、链接白名单、发送前校验，并可按目标切换账号发言/监听权限；素材仍在 AI 与规则基础配置中维护。
- AI 活跃群已经支持按账号 ID 配置稳定角色，会把同一任务内和跨任务的账号最近成功发言作为下一轮账号记忆，并按账号聚合历史成功发言、常用角色、常见意图和近期表达形成长期画像；生成前会额外汇总主线方向、最近真人上下文和上一轮 AI 已说内容，形成“话题脉络”，并按本轮发言数生成“承接 / 补充 / 互动 / 收束”的多轮话题计划，写入提示词、执行项和任务详情。更强的长期主题编排仍可继续增强。

## 2. 产品边界

平台只围绕自有、授权、可运营的 TG 账号和目标设计。

系统需要明确：

- 只能使用已接入账号。
- 只能操作已确认的运营目标。
- 只能监听已授权源群、群聊、频道或讨论组。
- 任务必须有时间窗口、账号池、频控和执行记录。
- 所有发送、监听、过滤、转换、AI 生成、失败、重试都必须留痕。

系统不是简单的批量刷量工具，而是一个面向运营流程的后台。

新版功能设计的核心原则是：

```text
可配置
可自动执行
可追踪
可暂停
可重试
可解释
可风控
可审计
```

特别说明：

> AI 活跃群和转发监听群不做人工审核。  
> 系统通过自动校验、规则过滤、风控、限速、上下文检查、失败重试来保证任务可控。

## 3. 新版系统总架构

新版系统拆成 10 个功能中心。

1. 运营概览中心
2. 系统初始化中心
3. TG 账号中心
4. 运营目标中心
5. 消息发送中心
6. 任务中心
7. 监听中心
8. 规则中心
9. AI 内容中心
10. 数据、归档与审计中心

它们之间的关系是：

```text
系统初始化中心
  提供 TG 开发者应用、AI 供应商、全局规则、风控、素材等基础配置。

TG 账号中心
  提供可登录、可同步、可发送、可监听的账号资源。

运营目标中心
  把账号同步到的群、频道、联系人整理成可运营目标。

规则中心
  提供过滤、转换、路由、限速、拦截、发送账号分配等系统级规则。

AI 内容中心
  提供群上下文分析、批量对话生成、评论生成、回复生成、转发改写等能力。

监听中心
  负责监听频道、讨论组、源群、群上下文、Reaction、评论和回复事件。

任务中心
  负责任务创建、配置、启动、暂停、重置、停止和详情追踪。

消息发送中心 / 执行中心
  负责把任务生成的发送计划真正执行出去。

数据与审计中心
  负责统计、执行记录、失败追踪、运营复盘和敏感操作留痕。
```

- AI 模型配置

### 6.1 TG 开发者应用

TG 开发者应用用于账号登录和 Telethon 接入。

需要支持：

- 新增开发者应用
- 编辑 api_id / api_hash
- 健康检查
- 启用 / 禁用
- 绑定到 TG 账号
- 异常提示

没有可用开发者应用时：

- 不能新增 TG 账号
- 不能启动账号登录
- 不能执行依赖账号的任务

### 6.2 AI 供应商配置

AI 供应商用于：

- AI 活跃群
- 频道评论生成
- 频道回复生成
- 转发内容改写
- 群上下文摘要
- 内容自动校验
- 账号一致性检查

支持：

- OpenAI-Compatible 供应商
- 模型选择
- API Key 检查
- 默认模型
- 备用模型
- 失败回退
- Token / 费用统计

### 6.3 全局风控配置

全局风控用于控制所有任务边界。

当前实现状态：

- 发送节奏配置已经支持最小/最大抖动、批次间隔、是否遵守发送时间窗。
- 已补充全局静默时段和默认失败处理策略，任务中心新建任务会使用这些配置作为默认值，任务自身仍可覆盖；发送调度会执行账号跨目标小时/日限额、账号全局冷却、账号受限默认处理和 API 限流等待重试/暂停策略。
- 运营数据页已经汇总全局风控状态：静默期、默认失败策略、敏感词规则、群日上限、账号/群冷却、链接白名单、任务小时/日上限、最近内容拦截、频控拦截和重复指纹。
- 运营数据页已经补充全局风控策略抽屉，可编辑最小/最大发送抖动、批次间隔、是否遵守发送窗口、全局静默时段、默认重试/失败处理策略、账号跨目标小时/日限额和账号全局冷却，并可新增或启停敏感词规则。

## 10. 任务中心总架构

任务中心是新版最核心模块。

新版任务中心分成三条核心任务主线：

1. 频道互动任务
2. AI 活跃群任务
3. 转发监听群任务

三个任务类型的执行模型不同。

```text
频道互动任务：
父任务 → 目标频道消息 → 动作子任务 → 账号执行项

AI 活跃群任务：
父任务 → 活跃循环 Cycle → 发言 Turn → 账号发送项

转发监听群任务：
父任务 → 源群事件 → 规则处理 → 转发批次 → 目标群发送项
```

## 11. 任务中心通用能力

所有任务都要支持：

- 创建
- 保存草稿
- 启动
- 暂停
- 继续
- 停止
- 重试
- 重置
- 编辑
- 删除
- 查看详情
- 查看执行记录
- 查看失败原因
- 查看审计记录

### 11.1 任务统一状态

建议统一任务状态：

- 草稿
- 已排期
- 运行中
- 已暂停
- 已达目标但未结束
- 收尾中
- 已完成
- 人工停止
- 异常失败
- 已删除

核心规则：

```text
任务不会因为提前达标就自然结束。
任务自然结束只由 end_at 控制。
管理员可以人工停止。
任务失败后可以重试或重置。
```

即：

```text
达到目标数量 ≠ 任务完成
到达结束时间 = 任务自然完成
```

### 11.2 重试、重置、暂停、停止的区别

前端必须解释清楚：

```text
暂停：
暂时停止继续执行，可恢复。

重试：
只重试失败项，不影响已成功项。

重置：
重新规划未完成部分，但不会重复执行已成功动作。

停止：
任务彻底停止，不再自动恢复。

删除：
隐藏或删除任务记录，正式任务建议保留审计。
```

## 12. 频道互动任务功能架构

频道互动任务包括：

- 频道消息浏览
- 频道消息点赞
- 频道消息评论/回复

当前实现说明：

- 当前后端任务类型是 `channel_view`、`channel_like`、`channel_comment`。
- `channel_comment` 同时承载评论和回复口径，前端展示为“频道消息评论/回复”。
- 当前可在任务配置里选择评论、回复或混合模式；回复模式要求填写回复目标消息 ID，执行项会保留 `comment_mode`、`reply_to_message_id` 和 `reply_target_label`，网关会把回复目标传给频道评论发送层。
- 频道消息详情支持同步讨论区评论树，后端用 `channel_message_comments` 缓存一级评论、二级回复、作者、父级回复消息 ID、内容预览、回复数和发布时间。
- 任务中心回复对象字段支持从已采集评论中选择，也支持继续手动输入 Telegram 消息 ID；这样在不新增单独 `channel_reply` API 类型的情况下，已经能完成“采集评论对象 -> 点选回复对象 -> 生成回复执行项 -> 网关按 reply_to 发送”的闭环。
- 独立的“频道回复”任务类型、完整评论树可视化折叠视图可以作为后续体验增强，但不是当前主流程阻塞项。

新版重点是：

> 一个频道任务下面，每条频道消息的点赞、评论、回复都应该是独立子任务。

### 12.1 频道任务结构

频道任务结构：

```text
父任务
  ↓
目标频道消息
  ↓
动作子任务
  ↓
账号执行项
```

示例：

```text
父任务：频道最新 5 条消息点赞

消息 101 点赞子任务
  - 账号 A 执行
  - 账号 B 执行
  - 账号 C 执行

消息 102 点赞子任务
  - 账号 A 执行
  - 账号 D 执行
  - 账号 E 执行
```

如果是评论任务：

```text
父任务：频道最新 5 条消息评论

消息 101 评论子任务
消息 102 评论子任务
消息 103 评论子任务
```

如果是回复任务：

```text
父任务：频道消息回复任务

消息 101 / 评论 A 回复子任务
消息 101 / 评论 B 回复子任务
消息 102 / 评论 C 回复子任务
```

### 12.2 创建频道任务流程

```text
选择任务类型
  ↓
选择频道目标
  ↓
选择消息范围
  ↓
选择动作类型
  ↓
配置每条消息目标数
  ↓
选择账号池
  ↓
配置节奏和风控
  ↓
容量检查
  ↓
确认创建
```

消息范围支持：

- 指定消息
- 最新 N 条
- 日期范围
- 动态新消息

当前默认值：

- 频道浏览、频道点赞、频道评论/回复默认使用动态新消息。
- 动态新消息任务不会因为某一轮达到数量就自然结束，而是按监听间隔继续采集；只有到达任务结束时间或人工停止才结束。

动作类型支持：

- 浏览
- 点赞 / Reaction
- 评论/回复

### 12.3 子任务生成逻辑

例如创建：

```text
频道最新 10 条消息
每条消息点赞 50 个
```

系统生成：

```text
10 个点赞子任务
每个子任务目标 50 个账号执行
```

例如创建：

```text
频道最新 10 条消息
每条消息评论 10 条
```

系统生成：

```text
10 个评论子任务
每个子任务目标 10 条评论
```

### 12.4 去重规则

频道任务必须强去重。

规则：

```text
同一个账号
同一个频道
同一条消息
同一种动作
只能有效执行一次
```

即使任务被重置，也不能让同一个账号重复对同一条消息执行同一动作。

这能防止：

- Worker 重复消费
- 任务重置导致重复执行
- 跨任务重复动作
- 账号重复点赞同一消息
- 账号重复评论同一消息

### 12.5 频道监听优化

频道任务不能每个任务单独监听频道。

新版采用：

```text
一个频道只启动一个监听实例。
多个任务订阅这个频道事件。
```

例如同一个频道同时存在：

- 点赞任务
- 评论任务
- 回复任务
- 动态新消息任务

系统只监听一次频道和讨论组。

监听到事件后：

```text
频道监听中心接收事件
  ↓
标准化事件
  ↓
事件去重
  ↓
分发给订阅该消息 / 动作的子任务
  ↓
更新子任务进度
  ↓
更新父任务总进度
```

### 12.6 频道任务详情页

频道任务详情页要围绕“消息子任务”展示。

包含：

- 任务总览
- 消息子任务
- 账号执行项
- 监听状态
- 容量分析
- 失败与重试
- 审计记录

消息子任务里要看到：

- 每条消息目标多少
- 已完成多少
- 失败多少
- 重复多少
- 哪些账号已执行
- 哪些账号失败
- 为什么失败
- 是否已达标
- 是否容量不足

示例展示：

```text
消息 101：点赞目标 50，已完成 38，失败 2，重复 1，状态运行中
消息 102：点赞目标 50，已完成 50，失败 0，重复 0，状态已达标
消息 103：点赞目标 50，已完成 12，失败 5，重复 2，状态容量不足
```

## 13. AI 活跃群任务功能架构

AI 活跃群是循环型任务。

它不是一次性群发，也不是简单监听转发，而是：

```text
根据群上下文
周期性生成自然对话
分配给不同账号
按节奏发送
一轮完成后再进入下一轮
```

### 13.1 AI 活跃群任务结构

```text
父任务
  ↓
活跃循环 Cycle
  ↓
本轮发言计划 Turn
  ↓
账号发送
```

核心规则：

```text
每一轮先拉取群上下文。
一次性生成多个账号的对话内容。
分配给不同账号按节奏发送。
本轮全部发送完成、取消或过期后，才进入下一轮。
```

这样可以：

- 节约 Token
- 保证对话连贯
- 避免多个账号单独生成导致重复
- 保证账号逻辑一致
- 控制发言节奏

### 13.2 创建 AI 活跃群任务流程

```text
选择 AI 活跃群任务
  ↓
选择授权群聊
  ↓
选择账号池
  ↓
选择监听账号
  ↓
配置活跃账号比例
  ↓
配置每轮参与账号数
  ↓
配置每轮发言数
  ↓
配置活跃时间段
  ↓
配置静默时间段
  ↓
配置每日爬坡策略
  ↓
配置话题方向和语气
  ↓
配置上下文范围
  ↓
配置自动校验和风控
  ↓
确认创建
```

### 13.3 活跃账号比例

支持配置：

- 活跃账号比例
- 每轮最少参与账号数
- 每轮最多参与账号数
- 每轮最少发言数
- 每轮最大发言数
- 单账号每轮最多发言数
- 单账号冷却时间

示例：

```text
账号池 40 个
活跃比例 30%
每轮最多 10 个账号
```

系统本轮选择：

```text
10 个账号参与
```

如果账号池较小，则按最小参与数兜底；如果账号池较大，则按最大参与数限制。

### 13.4 一次生成多账号对话

AI 不应该每个账号单独生成一次。

正确方式是：

```text
拉取一次群上下文
  ↓
AI 一次生成一轮对话计划
  ↓
计划里包含多条发言
  ↓
每条发言带角色、意图、延迟、回复对象
  ↓
系统再分配给具体账号
```

本轮对话计划可以包含：

- 第 1 条：引导型账号，回应用户问题
- 第 2 条：补充型账号，补充说明
- 第 3 条：提问型账号，引出讨论
- 第 4 条：总结型账号，做轻量总结

AI 输出的不是一段自由文本，而是本轮对话计划。

### 13.5 账号一致性

账号可以重复发送，但要保证行为一致。

每个账号需要有稳定角色：

- 引导型
- 解答型
- 补充型
- 新人型
- 总结型
- 轻松闲聊型

同一个账号要保持：

- 语气一致
- 角色一致
- 话题记忆一致
- 不要前后观点冲突
- 不要重复自己刚说过的话
- 不要突然从新人变成专家
- 不要连续刷屏

系统需要记录：

- 账号最近说过什么
- 今天说了多少
- 最近参与哪些话题
- 上次发言时间
- 当前冷却状态
- 近期失败情况

### 13.6 静默时间段

静默期不是完全停止，而是进入低频模式。

静默期规则：

- 只允许少量账号参与
- 通常 4～5 个账号以内
- 每轮只发少量消息
- 间隔时间拉长
- 不主动制造复杂话题
- 不密集互相回复
- 连续无人响应后逐步停止
- 最后进入休眠，只监听不发言

静默期适合：

- 轻量回应
- 收尾
- 简单提醒
- 低打扰话题
- 承接白天话题

静默期不适合：

- 大量提问
- 密集讨论
- 长篇解释
- 强推广
- 频繁互相回复

### 13.7 每日爬坡

每天活跃时间开始后，不应该马上全量启动。

应该分阶段：

```text
启动期：少量账号，少量消息
预热期：逐步增加账号和消息
正常期：按配置比例执行
收尾期：逐步降低
静默期：低频或休眠
```

示例：

```text
09:00 - 09:30 启动期
09:30 - 10:30 预热期
10:30 - 22:30 正常期
22:30 - 23:00 收尾期
23:00 后 静默期
```

每日重新启动时，系统应先观察群内真实活跃度，再逐步提高 AI 发言量。

### 13.8 上下文快照

每轮开始时，系统先拉取群上下文。

上下文包括：

- 最近群消息
- 上一轮 AI 发言
- 真实用户问题
- 管理员消息
- 群当前话题
- 群活跃度
- 账号角色记忆
- 风险话题
- 今日已发送情况

AI 生成要基于这份上下文快照。

这样后续可以追踪：

- AI 为什么生成这句话
- 基于哪些群消息生成
- 是否和当时话题一致

### 13.9 上下文过期处理

AI 生成内容后，不是无脑全部发完。

每条发送前需要检查：

- 群里是否出现大量新消息
- 话题是否明显变化
- 管理员是否发了新消息
- 是否出现敏感内容
- 是否有人明确要求停止

如果上下文过期：

```text
取消当前 Cycle 剩余未发送内容
标记为上下文过期
重新拉取上下文
重新生成下一轮
```

### 13.10 AI 活跃群不走人工审核

AI 活跃群消息不需要人工审核后发送。

流程是：

```text
AI 生成
  ↓
系统自动校验
  ↓
通过后自动发送
```

不再有：

- 待审核
- 人工审核
- 审核通过
- 审核驳回
- 审核队列

保留的是自动校验。

自动校验包括：

- 空内容检查
- 敏感词检查
- 重复内容检查
- 长度检查
- 外链检查
- @ 成员检查
- 账号冷却检查
- 账号发送上限检查
- 目标群频控检查
- 上下文是否过期检查
- AI 输出格式检查

自动校验结果：

- 通过：进入发送队列
- 跳过：不发送，记录原因
- 拦截：不发送，记录原因
- 重生成：整轮内容异常时重新生成
- 延迟：账号冷却或频控导致延后发送

### 13.11 AI 活跃群详情页

详情页包含：

- 任务总览
- 当前模式
- 当前轮次
- 今日发言数
- AI 消息占比
- 当前参与账号
- 下一轮时间
- 活跃循环列表
- 本轮发言计划
- 上下文快照
- 账号角色与记忆
- 自动校验结果
- 发送记录
- 失败与跳过原因
- AI 生成记录（按 Cycle 展示生成条数、Token、上下文消息数和账号记忆数，并展示话题脉络、话题计划、账号长期画像摘要）
- 审计记录

运营人员要能看到：

- AI 是根据什么上下文生成的
- 本轮计划发哪些话
- 哪些已经发了
- 哪些还没发
- 哪些被自动跳过
- 为什么取消
- 为什么降频
- 哪个账号说过什么

## 14. 转发监听群任务功能架构

转发监听群是事件驱动任务。

它的目标是：

> 监听一个或多个源群，源群有新消息后，系统根据系统级规则判断是否转发、如何转换、发到哪些目标群、由哪些账号发送。

### 14.1 转发监听群任务结构

```text
父任务
  ↓
源群监听
  ↓
源消息事件
  ↓
过滤规则
  ↓
转换规则
  ↓
路由规则
  ↓
目标群发送
```

完整流程：

```text
源群出现新消息
  ↓
监听中心捕获消息
  ↓
生成源消息事件
  ↓
匹配任务订阅
  ↓
执行过滤规则
  ↓
通过后执行转换规则
  ↓
执行路由规则
  ↓
分配发送账号
  ↓
自动发送到目标群
  ↓
记录结果
```

### 14.2 创建转发监听群任务流程

```text
选择转发监听群任务
  ↓
选择一个或多个源群
  ↓
选择一个或多个目标群
  ↓
选择系统级规则集
  ↓
选择规则集版本
  ↓
选择发送账号池
  ↓
选择监听账号
  ↓
配置发送策略
  ↓
配置突发消息处理
  ↓
配置失败重试策略
  ↓
确认创建
```

### 14.3 源群选择

支持：

- 单个源群
- 多个源群
- 源群分组

源群必须满足：

- 已授权监听
- 系统账号可访问
- 存在可用监听账号

### 14.4 目标群选择

支持：

- 单个目标群
- 多个目标群
- 目标群分组
- 按规则路由到不同目标群

目标群必须满足：

- 已授权运营
- 有账号可发言
- 目标未处于风控冻结

### 14.5 系统级规则集

转发监听的过滤和转换规则必须作为系统级配置。

任务创建时不直接临时写规则，而是选择：

- 规则集
- 规则集版本

规则中心维护：

- 过滤规则
- 转换规则
- 路由规则
- 发送账号策略
- 限速策略
- 拦截策略
- 重试策略

这样好处是：

- 规则统一管理
- 多个任务复用规则
- 规则可测试
- 规则可版本化
- 规则变更可审计
- 规则出问题可以回滚

### 14.6 过滤规则

过滤规则决定源消息要不要处理。

支持：

- 关键词包含
- 关键词排除
- 消息类型过滤
- 发送人过滤
- 管理员消息过滤
- 链接过滤
- 长度过滤
- 重复内容过滤
- 敏感词过滤
- 时间段过滤

过滤结果：

- 通过：继续转换和发送
- 忽略：不发送，只记录原因
- 拦截：不发送，只记录原因
- 规则异常：不发送，记录失败

没有人工审核结果。

### 14.7 转换规则

转换规则决定源消息如何变成目标群要发送的内容。

支持：

- 原样复制
- 去掉链接
- 替换链接
- 关键词替换
- 添加前缀
- 添加尾注
- 删除 @
- 删除来源署名
- 格式整理
- 图片保留或丢弃
- 长文本拆分
- 短消息合并

P0 阶段先做固定转换规则。

后续可以支持 AI 改写，但 AI 改写也不走人工审核，而是：

```text
AI 改写
  ↓
自动校验
  ↓
通过后发送
```

### 14.8 路由规则

路由规则决定消息发到哪些目标群。

支持：

- 全部目标群分发
- 源群到目标群映射
- 关键词路由
- 消息类型路由
- 目标标签路由
- 规则命中路由

示例：

```text
包含“公告” → 发到公告群
包含“活动” → 发到活动群
源群 A 的消息 → 发到目标群 1、2
源群 B 的消息 → 发到目标群 3
```

### 14.9 发送账号分配

转发监听群输出通过 TG 账号发送到目标群，可以由不同账号发送，也允许账号重复发送，但要保证账号发送逻辑的一贯性。

建议默认采用：

```text
目标群粘性账号策略
```

含义是：

- 同一个目标群尽量由稳定的一组账号发送。
- 同一种类型的内容尽量由同类角色账号发送。
- 账号不可用时再切换备用账号。

支持策略：

- 固定账号
- 轮询
- 随机
- 权重随机
- 目标群粘性
- 源群-目标群粘性
- 规则类型粘性

账号可以重复发送，但必须受限制：

- 单账号冷却
- 每日上限
- 每小时上限
- 同目标群连续发送限制
- 失败后自动冷却
- 账号健康分限制

### 14.10 源群监听聚合

转发监听不能每个任务开一个监听。

新版采用：

```text
一个源群只监听一次。
多个任务订阅这个源群事件。
```

例如：

```text
源群 A
  - 任务 1 订阅
  - 任务 2 订阅
  - 任务 3 订阅
```

源群出现新消息后：

```text
群监听中心收到消息
  ↓
生成源消息事件
  ↓
分发给订阅该源群的任务
  ↓
每个任务按自己的规则集处理
```

### 14.11 转发监听群不走人工审核

转发监听群消息不需要人工审核后发送。

流程是：

```text
源消息事件
  ↓
规则过滤
  ↓
内容转换
  ↓
路由目标群
  ↓
账号分配
  ↓
自动发送
```

不再有：

- 待审核
- 人工审核
- 审核通过
- 审核驳回
- 审核队列

规则处理结果只有：

- 通过
- 忽略
- 拦截
- 转换失败
- 发送失败
- 部分失败
- 已发送

### 14.12 转发监听任务详情页

详情页包含：

- 任务总览
- 源消息事件
- 规则处理结果
- 转发批次
- 目标群发送项
- 监听状态
- 账号分配
- 失败与重试
- 审计记录

运营人员要能看到：

- 源群来了哪些消息
- 哪些被过滤了
- 为什么被过滤
- 哪些被拦截了
- 为什么被拦截
- 哪些通过了
- 转换后是什么内容
- 发到了哪些目标群
- 由哪个账号发送
- 哪些失败了
- 失败原因是什么
- 是否重试过

## 15. 监听中心功能架构

监听中心是新版系统的关键基础设施。

它从任务里抽离出来，成为独立功能中心。

当前实现状态：

- 已有独立“监听中心”页面和汇总接口。
- 页面按频道 / 群聊聚合展示运行中或待运行任务的监听关系。
- 当前已经展示状态汇总、关联任务、订阅事件类型、待分发执行项、去重键统计、监听账号、备用账号、切换建议、最近事件、最后事件、错误和积压。
- 群监听和频道监听已经支持在页面上启用备用监听账号。群监听会校验备用账号在线且具备对应群发送/覆盖权限，并将原异常监听账号停用；频道监听会把相关任务里的离线账号替换为备用账号。
- 订阅事件类型、待分发统计和去重键统计已经落到监听中心页面；独立监听运行层已接入任务中心 drain，能按源群聚合 AI / 转发任务的监听需求、统一采集群上下文，在采集到新事件后唤醒订阅任务，并在没有健康监听账号但存在可用群账号时自动挂载备用监听账号。投递守护会为执行项写入执行租约，恢复租约过期、worker 心跳过期或长时间卡在 `executing` 的执行项，记录恢复阶段、恢复原因、恢复时间、原始结果快照、最近恢复执行项、恢复次数和上一个租约持有者，并在重试调度时保留上一轮失败摘要；任务中心 drain 会刷新 `worker_heartbeats`，运营数据页能展示 worker 心跳活跃/过期状态，避免中途崩溃后永久卡住或丢失异常现场。

分成两类：

- 频道监听中心
- 群监听中心

### 15.1 频道监听中心

服务于：

- 频道消息浏览
- 频道点赞
- 频道评论
- 频道回复
- 频道新消息任务

负责监听：

- 频道新消息
- Reaction 变化
- 频道讨论组评论
- 频道讨论组回复
- 消息状态变化

频道监听中心的原则：

```text
一个频道只监听一次。
多个频道任务共享监听事件。
任务通过订阅获得事件。
```

### 15.2 群监听中心

服务于：

- AI 活跃群
- 转发监听群
- 群归档
- 群上下文分析

负责监听：

- 源群新消息
- 群消息编辑
- 群消息删除
- 群上下文变化
- 群真实用户活跃度

群监听中心的原则：

```text
一个源群只监听一次。
多个转发监听任务共享源群事件。
AI 活跃群通过监听获取上下文变化。
```

### 15.3 监听中心统一流程

```text
目标产生事件
  ↓
监听中心接收
  ↓
事件标准化
  ↓
事件去重
  ↓
查找订阅任务
  ↓
分发给对应任务模块
  ↓
任务模块处理
  ↓
更新任务详情和执行记录
```

### 15.4 监听状态页

监听中心需要有独立状态页，展示：

- 监听对象
- 监听类型
- 监听账号
- 备用监听账号
- 最后事件时间
- 订阅任务数
- 事件积压
- 错误状态
- 最近错误原因
- 是否需要切换监听账号

## 16. 规则中心功能架构

规则中心主要服务转发监听，也服务 AI 和风控。

当前实现状态：

- 已有独立“规则中心”页面。
- 支持规则集、规则版本、新建版本、发布版本。
- 规则内容支持常用规则模板生成 JSON，也支持直接编辑 JSON，包含过滤、转换、路由、账号策略、限速和重试策略。
- 新建规则版本默认继承当前活动版本，避免每次从空白 JSON 开始重写。
- 规则测试器当前支持关键词规则命中、是否阻断、阻断原因，并可选择规则版本和源群 ID 预览过滤结果、转换后内容、目标路由、可用账号数量、账号策略和限流口径。
- 页面已展示规则冲突与缺口，可发现未发布规则集、转发任务绑定缺失或绑定版本不一致。
- 常用字段可视化编辑已覆盖白 / 黑名单、长度、媒体类型、屏蔽用户、仅文本/仅媒体、组合条件表达式、前后缀、提及/链接处理、默认运营目标、兼容目标群、源群映射、关键词路由、源群+关键词路由、账号策略、账号权重、目标限速、冷却和重试；复杂条件可用“全部满足 / 任一满足”模式配置，嵌套表达式可通过 JSON 递归表达并参与后端执行和规则测试。
- 规则命中统计已覆盖规则版本维度的任务数、执行项数、成功、失败、跳过、待执行和最近命中时间。
- 规则效果钻取已覆盖目标群、发送账号和关键词维度的执行项、成功、失败、跳过、待执行和最近命中时间。
- 规则长期趋势已覆盖最近 14 天执行项、成功、失败、跳过、待执行和成功率；规则转化对比已覆盖近 7 天与前 7 天执行量、成功量、成功率和成功率变化；规则跨维归因已覆盖规则版本、目标群和发送账号组合维度。转发执行项已记录规则命中摘要、源事件归因键和素材指纹，转发批次层已汇总源事件数、素材数和规则版本数，能追踪过滤、转换、路由、账号策略、源事件和素材归因；规则中心提供转发归因 CSV 导出，字段覆盖批次、源事件、素材指纹、规则版本、目标、账号、状态、重试和错误，并提供素材归因报表，按素材指纹聚合源事件、目标、账号、执行项、成功率、失败/跳过和重试。

规则中心包含：

- 转发规则集
- 过滤规则
- 转换规则
- 路由规则
- 关键词规则
- 敏感词规则
- 发送策略
- 限速策略
- 拦截策略
- 重试策略
- 规则测试器
- 规则版本管理

### 16.1 规则版本

每次修改规则都生成新版本。

任务绑定具体规则版本。

这样可以保证：

- 运行中的任务不会被突然影响。
- 规则变更可追踪。
- 规则出问题可以回滚。
- 不同任务可以使用不同版本。

### 16.2 规则测试器

规则测试器很重要。

管理员可以输入一条模拟源消息，系统展示：

- 是否通过过滤
- 命中了哪些规则
- 为什么被忽略
- 为什么被拦截
- 转换后内容是什么
- 会发到哪些目标群
- 每个目标群当前有多少可发送账号
- 预计使用哪种账号策略
- 是否触发限速

## 17. AI 内容中心功能架构

AI 内容中心不只是配置模型，还负责 AI 任务中的内容生产。

主要能力：

- 群上下文摘要
- AI 活跃群对话生成
- 频道评论生成
- 频道回复生成
- 转发内容改写
- 内容自动校验
- 账号角色一致性检查
- Token 和费用统计
- AI 生成记录

### 17.1 AI 生成原则

AI 输出应该结构化。

例如 AI 活跃群不是输出一段话，而是输出：

- 本轮目的
- 多条发言
- 每条发言的角色
- 每条发言的意图
- 每条发言的延迟
- 是否回复某条消息
- 风险级别

频道评论 / 回复也应输出：

- 评论内容
- 评论风格
- 关联消息
- 使用账号角色
- 发送建议时间
- 自动校验状态

### 17.2 AI 自动校验

AI 生成内容不走人工审核，但必须自动校验。

自动校验包括：

- 敏感词检查
- 重复内容检查
- 长度检查
- 外链检查
- @ 成员检查
- 账号角色一致性检查
- 目标群规则检查
- 上下文过期检查
- 格式检查

校验结果：

- 通过
- 跳过
- 拦截
- 重生成
- 延迟发送

## 18. 执行中心功能架构

执行中心负责把任务计划真正落地。

它是后台执行能力，不一定是单独页面，但需要在任务详情和执行记录里体现。

主要能力：

- 任务计划生成
- 账号分配
- 发送排队
- 冷却判断
- 失败重试
- 执行状态更新
- 上下文过期取消
- 任务进度汇总
- 异常恢复

### 18.1 执行前检查

每个执行项发送前都要检查：

- 任务是否仍在运行
- 当前时间是否允许执行
- 账号是否在线
- 账号是否可访问目标
- 账号是否可发言
- 账号是否冷却结束
- 是否超过上限
- 内容是否通过自动校验
- 目标是否仍可用
- 上下文是否过期
- 是否命中去重规则

### 18.2 失败处理

失败后需要明确：

- 失败原因
- 是否可重试
- 是否换账号重试
- 是否进入冷却
- 是否暂停目标
- 是否暂停任务
- 是否需要重新规划

失败原因要能被运营人员看懂，例如：

- 账号离线
- 账号无目标权限
- 目标群不可发言
- 账号冷却中
- 超过每日上限
- 内容被自动拦截
- 上下文过期
- 发送失败
- 媒体不可发送
- 规则转换失败

当前任务中心已按默认失败策略处理三类高频异常：

- 账号不可用 / 账号受限：可配置为跳过账号、暂停任务或停止任务，并记录最近一次策略命中；停止任务时会跳过同任务剩余待执行项。
- API 限流 / 慢速模式：可配置为等待重试、跳过或暂停；等待重试会从错误信息中解析等待秒数并延后执行项。
- 内容拦截：可配置为跳过消息、改写重试或暂停任务；改写重试会先对命中的关键词、群禁词和非白名单链接做确定性改写，改写后延后重试，无法产生可用改写时才标记为待改写。

## 25. 最终新版架构总结

新版 TG 运营管理平台应该重构成：

> 以账号中心、运营目标中心、任务中心、监听中心、规则中心、AI 内容中心、执行中心、数据中心和审计中心为核心的 Telegram 运营后台。

三类核心任务分别是：

```text
频道互动任务：
父任务 → 频道消息 → 动作子任务 → 账号执行项

AI 活跃群任务：
父任务 → 活跃循环 Cycle → 发言 Turn → 自动校验 → 账号发送

转发监听群任务：
父任务 → 源群事件 → 规则过滤 / 转换 / 路由 → 转发批次 → 目标群发送
```

关键变化是：

```text
旧模式：
任务直接操作账号和目标，执行记录混在一起。

新版：
账号中心提供资源。
目标中心确认范围。
规则中心定义规则。
监听中心捕获事件。
任务中心负责编排。
AI 中心生成内容。
执行中心负责发送。
数据与审计中心负责追踪。
```

最终运营链路变成：

```text
账号
  ↓
目标
  ↓
规则
  ↓
任务
  ↓
监听 / AI / 执行
  ↓
数据
  ↓
审计
```

最重要的新版原则是：

1. 频道任务按消息拆成子任务。
2. 频道和源群监听都要聚合，不能按任务重复监听。
3. AI 活跃群按 Cycle / Turn 循环执行。
4. 转发监听群按源事件和系统级规则自动执行。
5. AI 活跃群和转发监听群都不需要人工审核后发送。
6. 所有自动生成、过滤、转换、发送、跳过、失败、重试都必须可追踪、可解释、可审计。
