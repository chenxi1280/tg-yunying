## 1. 项目定位

TG 运营管理平台是一个面向 Telegram 运营团队的统一后台，用于集中管理：

- TG 开发者应用
- TG 账号与账号分组
- 单账号代理绑定
- 群、频道、联系人等 TG 资产
- 运营目标
- 消息发送
- 自动化任务
- 频道互动
- AI 群活跃
- 转发监听
- 监听事件
- 系统级规则
- 风控中心
- AI 内容生成
- 素材库
- 归档
- 运营数据
- 执行记录与审计

平台的核心目标是：

> 把 TG 账号、群频道目标、消息发送、频道互动、AI 群活跃、转发监听、风控处置、归档、数据统计和审计统一到一个可运营、可追踪、可控制的后台系统里。

新版系统不再以旧的多租户 SaaS、卡密、订阅套餐、旧 Campaign、AI 草稿审核流作为主线。

新版主线是：

```text
账号接入
  ↓
资产同步
  ↓
目标确认
  ↓
规则配置
  ↓
风控策略
  ↓
任务创建
  ↓
监听 / AI / Worker 执行
  ↓
任务详情追踪
  ↓
数据复盘
  ↓
审计留痕
```

## 0. 当前实现对齐结论（2026-05-13）

本次检查以当前代码为准，重点核对了前端主导航、页面视图、后端路由、任务中心模型、监听中心、规则中心和运营数据接口。结论是：新版主线已经基本落到当前页面与接口里，但仍有几处需要继续收敛，文档后续章节按这个口径理解。

- 任务中心当前主任务类型为 5 类：AI 活跃群、转发监听群、频道消息浏览、频道消息点赞、频道消息评论/回复。
- 任务中心已经支持创建、创建并启动、启动、暂停、继续、停止、重试、重置、编辑配置、删除、详情查看和执行记录追踪。

- 规则中心已经补充规则冲突与绑定缺口提示，可发现未发布规则集、转发任务绑定缺失或绑定版本不一致等问题。

### 0.2 仍需继续优化的部分

- 运营目标仍然处在 `groups` 和 `operation-targets` 双轨过渡期：频道任务、群类任务新建/编辑入口、群管理“用此群创建任务”、归档和消息发送里的群/频道目标已经优先使用运营目标，群类任务前端创建/编辑已经只提交运营目标 ID，任务执行动作也会保留运营目标 ID 追溯字段，消息发送的手动输入已收窄为个人目标；AI 活跃群、转发监听群和监听运行层已经支持运行时从运营目标 ID 直接解析授权群资产。执行层仍需要群资产承载 TG 授权、账号权限和上下文采集关系，后续主要是继续减少旧群 ID 兼容字段暴露。
- 监听中心当前已经有聚合状态、采集节流、订阅事件类型、待分发统计、去重键统计、备用账号建议、群监听切换和频道任务账号切换；任务中心 drain 已接入独立监听运行层，会按源群聚合订阅任务、统一采集上下文，并在采集到新事件后唤醒订阅任务，同时可在存在可用账号时自动恢复监听账号。投递守护已能给执行项写入 `lease_owner` / `lease_expires_at` 执行租约，把过期租约、持有者 worker 心跳过期或老式长时间 `executing` 执行项标记为超时失败，记录恢复阶段、恢复原因、恢复时间、原始结果快照、最近恢复执行项、恢复次数和上一个租约持有者，并交给既有重试策略；任务中心 drain 也会写入 `worker_heartbeats` 进程心跳，运营数据风控指标可看到 2 分钟内活跃和过期 worker。发送失败时也会按默认失败策略处理账号受限、API 限流和内容拦截。
- 规则中心当前已经有常用规则模板、多条件过滤、组合条件表达式、默认运营目标路由、源群映射、关键词路由、源群+关键词路由、账号权重、冲突提示、规则效果钻取、14 天趋势、近 7 天 / 前 7 天转化对比、规则版本 × 目标群 × 发送账号跨维归因，以及转发执行项里的过滤/转换/路由/账号策略命中摘要；转发批次层也会统计源事件数、素材指纹数和规则版本数，单条执行项保留源事件归因键与素材指纹，并可从规则跨维归因区域导出转发归因 CSV。规则中心页面已经补充素材归因报表，按素材指纹聚合源事件、目标、账号、执行项、成功率、失败/跳过和重试。后续体验重点是取消独立关键词库入口，将关键词、敏感词、白名单词、路由词统一收进规则集条件，并把输入过滤、输出校验和逐条转换结果做成可视化编辑和测试闭环。
- 频道评论和频道回复当前仍在一个 `channel_comment` 类型里承载，前端显示为“频道消息评论/回复”；任务配置已支持选择“评论频道消息 / 回复指定评论 / 评论+回复”。运营目标详情可从频道消息行同步讨论区评论树，系统会缓存一级评论和二级回复对象，任务配置可从已采集评论中可视化选择回复对象，也保留手动输入消息 ID 兼容；执行项 payload 会区分 `comment_mode`、`reply_to_message_id` 和 `reply_target_label`。独立频道回复任务类型仍可作为后续增强，但当前主流程已经能完成二级回复对象采集、选择和执行。
- 系统仍保留少量兼容字段和兼容模型，例如 tenant、ReviewQueue、旧 Campaign 服务、旧用量字段等。它们不应再进入新版主流程，但可以作为迁移兼容层逐步清理。
- 后台账号与权限目前只有字段和前端雏形，尚未形成完整闭环：需要恢复数据库后台账号登录、用户管理接口、菜单 / 按钮权限返回、前端路由和按钮控制、后端写接口权限校验，以及权限变更审计。
- 当前执行层已具备独立 worker、Redis 队列、Task / Action 模型、执行租约、worker heartbeat、监听运行层和账号容量检查，能够支撑 100 个左右账号的运营试运行；但面向 1000 个账号、20-30 个持续任务时，仍需先升级为 Planner / Dispatcher / Listener / Recovery / Metrics 分工、Action 原子领取、多 worker 安全并发、有界并发、账号 / 代理 / 目标多维限流和运行指标可观测。详细方案见 `docs/capacity-and-dispatch-upgrade-plan.md`。
- 系统设置只保留 TG 开发者应用、AI 供应商、AI 与素材基础配置、本地代理资源和系统运行配置；不再承载“发送节奏与风控”。全局发送节奏、全局静默时段、默认失败处理策略、账号评分、处置队列、命中记录和策略审计统一归入风控中心；运营数据只做统计复盘和跳转，不再提供策略编辑抽屉。关键词、敏感词和白名单词等内容条件统一回到规则中心维护。运营目标详情已经补充群目标级风控策略和目标账号覆盖编辑，可维护发送时间窗、群日上限、账号/群冷却、链接白名单、发送前校验，并可按目标切换账号发言/监听权限；素材仍在 AI 与规则基础配置中维护。后续素材库要从“URL/文本配置”增强为可实时上传、可缓存 Telegram 媒体引用、可被消息发送、转发监听和 AI 活跃群共用的基础能力。
- 单账号代理、代理健康检查、代理告警状态机和独立风控中心属于设计增强项；当前实现若只有账号状态、发送节奏和运营数据风险指标，应按“过渡入口”理解，不能误认为已经完成代理处置闭环。
- AI 活跃群已经支持按账号 ID 配置稳定角色，会把同一任务内和跨任务的账号最近成功发言作为下一轮账号记忆，并按账号聚合历史成功发言、常用角色、常见意图和近期表达形成长期画像；生成前会额外汇总主线方向、最近真人上下文和上一轮 AI 已说内容，形成“话题脉络”，并按本轮发言数生成“承接 / 补充 / 互动 / 收束”的多轮话题计划，写入提示词、执行项和任务详情。更强的长期主题编排仍可继续增强。

## 2. 产品边界

平台只围绕自有、授权、可运营的 TG 账号和目标设计。

系统需要明确：

- 只能使用已接入账号。
- 只能操作已确认的运营目标。
- 只能监听已授权源群、群聊、频道或讨论组。
- 任务必须有时间窗口、账号池、频控和执行记录。
- 所有发送、监听、过滤、转换、AI 生成、失败、重试都必须留痕。
- 后台支持多个内部账号共同使用，不再假设所有后台账号都是同级 admin。
- 权限只做到功能入口和操作按钮级别，不做数据权限、账号池隔离、目标隔离或任务归属隔离。
- 高风险动作仍依靠二次确认、原因填写和审计追踪控制；菜单和按钮权限只决定“能不能看到入口、能不能点击动作”。

系统不是简单的批量刷量工具，而是一个面向运营流程的后台。

新版功能设计的核心原则是：

```text
可配置
可自动执行
可追踪
可暂停
可重试
可解释
可风控
可审计
```

特别说明：

> AI 活跃群和转发监听群不做人工审核。  
> 系统通过自动校验、规则过滤、风控、限速、上下文检查、失败重试来保证任务可控。

## 3. 新版系统总架构

新版系统拆成 13 个功能中心。

1. 运营概览中心
2. 系统初始化中心
3. TG 账号中心
4. 运营目标中心
5. 消息发送中心
6. 任务中心
7. 监听中心
8. 规则中心
9. 风控中心
10. AI 内容中心
11. 素材库
12. 后台账号与权限中心
13. 数据、归档与审计中心

它们之间的关系是：

```text
系统初始化中心
  提供 TG 开发者应用、AI 供应商、全局规则、风控、素材等基础配置。

TG 账号中心
  提供可登录、可同步、可绑定本地代理、可发送、可监听的账号资源。

运营目标中心
  把账号同步到的群、频道、联系人整理成可运营目标。

规则中心
  提供过滤、转换、路由、限速、拦截、发送账号分配等系统级规则。

风控中心
  负责全局发送节奏、账号评分、代理健康、策略分层、账号处置队列、命中记录和策略审计。

AI 内容中心
  提供群上下文分析、批量对话生成、评论生成、回复生成、转发改写等能力。

素材库
  提供文本、图片、表情包、文件、链接和组合消息的上传、资产版本、TG 媒体缓存、标签、可用状态、使用归因和失败回退。

后台账号与权限中心
  负责后台登录账号、角色、菜单权限、按钮权限、启停、重置密码和权限变更审计；不负责数据隔离。

监听中心
  负责监听频道、讨论组、源群、群上下文、Reaction、评论和回复事件。

任务中心
  负责任务创建、配置、启动、暂停、重置、停止和详情追踪。

消息发送中心 / 执行中心
  负责把人工发送请求和任务生成的发送计划真正执行出去；发送前必须通过账号状态、代理健康、风控策略、目标权限和规则内容检查。

数据与审计中心
  负责统计、执行记录、失败追踪、运营复盘和敏感操作留痕。
```

### 3.1 1000 账号容量升级口径

当前系统不需要推翻现有 Task / Action / Account / Rule / RiskControl 模型。1000 账号能力应在现有模型上做执行层升级：

```text
Task
  ↓
Planner Worker 生成 Action
  ↓
Dispatcher Worker 原子领取 Action
  ↓
账号 / 代理 / 目标 / 规则 / 风控准入
  ↓
Telegram Gateway 执行
  ↓
结果回写、失败策略、风控事件、运行指标
```

面向 1000 个账号和 20-30 个持续任务，系统后台能力需要拆成：

- Planner Worker：只负责任务扫描和 Action 生成。
- Dispatcher Worker：只负责原子领取 pending Action、执行 Telegram API 和回写结果。
- Listener Worker：只负责群、频道、讨论区、评论和上下文采集。
- Recovery Worker：只负责超时、租约过期、worker 失联、失败重试和任务状态修复。
- Metrics Worker：只负责队列积压、任务延迟、账号容量、代理健康和运行快照。

关键原则：

- 先做 Action 原子领取，再增加 worker 数量。
- Planner 使用 `plan_batch_key = task_id + 计划时间戳` 表示规划批次，使用 `action_dedupe_key = plan_batch_key + 业务维度` 防止单条 action 重复。
- Planner 必须有全局和单任务 pending 积压保护，超过阈值时暂停规划或降低规划频率。
- Dispatcher claim 必须包含任务状态、任务优先级、每任务 claim 配额、任务类型权重、账号池公平性、防饥饿 aging 和 action 到期时间。
- Dispatcher claim 必须两段式短事务实现，数据库行锁事务只做预领取和状态标记，Redis token、账号 in-flight lock、代理和目标配额不能放在 DB 行锁事务里等待。
- 账号转派在 claim 阶段提前完成，且必须先按当前 worker 的账号分片筛选候选账号；本分片无可用账号时延后 action。
- 多 worker 限流使用 Redis token bucket，覆盖全局、任务、任务类型、账号、代理、目标和媒体发送维度；Redis token 建议使用带 TTL 的 reservation，Redis 不可用时默认 fail-closed。
- 同一账号默认只能被一个 worker 同时使用，账号 in-flight lock 与 token bucket 分开处理。
- 发送类 action 必须区分 `unknown_after_send` 和明确失败可重试状态；只有已写入 `gateway_call_started_at` 的 attempt 才能由 Recovery 推断为 `unknown_after_send`，避免 Telegram 已发送但本地回写失败后自动重复发送。
- 先做账号、代理、目标的多维限流，再提高并发。
- 监听采集和发送执行分开扩容。
- Listener 必须维护事件水位、source claim 和唯一事件键，处理回补、去重、bot 消息过滤、media group 聚合、编辑和删除事件口径。
- Telegram API 调用期间不持有数据库事务。
- 任务列表读运行快照，任务详情按需查明细。
- worker 并发必须受 PostgreSQL 连接池公式约束，不能只按 TG API 吞吐配置。
- action、listener event、execution attempt 和 daily stats 必须有数据库唯一约束支撑幂等。
- 运行明细数据默认保留最近 5 个自然日，第 6 天先把第 1 天数据汇总成总数，再滚动删除第 1 天全部运行明细；长期只保留任务、账号、目标、任务类型、日期和状态维度的汇总总数以及清理审计，不再保留窗口外单条异常明细。日汇总按业务自然日幂等 upsert，清理任务重跑不能重复加总。
- 当前项目尚未上线，本轮不设计线上迁移、灰度回滚和旧 worker 兼容切换方案。
- 容量结论必须通过压测固化，静态评估不能直接等同线上承载证明。

## 5. 后台账号与权限体系

后台账号与权限体系用于把后台分给更多运营人员使用。它解决的是“谁能进入哪个功能、能点击哪些动作”，不解决“谁能看哪一批 TG 账号 / 哪些运营目标 / 哪些任务数据”。

设计边界：

- 做功能权限：一级菜单、页面 Tab、关键按钮、危险动作。
- 不做数据权限：不按账号池、运营目标、任务、归档、规则版本或创建人做数据隔离。
- 不做多租户 SaaS 权限：当前仍按一个运营后台、一套全局业务数据理解。
- 不做复杂组织架构：暂不需要部门、岗位层级、审批流、权限继承树。
- 所有权限变更、账号启停、重置密码和高风险按钮点击必须写入审计。

### 5.1 角色口径

角色只作为权限模板和管理边界，不承载数据范围。

| 角色 | 定位 | 默认权限 |
| --- | --- | --- |
| 系统管理员 | 平台 owner，负责配置、账号分配和危险动作 | 全部菜单和全部按钮 |
| 运营管理员 | 日常运营负责人，可配置任务和处理异常 | 运营相关菜单、常用按钮，默认不含系统初始化和权限管理 |
| 账号添加专员 | 只负责把 TG 账号接入系统 | 运营概览只读、TG账号管理入口、账号新增 / 完成登录 / 同步必要按钮 |
| 只读观察员 | 查看运行状态和审计复盘 | 运营概览、运营数据、审计记录只读 |

角色不是硬编码业务能力。最终以账号上保存的权限点为准，角色只用于创建账号时快速带出默认勾选。

### 5.2 权限粒度

权限分两层：

```text
菜单权限
  决定左侧导航、页面入口、路由是否可进入。

按钮权限
  决定页面内关键动作是否可见或可点击。
```

按钮权限只覆盖会改变系统状态、触发外部动作或影响运行安全的操作。普通筛选、刷新、查看详情、复制 ID、展开折叠不需要单独权限。

推荐权限编码：

| 权限点 | 类型 | 说明 |
| --- | --- | --- |
| `overview.view` | 菜单 | 查看运营概览 |
| `accounts.view` | 菜单 | 进入 TG 账号管理 |
| `accounts.create` | 按钮 | 新增 TG 账号 |
| `accounts.login` | 按钮 | 完成验证码 / 二维码 / 2FA 登录 |
| `accounts.sync` | 按钮 | 同步账号资料、群、频道和联系人 |
| `accounts.update_profile` | 按钮 | 修改 TG 账号资料 |
| `accounts.delete` | 按钮 | 移除 TG 账号 |
| `accounts.pool_manage` | 按钮 | 新增账号分组、移动账号分组 |
| `developer_apps.manage` | 按钮 | 维护 TG 开发者应用 |
| `targets.view` | 菜单 | 查看运营目标 |
| `targets.manage` | 按钮 | 同步目标、编辑目标、维护目标账号覆盖 |
| `message_sending.view` | 菜单 | 进入消息发送 |
| `message_sending.create` | 按钮 | 新建直接发送 |
| `tasks.view` | 菜单 | 进入任务中心 |
| `tasks.manage` | 按钮 | 创建、编辑、启动、暂停、停止、重置任务 |
| `listeners.view` | 菜单 | 进入监听中心 |
| `listeners.manage` | 按钮 | 切换监听账号、恢复监听、处理积压 |
| `rules.view` | 菜单 | 进入规则中心 |
| `rules.publish` | 按钮 | 新建、编辑、发布规则版本 |
| `risk.view` | 菜单 | 进入风控中心 |
| `risk.manage` | 按钮 | 修改全局策略、处置风险、暂停账号、调整冷却 |
| `archives.view` | 菜单 | 进入归档中心 |
| `archives.manage` | 按钮 | 启动归档、重跑归档、导出归档 |
| `usage.view` | 菜单 | 进入运营数据 |
| `system.view` | 菜单 | 进入系统设置 |
| `system.manage` | 按钮 | 修改 AI、素材、本地代理和运行配置 |
| `permissions.view` | 菜单 | 查看后台账号与权限 |
| `permissions.manage` | 按钮 | 新增后台账号、编辑权限、停用账号、重置密码 |
| `audits.view` | 菜单 | 查看审计记录 |
| `audits.export` | 按钮 | 导出审计 CSV |

#### 5.2.1 与现有前端菜单 key 的兼容

当前前端已经有一套 view key，例如 `accounts`、`targetManagement`、`messageSending`、`taskManagement`、`usageReports`。新版权限点不应该直接复用旧 key，但必须提供兼容映射，避免历史账号升级后菜单全空或权限错位。

迁移映射建议：

| 旧 key | 新权限点 |
| --- | --- |
| `overview` | `overview.view` |
| `accounts` | `accounts.view` |
| `targetManagement` / `groups` / `groupManagement` / `operationTargets` | `targets.view` |
| `messageSending` | `message_sending.view` |
| `taskManagement` | `tasks.view` |
| `listenerCenter` | `listeners.view` |
| `ruleCenter` | `rules.view` |
| `riskControl` | `risk.view` |
| `archives` / `groupArchives` | `archives.view` |
| `usageReports` | `usage.view` |
| `systemConfig` / `developerApps` / `aiConfig` | `system.view` |
| `audits` | `audits.view` |

兼容规则：

- 后端读取旧 `menu_permissions` 时，如果发现旧 key，先转换为新版 `.view` 权限点。
- 前端只使用后端返回的新版 `permissions` 控制菜单和按钮，不再在业务组件里直接判断旧 key。
- 保存后台账号权限时，只保存新版权限点；旧 key 只作为一次性迁移输入。
- `*` 表示全部权限，只允许系统管理员或 bootstrap 超级管理员持有。

### 5.3 账号添加专员场景

账号添加是当前最明确的分工场景。目标是让专员只做 TG 账号接入，不碰任务、规则、风控、系统配置和数据导出。

建议给“账号添加专员”的权限：

- `overview.view`
- `accounts.view`
- `accounts.create`
- `accounts.login`
- `accounts.sync`

页面表现：

- 左侧只显示运营概览和 TG账号管理。
- TG账号管理里只显示新增账号、完成登录、同步账号资料和必要的账号详情查看。
- 删除账号、移动分组、维护开发者应用、修改资料、代理绑定、风控处置、任务创建等按钮隐藏或置灰。
- 如果新增账号前缺少开发者应用，页面只提示联系系统管理员配置，不给专员开放开发者应用维护入口。

后端表现：

- 任何创建账号、登录、同步等写操作都要校验对应按钮权限，不能只靠前端隐藏按钮。
- 无权限时统一返回 403，并写入可选的拒绝审计，方便排查误配。
- 审计里的 actor 使用后台账号 ID、名称和角色，不再只写“系统管理员”或“普通用户”。

账号添加专员默认不包含：

- `accounts.view_codes`：查看或提取 TG 官方验证码。
- `accounts.update_profile`：修改 TG 昵称、用户名、头像、简介等资料。
- `accounts.pool_manage`：移动账号分组或维护账号池。
- `accounts.proxy_bind`：绑定或切换代理。
- `accounts.clone`：克隆账号配置。
- `accounts.manual_send`：用账号直接私发或手动发送。
- `accounts.delete`：移除 TG 账号。

如果后续确实需要专员协助验证码提取，必须单独授予 `accounts.view_codes`，并在审计中记录查看人、查看账号、验证码来源和查看时间。

### 5.4 交互设计

后台账号与权限入口建议放在系统设置中，作为“账号与权限”Tab；如果后续后台账号数量增多，可以升为独立菜单。

账号列表展示：

- 姓名 / 登录名 / 邮箱 / 手机号。
- 角色模板。
- 启用状态。
- 最近登录时间。
- 已授权菜单数量和按钮数量。
- 最近权限更新时间和操作人。

账号编辑弹窗包含：

- 基础信息：名称、邮箱、手机号、初始密码或重置密码。
- 角色模板：系统管理员、运营管理员、账号添加专员、只读观察员。
- 菜单权限：按主导航勾选。
- 按钮权限：按菜单分组展示，只在已勾选菜单下显示。
- 启用状态：允许登录 / 停用。

保存规则：

- 系统管理员至少保留一个启用账号，避免把自己锁出后台。
- 不能删除最后一个系统管理员。
- 非系统管理员不能给别人授予 `permissions.manage`。
- 修改自己的权限时，如果会移除当前页面或权限管理入口，需要二次确认。
- 权限变更立即对新请求生效；已打开页面可以在下一次接口请求或刷新后体现。

### 5.5 数据与接口设计

已有基础可复用：

- `app_users` 可作为后台账号表继续使用。
- `role` 可保留为角色模板字段。
- `menu_permissions` 可升级为权限点集合；如果继续用文本字段，建议存储逗号分隔的权限编码；如果改为 JSONB，后续查询和分组展示更方便。
- 当前前端已有后台用户编辑弹窗和 `menu_permissions` 表单雏形，可以保留交互方向并补齐真实接口。

建议接口：

| 接口 | 说明 | 权限 |
| --- | --- | --- |
| `GET /api/admin/users` | 后台账号列表 | `permissions.view` |
| `POST /api/admin/users` | 新增后台账号 | `permissions.manage` |
| `PATCH /api/admin/users/{id}` | 修改账号资料、角色、权限、状态 | `permissions.manage` |
| `POST /api/admin/users/{id}/reset-password` | 重置密码 | `permissions.manage` |
| `GET /api/auth/me` | 当前账号信息和权限点 | 登录即可 |
| `GET /api/auth/permissions` | 当前系统支持的权限点清单 | `permissions.view` |

`/api/auth/me` 返回契约建议：

```json
{
  "id": 12,
  "name": "账号添加专员A",
  "role": "后台用户",
  "role_template": "账号添加专员",
  "is_super_admin": false,
  "permissions": ["overview.view", "accounts.view", "accounts.create", "accounts.login", "accounts.sync"],
  "permission_version": 3,
  "is_active": true
}
```

字段口径：

- `permissions` 是唯一前端权限来源，菜单和按钮都按它判断。
- `role_template` 只用于展示和快速套模板，不作为最终授权依据。
- `is_super_admin=true` 等价于 `permissions=["*"]`，只给 bootstrap 管理员或明确保留的系统管理员。
- `permission_version` 用于权限立即生效；账号权限、状态、角色模板变化后递增。

后端需要提供统一校验：

```text
require_permission("accounts.create")
require_permission("tasks.manage")
require_permission("permissions.manage")
```

路由层先校验登录，再校验权限。服务层仍要保持业务校验，例如账号状态、代理、开发者应用、风控和目标权限，不能把按钮权限当作业务合法性。

#### 5.5.1 登录与权限实时生效

后台账号登录不能只相信 token 里的旧角色。每次请求至少需要完成：

```text
解析 token
  ↓
按 sub 查询 app_users
  ↓
校验 is_active
  ↓
校验 permission_version 是否仍有效
  ↓
读取最新 permissions
  ↓
执行 require_permission(...)
```

如果账号被停用、密码被重置后要求重新登录、权限版本不一致或角色模板被降级，旧 token 应在下一次请求返回 401 或 403。这样才能保证“停用账号”和“收回权限”即时生效。

bootstrap 管理员口径：

- bootstrap 管理员只用于首次初始化、紧急恢复和创建第一个数据库系统管理员。
- 正常运行后应优先使用数据库后台账号登录。
- bootstrap 管理员的所有操作也必须写审计，actor 标记为 `bootstrap_admin`，避免审计里混同普通系统管理员。
- 如果配置允许禁用 bootstrap 登录，应在系统已有至少一个启用系统管理员后再关闭。

#### 5.5.2 角色字段落地

为了避免把旧 `普通用户`、`系统管理员` 与新角色模板混在一起，建议字段分层：

| 字段 | 说明 |
| --- | --- |
| `role` | 登录账号类型，例如 `系统管理员` / `后台用户`。用于区分超级管理边界。 |
| `role_template` | 权限模板，例如 `运营管理员` / `账号添加专员` / `只读观察员`。用于快速生成权限点。 |
| `permissions` | 最终权限点集合。系统实际校验只看这里。 |
| `permission_version` | 权限版本。权限或启停状态变化时递增。 |

如果短期不新增字段，也可以暂时继续使用 `role` 和 `menu_permissions`，但接口层必须对外统一输出 `role_template` 与 `permissions`，避免前端继续依赖旧枚举。

#### 5.5.3 敏感字段与危险动作

“不做数据权限”不等于所有字段都可见。以下能力应作为敏感字段或危险动作单独控制：

| 权限点 | 类型 | 说明 |
| --- | --- | --- |
| `accounts.view_phone` | 敏感字段 | 查看完整手机号；无权限时只显示脱敏号码。 |
| `accounts.view_codes` | 敏感字段 / 按钮 | 查看或提取 TG 官方验证码。 |
| `accounts.proxy_bind` | 按钮 | 绑定、切换或解绑账号代理。 |
| `accounts.clone` | 按钮 | 创建或执行账号克隆计划。 |
| `accounts.manual_send` | 按钮 | 从账号详情直接私发或手动发送。 |
| `system.secrets_manage` | 敏感配置 | 新增或修改 API Hash、AI Key、Telegram Bot Token 等密钥。 |
| `audits.view_sensitive` | 敏感字段 | 查看审计详情里的密钥摘要、错误堆栈、验证码事件详情。 |

默认规则：

- 只读观察员不看完整手机号、验证码、密钥、代理凭证和敏感审计详情。
- 账号添加专员不默认查看验证码；确有需要时单独授权并审计。
- 系统设置里的密钥输入和更新必须用 `system.secrets_manage`，不能只靠 `system.manage`。
- 导出类动作按危险动作处理，至少需要对应模块的 `*.export` 或 `audits.export`。

#### 5.5.4 路由权限矩阵

首期落地时至少覆盖所有写接口、导出接口和敏感读接口。建议矩阵：

| 路由范围 | 方法 / 动作 | 权限点 |
| --- | --- | --- |
| `/api/tg-accounts` | `POST` | `accounts.create` |
| `/api/tg-accounts/{id}/login/*` | 开始登录、验证码、扫码检查 | `accounts.login` |
| `/api/tg-accounts/{id}/sync-*` | 同步资料、群、目标、联系人 | `accounts.sync` |
| `/api/tg-accounts/{id}/verification-codes/*` | 提取验证码 | `accounts.view_codes` |

### 5.6 与现有实现的差距

当前实现已经出现了一部分权限雏形，但还没有形成完整后台账号权限体系：

- `AppUser` 模型里已有 `role`、`password_hash`、`is_active`、`menu_permissions`。
- 认证模块里已有 `parse_menu_permissions()` 和 `format_menu_permissions()`。
- 前端用户编辑弹窗已有角色、启停和菜单权限勾选字段。
- 当前运行链路仍以引导管理员为主，数据库用户 token 路径未作为正式后台登录路径使用。
- 前端初始化里后台用户列表仍为空数组，用户管理接口没有形成可用闭环。

因此实施顺序应是：

1. 恢复数据库后台账号登录和 `/auth/me` 权限返回。
2. 补齐后台账号管理接口和前端列表。
3. 用权限点驱动左侧导航和页面按钮。
4. 给关键写接口加后端权限校验。
5. 补旧菜单 key 到新权限点的迁移，并让前端统一使用 `/auth/me.permissions`。
6. 补敏感字段脱敏和危险动作权限，例如完整手机号、验证码、密钥、代理绑定、手动发送和导出。
7. 补审计：登录、登出、权限修改、密码重置、账号停用、无权限拒绝、敏感字段查看和危险动作。

### 5.7 验收标准

- 系统管理员可以新增后台账号，选择角色模板并微调菜单 / 按钮权限。
- 账号添加专员登录后只能看到运营概览和 TG账号管理，并且只能执行新增、登录、同步相关动作。
- 账号添加专员不能进入任务中心、规则中心、风控中心、系统设置、审计导出等页面。
- 直接调用无权限 API 返回 403，不会因为前端隐藏按钮而漏防。
- 最后一个系统管理员不能被停用、降权或删除。
- 权限变更、重置密码、停用账号、敏感字段查看和关键业务按钮点击都有审计记录。
- 权限变更后旧 token 不能继续使用旧权限；新请求应按最新账号状态和权限版本判断。
- 旧 `menu_permissions` key 能平滑迁移到新版 `permissions`，不会导致历史账号菜单丢失。

### 7.3 素材库与 TG 媒体缓存

素材库是消息发送、转发监听和 AI 活跃群的共同基础能力，不再只是提示词页面里的简单 URL 配置。

素材库支持：

- 文本素材。
- 图片素材。
- 表情包素材。
- 文件素材。
- 链接素材。
- 组合消息素材。
- 从历史源消息或归档消息收藏为素材。

新增图片、文件或表情包时，平台只做临时中转，不默认持久化图片原文件。文件上传后先写入最长 10 天 TTL 临时目录，完成安全校验后由指定素材缓存账号上传到 Telegram 素材缓存会话、私有频道或收藏夹；上传成功后删除本地临时文件，只保存 Telegram 缓存消息 ID、引用摘要、素材指纹和刷新状态。发送任务不是媒体引用转发，执行层只能消费已经缓存可用的素材：从 TG 缓存消息临时下载，再由发送账号重新上传到目标。缓存未就绪、缓存消息不可用或素材缓存账号处于 FloodWait 时，发送现场不等待、不上传、不阻塞批次，按规则降级为文本、换已就绪素材或跳过。素材库独立刷新队列可以后续自动处理，但不提供人工缓存或人工重新缓存入口；缓存失败就丢弃对应媒体缓存。

素材库必须记录：

- 素材标题、类型、标签、状态和默认 caption。
- 文件名、MIME、大小、宽高和预览状态。
- 素材指纹和来源：上传、URL、历史消息、归档消息。
- 外部可恢复 URL 或原始来源说明；不保存本地原图路径作为长期依赖。
- Telegram 缓存账号、缓存 peer、缓存消息 ID 和缓存状态。
- `delivery_mode=download_reupload`、表情包子类型、缓存就绪状态和缓存账号 FloodWait 截止时间。
- 使用次数、最近使用时间、最近失败原因。

素材资产必须按版本引用。替换文件、修改 URL、重建组合消息或重新转码时生成新的资产版本；已发布规则版本、运行中任务、执行项和发送记录要固化 `material_id + asset_version_id + asset_fingerprint`，不能因为后台编辑素材而静默改变历史执行口径。

已被规则版本、任务、发送记录或审计引用的素材不允许物理删除，只能停用或软删除。停用被运行中任务引用的素材时必须展示影响范围，后续执行项按规则换素材、降级文本或跳过。

服务器磁盘只允许保存上传、下载、预览、缓存刷新和重新发送过程中的临时文件，TTL 上限为 10 天。临时目录不进入备份，上传或发送成功后立即删除，后台启动时要清理过期临时文件。

Telegram 缓存会话是媒体素材的核心资产，但不是灾备系统。平台只维护当前可用缓存引用、素材内容版本 `asset_version_id` 与 Telegram 引用版本 `tg_ref_version_id`；数据库备份只能恢复素材元数据、指纹、规则关系和 TG 缓存位置，不能保证恢复已经被 Telegram 删除、账号失权或缓存频道不可访问的媒体本体。不需要设计备用缓存引用作为承诺能力。

素材库需要提供 TG 缓存健康入口，展示缓存账号异常、缓存频道不可访问、缓存消息疑似删除、待刷新队列、缓存账号 FloodWait、不可恢复素材、队列保护触发和缓存清理记录。等待队列爆满、不可恢复素材暴增或兜底扫描频繁唤醒时，系统直接放弃受影响素材缓存并清理等待项。素材预览应从 TG 缓存临时拉取或流式返回，不把预览图长期落盘。

素材库页面可以先作为系统设置或 AI 内容中心的独立 Tab。后续素材量增大后，再升为一级入口。

## 8. TG 账号中心

TG 账号中心是系统的账号资源管理台，负责把 TG 账号从“可登录”管理到“可被任务安全使用”。它不是风控中心的替代品，也不是任务中心的选号器；它提供账号的身份、登录、同步、代理、目标覆盖、状态证据和处置入口，最终能不能参与任务仍由风控中心、运营目标中心和规则中心共同判定。

账号中心的设计目标是：

- 让运营人员一眼知道账号现在能不能用、为什么不能用、下一步该做什么。
- 把登录、同步、验证码、代理绑定、目标权限和验证辅助集中在账号详情里处理。
- 把“账号级受限”和“目标级不可发言”分开，避免把群验证误判成账号封禁，也避免把账号受限误认为点确认就能恢复。
- 为任务中心、消息发送、监听中心和风控中心提供稳定的账号资源口径。

账号中心负责：

- 账号列表、账号分组和账号详情。
- 新增 TG 账号、验证码登录、二维码登录和 2FA 登录。
- Session 状态、重新登录、官方验证码提取和登录过程追踪。
- 账号资料、联系人、群、频道和目标覆盖同步。
- 单账号本地代理绑定和代理健康摘要展示。
- 账号在目标上的可访问、可发言、可监听、可互动能力展示。
- 账号执行记录、同步记录、验证辅助任务和最近失败原因展示。
- 账号角色、账号标签、任务用途和行为一致性基础配置。
- 风控摘要、处置建议和跳转到风控中心的入口。

账号中心不负责：

- 不直接编辑全局发送节奏、小时 / 日限额、默认失败策略和账号评分算法。
- 不绕过风控中心强行恢复受限账号。
- 不在任务执行时承诺“指定账号一定发送成功”。
- 不把代理节点、机场订阅或线路选择暴露成日常运营配置。

账号中心和风控中心的边界：

| 能力 | 账号中心 | 风控中心 |
| --- | --- | --- |
| 新增账号、登录、验证码、二维码、2FA | 负责完整流程 | 只提示阻塞原因和处置入口 |
| 资料、联系人、群频道同步 | 负责发起和展示结果 | 将同步异常纳入评分和处置 |
| 单账号 `proxy_id` 绑定 | 负责绑定和展示摘要 | 判断代理健康、复用风险和告警状态 |
| Session 失效和重新登录 | 负责重新登录 | 暂停任务参与并生成处置项 |
| 目标覆盖和账号能力 | 展示账号对目标的可访问、可发言、可监听能力 | 判断目标权限异常是否阻塞执行 |
| 账号受限、FloodWait、疑似封禁 | 展示原因、证据和处理入口 | 计算等级、冷却、暂停、转派和恢复条件 |
| 账号分组、标签、角色 | 负责维护基础属性 | 作为选号和策略解释的输入 |
| 限额、冷却、转派和恢复 | 展示摘要 | 负责策略和执行判定 |

### 8.1 页面结构

TG 账号中心建议拆成三个层级：

```text
账号总览
  ├─ 账号列表
  ├─ 账号分组
  ├─ 待处理账号
  └─ 新增账号

账号详情
  ├─ 概览
  ├─ 登录与 Session
  ├─ 资料与同步
  ├─ 目标覆盖
  ├─ 验证辅助
  ├─ 代理绑定
  ├─ 执行记录
  ├─ 角色与标签
  └─ 审计记录

账号操作
  ├─ 完成登录
  ├─ 重新登录
  ├─ 同步资料
  ├─ 同步联系人
  ├─ 同步群频道
  ├─ 提取官方验证码
  ├─ 绑定代理
  ├─ 暂停任务
  ├─ 设置冷却
  ├─ 健康检查
  └─ 跳转风控处置
```

账号列表不是单纯表格，应该是运行状态工作台。默认展示：

- 账号名称、手机号、用户名和分组。
- 登录状态、Session 状态和最后登录时间。
- 当前运行状态：在线、冷却、受限、需重新登录、禁用等。
- 健康分和风控等级摘要。
- 绑定代理和代理健康摘要。
- 目标覆盖数量：可发言群、可监听群、可互动频道、联系人。
- 最近失败原因，不展示泛泛的“高风险”，要展示具体原因。
- 待处理事项：验证码、重新登录、验证辅助、代理异常、FloodWait、账号受限。
- 主操作按钮：未登录账号只展示“完成登录”；正常账号展示“详情 / 同步 / 健康检查”；受限账号展示“查看原因 / 去处置”。

### 8.2 账号状态模型

账号状态分成三层，页面必须同时表达，不能只用一个大状态覆盖全部问题。

第一层是登录状态：

- 未登录
- 等待验证码
- 等待扫码
- 等待 2FA
- 在线
- 离线
- 需重新登录
- Session 失效

第二层是运行状态：

- 可用
- 观察
- 保守
- 冷却中
- 受限
- 疑似封禁
- 已封禁
- 禁用

第三层是目标能力：

- 可访问目标
- 可发言
- 可监听
- 可浏览频道
- 可点赞
- 可评论
- 可回复
- 目标权限异常
- 群验证待处理
- 慢速模式

任务选号和发送准入不能只看“在线”。账号必须同时满足：

```text
登录状态可用
  ↓
Session 可用
  ↓
代理健康或策略允许
  ↓
风控等级允许参与
  ↓
账号未处于冷却或暂停
  ↓
对目标具备对应能力
  ↓
规则和内容检查通过
```

未登录、等待验证码、等待扫码、等待 2FA、需重新登录、Session 失效、受限、疑似封禁、已封禁、禁用账号不能参与新执行项。冷却中账号不参与即时执行，但可以在冷却结束后重新进入候选池。

### 8.3 受限账号与群验证分流设计

账号中心必须把“账号本身受限”和“账号在某个群里被管理机器挡住无法发言”拆开处理。两者的影响范围、处理动作、恢复证据都不同，不能共用一个“确认处理”按钮。

核心原则：

- 账号级受限是账号运行风险，影响该账号参与所有或大部分任务。
- 群管理机器拦截是账号和目标之间的权限 / 验证问题，只影响某个群或某类目标。
- 内容规则拦截是文案或规则问题，不能把账号标成受限。
- 代理异常是运行环境问题，不能直接归因成账号封禁。

两类最关键的问题要分开建模：

| 问题 | 范围 | 常见原因 | 状态应该挂在哪里 | 主处理动作 |
| --- | --- | --- | --- | --- |
| 账号级受限 | 账号全局 | TG 官方限制、疑似封禁、频繁操作、账号异常、Session 失效、FloodWait 过多 | 账号运行状态、风控处置队列 | 暂停账号、系统定期探测、设置冷却、健康检查、必要时重新登录、去风控中心处置 |
| 群管理机器挡住发言 | 单个账号 + 单个目标 | 入群验证、按钮验证、关注频道、禁言、管理员审核、慢速模式、群权限不足 | 目标覆盖能力、验证辅助任务 | 管理员在账号详情里进入“解除群限制”流程，系统辅助处理并重新检查目标能力 |

完整分型如下：

| 类型 | 典型来源 | 账号中心展示 | 推荐动作 | 恢复证据 |
| --- | --- | --- | --- | --- |
| 登录不可用 | Session 失效、开发者应用凭据变更、未完成 2FA | 登录状态异常、最近登录失败原因 | 完成登录、重新登录、同步资料 | 登录成功、Session 可用、资料同步成功 |
| 账号级受限 | 账号受限、疑似封禁、官方限制、频繁操作 | 账号受限原因、最近 TG 返回、影响任务范围、下一次自动探测时间 | 暂停账号、设置冷却、系统自动探测恢复、风控中心处置；仅在 Session 失效或登录态异常时重新登录 | 系统探测到限制解除、健康检查通过、后续安全动作成功 |
| FloodWait | TG API 限流、短时间动作过密 | 等待时间、触发动作、触发目标 | 自动冷却、等待后恢复、降频 | 冷却时间结束、后续 API 探测通过 |
| 群管理机器拦截 | 群验证、按钮验证、关注频道、管理员审核 | 具体群、验证类型、被拦截原因、解除群限制入口 | 管理员在账号详情里处理：系统可自动关注 / 点击按钮 / 发送验证回复；无法自动处理时展示人工步骤并记录处理结果 | 目标能力重查通过，或该账号在该群安全测试发言成功 |
| 目标权限不足 | 被禁言、群无权限、不在群内、慢速模式 | 具体目标、权限标签、可发言状态 | 重新入群、联系管理员、等待慢速模式、切换账号 | 目标权限同步或发送前预检查通过 |
| 代理异常 | 本地代理不可达、认证失败、同代理异常聚集 | `proxy_id`、最近健康错误、影响账号数 | 检查代理、切换 `proxy_id`、等待风控中心解除阻塞 | 代理健康检查通过 |
| 内容或规则拦截 | 文案违规、链接不在白名单、重复内容 | 最近被拦截内容和规则版本 | 调整内容或规则 | 规则测试或发送前内容检查通过 |

账号级受限的主按钮不应该是“标记已处理”，也不应该默认要求用户重新登录。推荐按钮是：

- 暂停账号
- 设置冷却
- 立即探测
- 健康检查
- 查看最近失败
- 去风控中心处置
- 重新登录（仅登录态异常时展示）
- 禁用账号

账号级受限的页面文案应该强调：

> 该账号存在全局运行风险，已暂停参与新任务。系统会按冷却策略定期探测账号限制是否解除；只有检测到 Session 失效、2FA 未完成或登录态异常时，才需要重新登录。

系统自动探测需要覆盖：

- Session 是否仍然有效。
- 是否仍返回账号受限、疑似封禁、FloodWait 或同类 TG 错误。
- 是否可以完成轻量安全动作，例如获取自身资料、同步少量目标、读取目标能力。
- 冷却时间是否结束，是否满足风控中心恢复条件。
- 如果探测失败，记录失败原因、下一次探测时间和是否需要人工介入。

账号级限制解除后，系统可以自动把账号从“阻塞”降为“观察”或“保守”，但恢复到标准强度必须满足风控中心策略。重新登录只是恢复路径之一，不是所有受限账号的默认动作。

群管理机器拦截的主入口应该是账号详情里的“解除群限制”，不是让运营人员离开系统自行处理完再回来简单勾选。这个入口属于目标能力修复流程，绑定 `account_id + group_id / operation_target_id`，只处理该账号在该目标上的发言权限。

解除群限制流程建议包括：

- 展示被拦截的群、账号、触发任务、失败时间和原始错误。
- 判断拦截类型：按钮验证、关注频道、发送验证回复、管理员审核、被禁言、慢速模式、目标权限不足。
- 对可控动作由系统通过该账号执行，例如关注频道、点击验证按钮、发送指定验证回复或 `/start`。
- 对不可控动作展示管理员处理步骤，例如联系群管理员、等待管理员审核、等待慢速模式、确认该账号没有被群内禁言。
- 处理后立即执行目标能力重查，必要时发送安全测试消息。
- 只有目标能力重查通过，才恢复该账号在该群的可发言候选资格。

如果某些动作必须人工在 Telegram 客户端完成，账号详情里的按钮也不应叫“确认处理”，而应是“已完成群内解除，重新检查”。按钮文案必须明确：

> 管理员确认已在该群完成解除限制后，系统会重新检查该账号在当前目标上的发言能力；检查通过才恢复该目标候选资格。

群管理机器拦截处理后，系统只应恢复这个账号对这个目标的候选资格，不能把账号全局风控等级直接上调，也不能清空其他目标上的异常。

如果确认后没有成功登录、同步、目标权限检查或发送证据，账号健康分不能直接上升，只能关闭当前验证事项。恢复任务参与仍要经过发送前预检查。

### 8.4 账号详情设计

账号详情是账号中心的核心，不应该只是一组基础字段。建议按 Tab 或分区展示。

概览：

- 账号基本信息：昵称、用户名、手机号、账号 ID、分组、标签、角色。
- 当前登录状态、运行状态、风控等级、健康分。
- 当前是否可参与任务，以及不可参与的首要原因。
- 绑定代理、代理健康和最近检查时间。
- 最近一次成功动作、最近一次失败动作、最近一次同步。
- 待处理事项列表。

登录与 Session：

- 当前登录方式：验证码、二维码、2FA。
- 登录流程状态和最近错误。
- Session 是否可解密、是否可用、是否需要重新登录。
- 官方验证码提取列表。
- 完成登录、重新登录、提取验证码、刷新登录状态。

资料与同步：

- TG 昵称、用户名、头像、简介等资料。
- 最近资料同步记录。
- 联系人同步记录。
- 群 / 频道同步记录。
- 手动同步资料、同步联系人、同步群频道。

目标覆盖：

- 该账号可见的联系人、群、频道和运营目标。
- 每个目标的能力：可访问、可发言、可监听、可浏览、可点赞、可评论、可回复。
- 每个目标的最近失败、慢速模式、验证任务和权限标签。
- 支持从目标行跳转到运营目标详情或创建任务。

验证辅助：

- 待处理验证任务。
- 已处理和已忽略验证任务。
- 验证类型：群发言不可用、需验证或关注、慢速模式、频道加入、按钮验证、发送验证回复。
- 建议动作：人工处理、关注频道、点击按钮、发送验证回复。
- 处理结果：已处理、需人工处理、失败、已忽略。
- 处理后要刷新账号详情和目标能力。

代理绑定：

- 当前 `proxy_id`、本地代理地址、协议和最近健康结果。
- 代理是否处于告警、是否阻塞发送。
- 绑定变更记录。
- 切换代理时必须记录原因，并触发健康检查。

执行记录：

- 消息发送、频道浏览、点赞、评论、回复、AI 活跃、转发监听等执行项。
- 状态、目标、任务、规则版本、失败类型和失败详情。
- 支持按任务、目标、失败类型筛选。
- 可以跳转任务详情、目标详情、规则命中和风控命中。

角色与标签：

- 账号用途：AI 活跃、转发发送、频道互动、监听、备用。
- 账号角色：引导型、解答型、补充型、新人型、总结型、公告同步型、活动同步型。
- 账号画像摘要：常见语气、近期发言、常用目标、成功率。
- 这些配置用于任务选号和 AI 账号一致性，不替代风控评分。

### 8.5 账号分组与账号池

账号分组用于运营管理，账号池用于任务选择。两者可以共用底层数据，但页面语义要清楚。

建议默认分组：

- AI 活跃账号组
- 转发发送账号组
- 频道互动账号组
- 监听账号组
- 备用账号组
- 新号观察组
- 冷却观察组
- 高健康账号组

分组页需要展示：

- 分组账号数。
- 在线账号数。
- 可参与任务账号数。
- 受限 / 冷却 / 需重新登录账号数。
- 代理覆盖情况。
- 最近 24 小时成功、失败、FloodWait 和账号受限次数。

任务中心选择账号范围时可以选择：

- 全部账号
- 指定账号分组
- 手动选择账号

后端最终仍要在执行前根据账号状态、代理健康、风控等级、目标权限和规则策略做二次决策。

### 8.6 单账号代理绑定

账号建议绑定固定本地代理出口。服务器上可以运行 Clash、sing-box、v2ray、xray 或其他代理软件；机场订阅、节点配置、节点选择和固定路由都在代理软件里维护。

平台不直接理解机场订阅和机场节点。

平台只认识 `socks5/http` 本地代理地址：

```text
socks5://127.0.0.1:11001
http://127.0.0.1:12001
```

账号通过 `proxy_id` 绑定到某个本地代理端口：

```text
TG账号 -> proxy_id -> 本地代理端口 -> 代理软件固定路由 -> 机场节点
```

平台只维护：

- `proxy_id`
- 代理名称
- 协议：socks5 / http
- host
- port
- 本地代理认证信息
- 健康状态
- 绑定账号数
- 最近检查错误

平台不保存机场订阅链接，不展示机场节点名称，不在业务页面直接切换机场节点。节点调整由具备服务器代理维护权限的管理员在服务器代理软件中完成；平台只负责账号和本地代理地址的绑定、健康检测、发送准入和风险提示。

代理不是日常任务选择的主维度。没有代理告警时，运营人员仍然按账号分组 / 账号池来创建任务和选择账号；只有当本地代理不可达、超时、认证失败、同代理账号异常聚集或同出口 FloodWait 异常增加时，风控中心才发出代理告警并提醒处理。

### 8.7 账号健康与风控摘要

账号健康用于描述账号基础可用性，账号评分用于决定账号是否可以参与任务。账号中心展示健康证据和风控摘要，风控中心计算可解释评分和策略等级。

账号中心展示的健康维度包括：

- 是否在线。
- Session 是否有效。
- 是否最近登录成功。
- 开发者应用凭据是否仍可用。
- 是否绑定代理。
- 绑定代理是否可达、是否超时、是否认证失败。
- 是否能访问目标群。
- 是否能在目标群发言。
- 是否能访问频道。
- 是否能监听源群。
- 是否近期频繁失败。
- 是否超过发送上限。
- 是否处于冷却中。
- 是否命中 FloodWait 或账号受限。
- 是否存在待处理验证任务。

风控摘要在账号列表和账号详情中展示：

- 当前风控等级：A 健康、B 观察、C 保守、D 冷却、E 阻塞。
- 当前策略：正常、降频、保守、冷却、阻塞。
- 最近风险：Session 失效、FloodWait、目标权限异常、内容拦截、账号受限等。
- 是否可参与任务。
- 去风控中心查看详情或处置。

### 8.8 账号接入与恢复闭环

新增账号闭环：

```text
新增账号
  ↓
选择开发者应用
  ↓
绑定或确认 proxy_id
  ↓
验证码 / 二维码 / 2FA 登录
  ↓
账号在线
  ↓
同步资料、联系人、群、频道
  ↓
生成目标覆盖和账号能力
  ↓
风控中心生成初始评分
  ↓
进入可用资源池或待处理队列
```

账号级受限恢复闭环：

```text
发现账号级受限
  ↓
账号中心展示 TG 返回、失败记录和影响范围
  ↓
风控中心暂停该账号参与新执行项
  ↓
系统设置冷却和下一次自动探测时间
  ↓
后台探测 Session、TG 限制状态、轻量同步和安全动作
  ↓
如果是登录态异常，再提示重新登录
  ↓
如果是代理异常，进入代理处置
  ↓
如果限制仍存在，继续冷却并记录下一次探测
  ↓
探测通过后进入观察 / 保守状态
  ↓
风控中心解除阻塞或降低限制
  ↓
账号重新进入候选池
```

群管理机器拦截恢复闭环：

```text
发现某账号在某群无法发言
  ↓
账号中心目标覆盖中标记该目标能力异常
  ↓
生成验证辅助任务，绑定 account_id + group_id / operation_target_id
  ↓
管理员在账号详情打开“解除群限制”
  ↓
系统识别验证类型并执行可控动作
  ↓
无法自动处理时，账号详情展示人工处理步骤
  ↓
管理员点击“已完成群内解除，重新检查”
  ↓
重新同步目标权限或执行安全测试发言
  ↓
仅恢复该账号在该目标上的可发言候选资格
```

账号中心的恢复动作必须有证据来源：成功登录、成功同步、目标能力检查通过、代理健康恢复、冷却结束或一次安全测试动作成功。仅人工填写“已处理”不能直接证明账号全局恢复；群验证事项的“已处理”也只能证明该事项关闭，不能证明账号级风险解除。

### 8.9 当前实现优化优先级

P0：

- 账号列表和详情展示最近受限原因，避免只显示泛化高风险状态。
- 验证辅助把“账号级受限”和“群管理机器拦截”分开，群发言不可用类任务绑定具体目标，并在账号详情提供“解除群限制”流程。
- 账号级受限默认停止参与新执行项，提供自动探测、立即探测、健康检查、设置冷却、去风控中心处置入口；只有登录态异常时才提供重新登录。
- 群管理机器拦截只阻塞该账号在该目标上的发言候选，不应全局禁用账号。
- 账号详情中把登录状态、运行状态、目标能力分开展示。

P1：

- 账号详情补齐目标覆盖 Tab，展示每个目标的可访问、可发言、可监听和最近失败。
- 账号分组页展示可用账号数、受限账号数、代理覆盖和最近失败。
- 账号执行记录支持按任务、目标、失败类型筛选，并能跳转任务 / 目标 / 风控命中。
- 代理绑定变更触发健康检查并记录审计。

P2：

- 账号角色和账号画像与 AI 活跃群、转发监听选号联动。
- 账号恢复需要可配置的证据门槛，例如系统自动探测通过、重新同步成功、目标能力检查通过或安全测试动作成功。
- 账号中心增加批量操作：批量同步、批量健康检查、批量设置分组、批量暂停和批量冷却。

### 8.10 遗漏场景检查清单

账号中心后续实现时，还需要补齐以下容易遗漏的边界。

状态归属不能混：

| 问题 | 应归属 | 不能误判为 |
| --- | --- | --- |
| Session 失效、2FA 未完成 | 登录状态 | 群验证、目标权限 |
| TG 官方账号受限、疑似封禁 | 账号运行状态 + 风控处置 | 普通验证任务 |
| 群管理机器拦截 | 账号 + 目标能力 | 账号全局受限 |
| 代理不可达、认证失败 | 代理资源 + 风控告警 | 账号封禁 |
| 内容违规、链接拦截、重复内容 | 规则中心 + 内容风险 | 账号受限 |
| 慢速模式 | 目标能力 + 调度冷却 | 账号封禁 |

验证辅助任务必须带足上下文：

- `account_id`
- `group_id` 或 `operation_target_id`
- 触发任务、执行项和消息 ID
- 原始错误、归因类型和建议动作
- 是否可自动处理
- 自动处理尝试次数和结果
- 人工处理说明
- 目标能力重查结果
- 恢复证据和审计 ID

解除群限制要注意：

- 如果群需要管理员审核，系统不能假装能自动解除，只能展示步骤、记录处理结果并重查目标能力。
- 如果需要发送验证回复，必须控制内容和频率，避免验证动作本身触发新的风控。
- 如果需要安全测试发言，要有可配置的测试内容、目标限制和审计记录；不能在真实群里随意发送无意义测试消息。
- 同一个群、同一个账号的验证任务要去重，避免重复点击按钮、重复发送验证回复。
- 多个账号被同一个群管理机器拦截时，可以批量展示，但恢复仍然按 `account_id + target_id` 单独记录。

任务联动要注意：

- 某账号在某群被挡住时，任务中心优先在同任务账号池内转派，不应把整个任务立即判死。
- 如果目标只有这一个可用账号，则任务进入“容量不足 / 目标能力异常”，而不是账号全局异常。
- 账号级受限恢复后，正在运行任务也要重新做风控预检查，不能直接继续旧执行项。
- 群限制解除后，只允许该账号重新进入该目标候选池；其他目标上的异常不受影响。

自动探测要注意：

- 探测频率必须受冷却和退避控制，不能因为自动恢复机制反复触发 TG 限流。
- 探测失败要记录下一次探测时间和失败原因。
- 自动恢复只能从阻塞降到观察 / 保守，是否恢复标准强度由风控中心策略决定。
- 如果探测发现登录态异常，才进入重新登录流程。

前端交互要注意：

- 未完成登录账号只显示“完成登录”，不展示验证码、同步、验证辅助、健康检查等完整操作面。
- 账号级受限展示“自动探测 / 立即探测 / 设置冷却 / 去风控处置”，不展示“标记已处理”。
- 群管理机器拦截展示“解除群限制 / 重新检查目标能力 / 切换目标账号”。
- 每个限制都要展示具体原因，不展示空泛的“高风险”。

审计和数据保留要注意：

- 自动探测、解除群限制、点击按钮、关注频道、发送验证回复、人工确认、目标能力重查都要写审计。
- 处置记录需要保留处理前状态、处理动作、处理人 / 系统任务、处理后检查结果。
- 已恢复记录不能马上清理，至少要能回溯最近一次限制原因和恢复证据。

## 9. 运营目标中心

运营目标中心是新版必须统一的重点。

当前系统里存在 groups 和 operation-targets 两套对象，容易造成运营人员和开发人员混淆。

新版功能上统一为：

```text
账号同步出来的是 TG 资产。
运营人员确认后才是运营目标。
后续消息发送、任务中心、归档都优先选择运营目标；执行层通过运营目标解析到已授权 TG 资产。
```

也就是：

```text
TG 账号同步群 / 频道
  ↓
进入资产列表
  ↓
运营人员确认是否纳入运营
  ↓
形成运营目标
  ↓
任务、发送、归档都围绕运营目标执行
```

### 9.1 运营目标类型

支持：

- 群聊目标
- 频道目标

频道讨论组当前按群聊目标或频道关联资产处理；联系人目标、手动目标不属于当前主线功能，避免和新版“群 / 频道运营”边界混淆。

### 9.2 目标能力

每个目标都要展示它能做什么：

- 是否可发送（已落地）
- 是否可监听（已落地）
- 是否可归档（已落地）
- 是否可创建 AI 活跃群任务（已落地）
- 是否可创建转发监听任务（已落地）
- 是否可创建频道互动任务（已落地）
- 是否有账号覆盖（详情页已落地）
- 哪些账号可访问（详情页已落地）
- 哪些账号可发言（详情页已落地）
- 哪些账号可监听（详情页已落地）

### 9.3 目标详情页

目标详情页展示：

- 目标基础信息
- 账号覆盖情况
- 可发言账号
- 可监听账号
- 最近消息
- 最近频道消息
- 历史任务
- 发送记录
- 归档记录
- 风险状态
- 快捷创建任务入口

群聊目标支持快捷创建：

- AI 活跃群任务
- 转发监听群任务
- 归档任务（目标有关联群资产时可直接创建）
- 直接发送消息

频道目标支持快捷创建：

- 频道浏览任务
- 频道点赞任务
- 频道评论 / 回复任务（当前由同一个任务类型承载）
- 频道消息发送

## 10. 消息发送中心

消息发送中心负责人工直接发送，不负责复杂自动化。它不能绕过风控中心：前端只提交业务发送请求，真正发送前必须经过账号中心状态、代理健康、风控中心策略、运营目标权限和规则中心内容检查。

支持：

- 私聊发送
- 群聊发送
- 频道发送
- 批量发送
- 定时发送
- 图片素材发送
- 表情包发送
- 文件发送
- 组合消息发送
- 发送记录
- 重试
- 取消
- 失败详情

### 10.1 发送目标来源

新建发送时必须先选择发送账号，再按账号过滤可发送目标。目标来源要明确标识：

- 联系人
- 账号可见群
- 已授权运营目标
- 频道目标
- 手动输入目标

手动输入目标要提示风险：

> 该目标未在系统资产中验证，可能因权限不足、目标不存在、账号限制等原因发送失败。

发送前流程：

```text
选择发送类型
  ↓
选择发送账号
  ↓
按账号过滤联系人、群、频道和运营目标
  ↓
输入内容和素材
  ↓
发送前预检查
  ↓
通过：创建发送任务或进入队列
不通过：展示阻塞原因和处理入口
```

发送前预检查需要覆盖：

- 账号是否完成登录、Session 是否可用。
- 账号是否绑定 `proxy_id`，本地代理地址是否可达。
- 账号风控等级、小时 / 日限额、全局冷却和当前策略。
- 账号是否能向目标发送、评论、浏览或点赞。
- 目标是否授权、是否可发送、是否冷却或达到上限。
- 内容是否命中规则中心的输入过滤、输出校验、链接白名单或敏感词规则。
- 素材是否可用、是否停用、是否已经有可用 TG 缓存消息。
- 图片、表情包或文件是否可从 Telegram 缓存临时下载并重新上传；缓存未就绪或缓存账号 FloodWait 时不在发送现场刷新。
- 表情包子类型是否被目标、账号、网关和规则允许。
- 目标是否允许发送媒体，是否存在禁图、慢速或媒体频控限制。

前端交互原则：

- A 账号只能看到 A 账号可发送的目标，避免先选到不可达目标再报错。
- 多账号批量发送时，目标列表需要展示每个账号是否可达。
- 检查不通过时不允许真实发送，只展示更换账号、处理验证、重新登录、改为定时发送或修改内容等入口。

发送调用链：

```text
前端提交发送请求
  ↓
风险预检查
  ↓
创建 message-send-task / batch
  ↓
Worker 或派发器取待发送任务
  ↓
执行前再次校验账号、代理、风控、目标和规则
  ↓
读取账号 session 和开发者应用凭据
  ↓
读取账号 proxy_id 和本地代理地址
  ↓
通过账号绑定代理调用 Telegram 网关发送
  ↓
写入发送结果、风控命中、账号状态和审计记录
```

消息发送的素材处理链路：

```text
选择文本 / 图片 / 表情包 / 文件 / 链接 / 组合消息
  ↓
校验素材状态和类型
  ↓
检查 caption、链接、标签和目标媒体权限
  ↓
检查 Telegram 缓存消息是否就绪
  ↓
临时下载缓存文件并由发送账号重新上传
  ↓
发送成功：记录图片 / 表情包 / 文件是否成功发送和远端消息 ID
  ↓
发送失败：记录素材失败、账号失败、目标失败、规则失败或缓存未就绪
```

### 10.2 发送记录

发送记录展示给运营侧的字段保持简单：

- 发送账号
- 发送目标
- 发送时间
- 状态
- 图片 / 表情包 / 文件是否成功发送
- 失败原因
- 关联审计

目标来源、消息内容、素材 ID、素材指纹、重试次数、缓存版本、下载重发细节、等待队列、唤醒来源和临时文件清理只进入审计或诊断视图，不作为发送记录主展示字段。

## 11. 任务中心总架构

任务中心是新版最核心模块。

新版任务中心分成三条核心任务主线：

1. 频道互动任务
2. AI 活跃群任务
3. 转发监听群任务

三个任务类型的执行模型不同。

```text
频道互动任务：
父任务 → 目标频道消息 → 动作子任务 → 账号执行项

AI 活跃群任务：
父任务 → 活跃循环 Cycle → 发言 Turn → 账号发送项

转发监听群任务：
父任务 → 源群事件 → 规则处理 → 转发批次 → 目标群发送项
```

---

## 12. 任务中心通用能力

所有任务都要支持：

- 创建
- 保存草稿
- 启动
- 暂停
- 继续
- 停止
- 重试
- 重置
- 编辑
- 删除
- 查看详情
- 查看执行记录
- 查看失败原因
- 查看审计记录

### 12.1 任务统一状态

建议统一任务状态：

- 草稿
- 已排期
- 运行中
- 已暂停
- 已达目标但未结束
- 收尾中
- 已完成
- 人工停止
- 异常失败
- 已删除

核心规则：

```text
任务不会因为提前达标就自然结束。
任务自然结束只由 end_at 控制。
管理员可以人工停止。
任务失败后可以重试或重置。
```

即：

```text
达到目标数量 ≠ 任务完成
到达结束时间 = 任务自然完成
```

### 12.2 重试、重置、暂停、停止的区别

前端必须解释清楚：

```text
暂停：
暂时停止继续执行，可恢复。

重试：
只重试失败项，不影响已成功项。

重置：
重新规划未完成部分，但不会重复执行已成功动作。

停止：
任务彻底停止，不再自动恢复。

删除：
隐藏或删除任务记录，正式任务建议保留审计。
```

## 13. 频道互动任务功能架构

频道互动任务包括：

- 频道消息浏览
- 频道消息点赞
- 频道消息评论/回复

当前实现说明：

- 当前后端任务类型是 `channel_view`、`channel_like`、`channel_comment`。
- `channel_comment` 同时承载评论和回复口径，前端展示为“频道消息评论/回复”。
- 当前可在任务配置里选择评论、回复或混合模式；回复模式要求填写回复目标消息 ID，执行项会保留 `comment_mode`、`reply_to_message_id` 和 `reply_target_label`，网关会把回复目标传给频道评论发送层。
- 频道消息详情支持同步讨论区评论树，后端用 `channel_message_comments` 缓存一级评论、二级回复、作者、父级回复消息 ID、内容预览、回复数和发布时间。
- 任务中心回复对象字段支持从已采集评论中选择，也支持继续手动输入 Telegram 消息 ID；这样在不新增单独 `channel_reply` API 类型的情况下，已经能完成“采集评论对象 -> 点选回复对象 -> 生成回复执行项 -> 网关按 reply_to 发送”的闭环。
- 独立的“频道回复”任务类型、完整评论树可视化折叠视图可以作为后续体验增强，但不是当前主流程阻塞项。

新版重点是：

> 一个频道任务下面，每条频道消息的点赞、评论、回复都应该是独立子任务。

### 13.1 频道任务结构

频道任务结构：

```text
父任务
  ↓
目标频道消息
  ↓
动作子任务
  ↓
账号执行项
```

示例：

```text
父任务：频道最新 5 条消息点赞

消息 101 点赞子任务
  - 账号 A 执行
  - 账号 B 执行
  - 账号 C 执行

消息 102 点赞子任务
  - 账号 A 执行
  - 账号 D 执行
  - 账号 E 执行
```

如果是评论任务：

```text
父任务：频道最新 5 条消息评论

消息 101 评论子任务
消息 102 评论子任务
消息 103 评论子任务
```

如果是回复任务：

```text
父任务：频道消息回复任务

消息 101 / 评论 A 回复子任务
消息 101 / 评论 B 回复子任务
消息 102 / 评论 C 回复子任务
```

### 13.2 创建频道任务流程

```text
选择任务类型
  ↓
选择频道目标
  ↓
选择消息范围
  ↓
选择动作类型
  ↓
配置每条消息目标数
  ↓
选择账号池
  ↓
配置节奏和风控
  ↓
容量检查
  ↓
确认创建
```

消息范围支持：

- 指定消息
- 最新 N 条
- 日期范围
- 动态新消息

当前默认值：

- 频道浏览、频道点赞、频道评论/回复默认使用动态新消息。
- 动态新消息任务不会因为某一轮达到数量就自然结束，而是按监听间隔继续采集；只有到达任务结束时间或人工停止才结束。

动作类型支持：

- 浏览
- 点赞 / Reaction
- 评论/回复

### 13.3 子任务生成逻辑

例如创建：

```text
频道最新 10 条消息
每条消息点赞 50 个
```

系统生成：

```text
10 个点赞子任务
每个子任务目标 50 个账号执行
```

例如创建：

```text
频道最新 10 条消息
每条消息评论 10 条
```

系统生成：

```text
10 个评论子任务
每个子任务目标 10 条评论
```

### 13.4 去重规则

频道任务必须强去重。

规则：

```text
同一个账号
同一个频道
同一条消息
同一种动作
只能有效执行一次
```

即使任务被重置，也不能让同一个账号重复对同一条消息执行同一动作。

这能防止：

- Worker 重复消费
- 任务重置导致重复执行
- 跨任务重复动作
- 账号重复点赞同一消息
- 账号重复评论同一消息

### 13.5 频道监听优化

频道任务不能每个任务单独监听频道。

新版采用：

```text
一个频道只启动一个监听实例。
多个任务订阅这个频道事件。
```

例如同一个频道同时存在：

- 点赞任务
- 评论任务
- 回复任务
- 动态新消息任务

系统只监听一次频道和讨论组。

监听到事件后：

```text
频道监听中心接收事件
  ↓
标准化事件
  ↓
事件去重
  ↓
分发给订阅该消息 / 动作的子任务
  ↓
更新子任务进度
  ↓
更新父任务总进度
```

### 13.6 频道任务详情页

频道任务详情页要围绕“消息子任务”展示。

包含：

- 任务总览
- 消息子任务
- 账号执行项
- 监听状态
- 容量分析
- 失败与重试
- 审计记录

消息子任务里要看到：

- 每条消息目标多少
- 已完成多少
- 失败多少
- 重复多少
- 哪些账号已执行
- 哪些账号失败
- 为什么失败
- 是否已达标
- 是否容量不足

示例展示：

```text
消息 101：点赞目标 50，已完成 38，失败 2，重复 1，状态运行中
消息 102：点赞目标 50，已完成 50，失败 0，重复 0，状态已达标
消息 103：点赞目标 50，已完成 12，失败 5，重复 2，状态容量不足
```

## 14. AI 活跃群任务功能架构

AI 活跃群是循环型任务。

它不是一次性群发，也不是简单监听转发，而是：

```text
根据群上下文
周期性生成自然对话
分配给不同账号
按节奏发送
一轮完成后再进入下一轮
```

### 14.1 AI 活跃群任务结构

```text
父任务
  ↓
活跃循环 Cycle
  ↓
本轮发言计划 Turn
  ↓
账号发送
```

核心规则：

```text
每一轮先拉取群上下文。
一次性生成多个账号的对话内容。
分配给不同账号按节奏发送。
本轮全部发送完成、取消或过期后，才进入下一轮。
```

这样可以：

- 节约 Token
- 保证对话连贯
- 避免多个账号单独生成导致重复
- 保证账号逻辑一致
- 控制发言节奏

### 14.2 创建 AI 活跃群任务流程

```text
选择 AI 活跃群任务
  ↓
选择授权群聊
  ↓
选择账号池
  ↓
选择监听账号
  ↓
配置活跃账号比例
  ↓
配置每轮参与账号数
  ↓
配置每轮发言数
  ↓
配置活跃时间段
  ↓
配置静默时间段
  ↓
配置每日爬坡策略
  ↓
配置话题方向和语气
  ↓
配置上下文范围
  ↓
配置自动校验和风控
  ↓
确认创建
```

### 14.3 活跃账号比例

支持配置：

- 活跃账号比例
- 每轮最少参与账号数
- 每轮最多参与账号数
- 每轮最少发言数
- 每轮最大发言数
- 单账号每轮最多发言数
- 单账号冷却时间

示例：

```text
账号池 40 个
活跃比例 30%
每轮最多 10 个账号
```

系统本轮选择：

```text
10 个账号参与
```

如果账号池较小，则按最小参与数兜底；如果账号池较大，则按最大参与数限制。

### 14.4 一次生成多账号对话

AI 不应该每个账号单独生成一次。

正确方式是：

```text
拉取一次群上下文
  ↓
AI 一次生成一轮对话计划
  ↓
计划里包含多条发言
  ↓
每条发言带角色、意图、延迟、回复对象
  ↓
系统再分配给具体账号
```

本轮对话计划可以包含：

- 第 1 条：引导型账号，回应用户问题
- 第 2 条：补充型账号，补充说明
- 第 3 条：提问型账号，引出讨论
- 第 4 条：总结型账号，做轻量总结

AI 输出的不是一段自由文本，而是本轮对话计划。

### 14.5 账号一致性

账号可以重复发送，但要保证行为一致。

每个账号需要有稳定角色：

- 引导型
- 解答型
- 补充型
- 新人型
- 总结型
- 轻松闲聊型

同一个账号要保持：

- 语气一致
- 角色一致
- 话题记忆一致
- 不要前后观点冲突
- 不要重复自己刚说过的话
- 不要突然从新人变成专家
- 不要连续刷屏

系统需要记录：

- 账号最近说过什么
- 今天说了多少
- 最近参与哪些话题
- 上次发言时间
- 当前冷却状态
- 近期失败情况

### 14.6 静默时间段

静默期不是完全停止，而是进入低频模式。

静默期规则：

- 只允许少量账号参与
- 通常 4～5 个账号以内
- 每轮只发少量消息
- 间隔时间拉长
- 不主动制造复杂话题
- 不密集互相回复
- 连续无人响应后逐步停止
- 最后进入休眠，只监听不发言

静默期适合：

- 轻量回应
- 收尾
- 简单提醒
- 低打扰话题
- 承接白天话题

静默期不适合：

- 大量提问
- 密集讨论
- 长篇解释
- 强推广
- 频繁互相回复

### 14.7 每日爬坡

每天活跃时间开始后，不应该马上全量启动。

应该分阶段：

```text
启动期：少量账号，少量消息
预热期：逐步增加账号和消息
正常期：按配置比例执行
收尾期：逐步降低
静默期：低频或休眠
```

示例：

```text
09:00 - 09:30 启动期
09:30 - 10:30 预热期
10:30 - 22:30 正常期
22:30 - 23:00 收尾期
23:00 后 静默期
```

每日重新启动时，系统应先观察群内真实活跃度，再逐步提高 AI 发言量。

### 14.8 上下文快照

每轮开始时，系统先拉取群上下文。

上下文包括：

- 最近群消息
- 上一轮 AI 发言
- 真实用户问题
- 管理员消息
- 群当前话题
- 群活跃度
- 账号角色记忆
- 风险话题
- 今日已发送情况

AI 生成要基于这份上下文快照。

这样后续可以追踪：

- AI 为什么生成这句话
- 基于哪些群消息生成
- 是否和当时话题一致

### 14.9 上下文过期处理

AI 生成内容后，不是无脑全部发完。

每条发送前需要检查：

- 群里是否出现大量新消息
- 话题是否明显变化
- 管理员是否发了新消息
- 是否出现敏感内容
- 是否有人明确要求停止

如果上下文过期：

```text
取消当前 Cycle 剩余未发送内容
标记为上下文过期
重新拉取上下文
重新生成下一轮
```

### 14.10 AI 活跃群不走人工审核

AI 活跃群消息不需要人工审核后发送。

流程是：

```text
AI 生成
  ↓
系统自动校验
  ↓
通过后自动发送
```

不再有：

- 待审核
- 人工审核
- 审核通过
- 审核驳回
- 审核队列

保留的是自动校验。

自动校验包括：

- 空内容检查
- 禁止关键词检查
- 重复内容检查
- 长度检查
- 外链检查
- @ 成员检查
- 账号冷却检查
- 账号发送上限检查
- 目标群频控检查
- 上下文是否过期检查
- AI 输出格式检查

自动校验结果：

- 通过：进入发送队列
- 跳过：不发送，记录原因
- 拦截：不发送，记录原因
- 重生成：整轮内容异常时重新生成
- 延迟：账号冷却或频控导致延后发送

### 14.11 AI 活跃群图片与表情包策略

AI 活跃群允许发送图片和表情包，但必须低频、可解释、可追踪，不能让模型自由选择任意图片文件。

推荐流程：

```text
AI 生成文本、角色、意图、情绪和素材意图
  ↓
规则引擎根据素材意图、目标群、账号角色、素材标签、表情包子类型和频控选择素材
  ↓
输出校验检查文本、caption、素材标签和目标媒体限制
  ↓
执行层从已缓存素材临时下载，并按节奏发送文本、表情包或图片
  ↓
记录素材 ID、素材指纹、规则版本、发送结果和失败回退
```

AI 输出只表达意图：

```json
{
  "content": "这个点有点意思，先蹲一下后续",
  "role": "补充型",
  "intent": "附和",
  "mood": "轻松",
  "material_intent": "表情包:围观",
  "allow_material": true
}
```

规则引擎再把 `material_intent` 映射到具体素材。默认限制建议：

- 每轮最多 1 条图片或表情包。
- 同一目标群 20 到 30 分钟内最多 1 张图或表情包。
- 同一账号不连续发送媒体。
- 支持图片伪表情包、静态 sticker、animated sticker、video sticker 和 custom emoji；custom emoji 必须通过账号、目标和网关能力校验。
- 素材未缓存或素材缓存账号 FloodWait 时不现场上传，按规则换已就绪素材、降级文本或跳过。
- 素材失败默认跳过当前 Turn 或回退为文本，不让整轮 Cycle 失败。
- 文件、视频和大图默认不进入 AI 活跃群，除非规则集显式允许。

### 14.12 AI 活跃群详情页

详情页包含：

- 任务总览
- 当前模式
- 当前轮次
- 今日发言数
- AI 消息占比
- 当前参与账号
- 下一轮时间
- 活跃循环列表
- 本轮发言计划
- 上下文快照
- 账号角色与记忆
- 自动校验结果
- 发送记录
- 失败与跳过原因
- AI 生成记录（按 Cycle 展示生成条数、Token、上下文消息数和账号记忆数，并展示话题脉络、话题计划、账号长期画像摘要）
- 素材使用记录（素材 ID、素材标签、素材意图、规则版本、缓存就绪、下载重发、失败或回退原因）
- 审计记录

运营人员要能看到：

- AI 是根据什么上下文生成的
- 本轮计划发哪些话
- 哪些已经发了
- 哪些还没发
- 哪些被自动跳过
- 为什么取消
- 为什么降频
- 为什么选择或跳过某个表情包 / 图片
- 哪个账号说过什么

## 15. 转发监听群任务功能架构

转发监听群是事件驱动任务。

它的目标是：

> 监听一个或多个源群，源群有新消息后，系统根据系统级规则判断是否转发、如何转换、发到哪些目标群、由哪些账号发送。

### 15.1 转发监听群任务结构

```text
父任务
  ↓
源群监听
  ↓
源消息事件
  ↓
输入过滤规则
  ↓
内容转换规则
  ↓
媒体与素材处理
  ↓
路由分发规则
  ↓
目标群发送
```

完整流程：

```text
源群出现新消息
  ↓
监听中心捕获消息
  ↓
生成源消息事件
  ↓
匹配任务订阅
  ↓
执行输入过滤规则
  ↓
通过后执行内容转换规则
  ↓
按规则保留源媒体 / 替换素材 / 追加素材 / 降级文本
  ↓
执行路由分发规则
  ↓
分配发送账号
  ↓
自动发送文本、图片、表情包或组合消息到目标群
  ↓
记录结果
```

### 15.2 创建转发监听群任务流程

```text
选择转发监听群任务
  ↓
选择一个或多个源群
  ↓
选择一个或多个目标群
  ↓
选择系统级规则集
  ↓
选择规则集版本
  ↓
选择发送账号池
  ↓
选择监听账号
  ↓
配置发送策略
  ↓
配置突发消息处理
  ↓
配置失败重试策略
  ↓
确认创建
```

### 15.3 源群选择

支持：

- 单个源群
- 多个源群
- 源群分组

源群必须满足：

- 已授权监听
- 系统账号可访问
- 存在可用监听账号

### 15.4 目标群选择

支持：

- 单个目标群
- 多个目标群
- 目标群分组
- 按规则路由到不同目标群

目标群必须满足：

- 已授权运营
- 有账号可发言
- 目标未处于风控冻结

### 15.5 系统级规则集

转发监听的输入过滤、内容转换、路由、账号、限速和重试必须作为系统级规则集配置。关键词、敏感词、白名单词和路由词不再作为独立“关键词库”选择，而是规则集里的条件。

任务创建时不直接临时写规则，而是选择：

- 规则集
- 规则集版本

规则中心维护：

- 输入过滤规则
- 输出校验规则
- 内容转换规则
- 路由分发规则
- 发送账号策略
- 限速策略
- 重试策略
- 规则测试器
- 命中记录
- 版本发布和回滚

这样好处是：

- 规则统一管理
- 多个任务复用规则
- 规则可测试
- 规则可版本化
- 规则变更可审计
- 规则出问题可以回滚

监听转发执行时必须先过滤，再转换，再路由：

```text
源消息事件
  ↓
输入过滤
  ↓
不通过：跳过 / 拦截，记录原因
  ↓
通过：内容转换
  ↓
路由分发
  ↓
账号分配
  ↓
发送
```

### 15.6 输入过滤规则

输入过滤规则决定源消息要不要进入后续转换和发送流程。监听转发必须在入口直接过滤，风险消息不应进入路由和发送链路。

支持：

- 黑名单关键词
- 白名单关键词
- 正则匹配
- 消息类型过滤
- 发送人过滤
- 管理员消息过滤
- 链接过滤
- 长度过滤
- 重复内容过滤
- 敏感词过滤
- 时间段过滤

过滤结果：

- 通过：继续转换和发送
- 跳过：不发送，只记录原因
- 拦截：不发送，只记录原因
- 标记：继续处理，但记录风险标签
- 规则异常：不发送，记录失败

没有人工审核结果。

### 15.7 内容转换规则

内容转换规则决定源消息如何变成目标群要发送的内容。只有输入过滤通过后才执行转换。

支持：

- 原样复制
- 去掉链接
- 替换链接
- 关键词替换
- 添加前缀
- 添加尾注
- 删除 @
- 删除来源署名
- 格式整理
- 图片保留或丢弃
- 长文本拆分
- 短消息合并

后续可以支持 AI 改写，但 AI 改写也必须回到输出校验链路：

```text
AI 改写
  ↓
输出校验
  ↓
不通过：转换 / 丢弃
  ↓
通过后发送
```

#### 15.7.1 媒体与素材处理

转发监听不能把带图片或表情包的源消息简单降级为纯文本。源消息事件需要同时记录文本、caption、媒体类型、源媒体指纹、源消息 ID、源群、发送人和 TG 源媒体缓存状态；相册 / media group 需要额外记录 `source_media_group_id`、组内顺序、总数和每段 caption。

如果规则可能保留源媒体或基于媒体路由，监听中心应在捕获源消息后短时间内把源媒体上传到 TG 源媒体缓存，并保存 `source_media_asset_id`、媒体指纹、缓存状态和失败原因。平台不长期保存源图原件；最长 10 天 TTL 临时下载文件在上传 TG 缓存后删除。不能等到转发执行时再临时回源群取图；源消息删除、采集账号掉线或缓存不可用时，只能按规则降级文本或跳过。

源媒体缓存是转发监听的临时执行资产，不等同于平台素材库长期资产。任务结束、授权取消、TTL 到期、等待队列保护或不可恢复状态触发后，可以直接清理源媒体缓存，只保留元数据、指纹、失败原因和发送结果。

源媒体缓存遇到 FloodWait 时，必须先创建同一个 `source_media_asset_id` 的待缓存记录，并用源群、源消息 ID、相册 ID、组内序号和可得媒体摘要做去重。后续重试只更新这条记录的状态、重试次数和下一次重试时间，不能把同一张源图反复当成新图片入库。只有源消息不同、相册分段不同或指纹不同，才创建新的源媒体资产。

如果规则选择等待源媒体缓存完成，转发执行项进入 `waiting_material_cache`，由 PhotoWorker / 源媒体缓存 Worker 在缓存完成后发布 `source_media_cached` 事件唤醒同一 `source_media_asset_id` 关联的等待执行项。PhotoWorker 只负责缓存和事件通知，不直接发送消息；执行中心被唤醒后必须重新检查账号、目标、规则、风控、相册顺序和幂等键，通过后才由发送账号下载缓存文件并重新上传发送。

等待缓存不能无限期挂起。默认等待到任务本轮超时，建议上限不超过 30 分钟；超时后按规则丢弃媒体并降级文本、换已就绪素材或跳过，并记录 `material_cache_wait_timeout`。如果执行项已经因超时完成降级、换素材或跳过，后续缓存才完成也不能再补发，只记录迟到事件。`source_media_cached` 事件必须带 `source_media_asset_id`、`cache_version` / `tg_ref_version_id`、缓存 peer、缓存消息 ID 和完成时间；旧事件迟到不能唤醒已经换版本或已结束的执行项。`source_media_cached` 事件丢失时，由兜底扫描定期查找 `waiting_material_cache` 且源媒体已 ready 的执行项重新唤醒；兜底扫描频繁唤醒代表队列不健康，应触发保护并放弃受影响素材缓存。任务暂停时不发送，恢复后再检查；调度优先级是运行中任务优先于已暂停任务，高优先级任务优先于低优先级任务。任务停止、任务结束、源群取消授权、目标不可用、任务时间窗关闭、目标静默期生效、群上下文过期或规则版本失效时，唤醒后只能跳过并记录原因。同一源媒体可以唤醒多个目标执行项，但每个目标使用自己的执行项幂等键。相册按分段状态处理：ready 分段按原顺序发送，失败、超时或不可恢复分段可以剔除，允许去掉失败图继续发送剩余图片但不能乱序。等待队列需要按任务、源群和全局设置容量上限；超过上限时直接放弃受影响素材缓存并清理等待项。任务详情不提供人工缓存、人工重新缓存或人工刷新入口；失败就丢弃对应媒体缓存。源消息在成功下载前被删除且没有可用 TG media 摘要或临时文件时，源媒体应标记为 `unrecoverable`；缓存连续失败达到阈值时标记为 `cache_failed` 后放弃本轮缓存。

规则中心可以配置媒体处理策略：

| 策略 | 说明 |
| --- | --- |
| 保留源媒体 | 源消息带图、表情包或文件时，从 TG 源媒体缓存下载后重新发送到目标群。 |
| 保持相册顺序 | 源消息是相册 / media group 时，按原组内顺序逐段发送。 |
| 相册失败图剔除 | 某段缓存或发送失败时，去掉失败图并继续按原顺序发送剩余图片。 |
| 只保留文字 | 丢弃源媒体，只发送转换后的文本。 |
| 替换为素材库图片 | 根据源群、关键词、消息类型或标签选择平台素材。 |
| 追加素材 | 在转换后的文本后追加图片或表情包。 |
| 媒体失败跳过 | 媒体无法获取、类型不允许或缓存失败时跳过。 |
| 媒体失败降级文本 | 图片发送失败时发送文本并记录降级原因。 |

推荐流程：

```text
采集源消息文本、caption、媒体类型、媒体指纹和相册顺序
  ↓
输入过滤：文本条件 + 媒体类型条件
  ↓
内容转换：文字清洗 + 媒体策略
  ↓
素材选择：源媒体 / 平台素材 / 无媒体
  ↓
输出校验：caption、链接、素材标签和媒体类型
  ↓
路由和账号策略
  ↓
发送文本、图片、表情包或组合消息
  ↓
记录源事件、素材指纹、素材 ID、规则版本和发送结果
```

转发任务详情和规则归因必须能看到：源消息是否带媒体、媒体是否保留、相册顺序是否保持、失败图是否被剔除、是否替换为平台素材、命中的素材规则、最终素材 ID、Telegram 缓存是否就绪、图片 / 表情包 / 文件是否成功发送以及失败原因。

### 15.8 路由规则

路由规则决定消息发到哪些目标群。

支持：

- 全部目标群分发
- 源群到目标群映射
- 关键词路由
- 消息类型路由
- 目标标签路由
- 规则命中路由

示例：

```text
包含“公告” → 发到公告群
包含“活动” → 发到活动群
源群 A 的消息 → 发到目标群 1、2
源群 B 的消息 → 发到目标群 3
```

### 15.9 发送账号分配

转发监听群输出通过 TG 账号发送到目标群，可以由不同账号发送，也允许账号重复发送，但要保证账号发送逻辑的一贯性。

建议默认采用：

```text
目标群粘性账号策略
```

含义是：

- 同一个目标群尽量由稳定的一组账号发送。
- 同一种类型的内容尽量由同类角色账号发送。
- 账号不可用时再切换备用账号。

支持策略：

- 固定账号
- 轮询
- 随机
- 权重随机
- 目标群粘性
- 源群-目标群粘性
- 规则类型粘性

账号可以重复发送，但必须受限制：

- 单账号冷却
- 每日上限
- 每小时上限
- 同目标群连续发送限制
- 失败后自动冷却
- 账号健康分限制

### 15.10 源群监听聚合

转发监听不能每个任务开一个监听。

新版采用：

```text
一个源群只监听一次。
多个任务订阅这个源群事件。
```

例如：

```text
源群 A
  - 任务 1 订阅
  - 任务 2 订阅
  - 任务 3 订阅
```

源群出现新消息后：

```text
群监听中心收到消息
  ↓
生成源消息事件
  ↓
分发给订阅该源群的任务
  ↓
每个任务按自己的规则集处理
```

### 15.11 转发监听群不走人工审核

转发监听群消息不需要人工审核后发送，但必须在入口直接执行输入过滤。过滤不通过的源消息直接跳过或拦截，只记录原因，不进入转换、路由和发送流程。

流程是：

```text
源消息事件
  ↓
输入过滤
  ↓
不通过：跳过 / 拦截
  ↓
通过：内容转换
  ↓
路由分发
  ↓
账号分配
  ↓
自动发送
```

不再有：

- 待审核
- 人工审核
- 审核通过
- 审核驳回
- 审核队列

规则处理结果只有：

- 通过
- 跳过
- 拦截
- 转换失败
- 发送失败
- 部分失败
- 已发送

### 15.12 转发监听任务详情页

详情页包含：

- 任务总览
- 源消息事件
- 规则处理结果
- 转发批次
- 目标群发送项
- 监听状态
- 账号分配
- 失败与重试
- 审计记录

运营人员要能看到：

- 源群来了哪些消息
- 哪些被过滤了
- 为什么被过滤
- 哪些被拦截了
- 为什么被拦截
- 哪些通过了
- 转换后是什么内容
- 发到了哪些目标群
- 由哪个账号发送
- 哪些失败了
- 失败原因是什么
- 是否重试过

### 17.1 任务绑定规则

所有自动化任务都不再单独选择关键词库，而是选择规则集和规则版本。

```text
创建任务
  ↓
选择任务类型
  ↓
选择规则集
  ↓
选择规则版本，或使用当前发布版本
  ↓
任务启动
  ↓
执行时按规则版本处理
```

任务类型使用规则：

- 监听转发：必须使用输入过滤、内容转换、路由、账号、限速和重试。
- AI 回复：必须使用输入过滤、AI 输出校验、内容转换、账号和限速。
- AI 评论发送：必须使用输入过滤、AI 输出校验、内容转换、账号、限速和重试。
- 普通消息发送：建议使用输出校验、账号和限速。
- 风控类任务：必须使用输入过滤、输出校验和拦截。

### 17.2 监听转发规则链路

监听转发必须在入口直接过滤。

```text
源群新消息
  ↓
输入过滤规则
  ↓
不通过：跳过 / 拦截，记录原因
  ↓
通过：内容转换
  ↓
路由分发
  ↓
账号选择
  ↓
限速检查
  ↓
发送
  ↓
失败重试 / 记录结果
```

### 17.3 AI 回复和 AI 评论规则链路

AI 回复和 AI 评论都是“AI 生成内容后再发送”，因此共用输入过滤、输出校验和逐条处理逻辑。

AI 回复流程：

```text
用户消息 / 群上下文
  ↓
输入过滤规则
  ↓
不通过：不处理，不进入 AI
  ↓
通过：AI 生成多条候选回复
  ↓
逐条输出校验
  ↓
不通过：转换后再校验 / 丢弃该条
  ↓
通过：发送
```

AI 评论流程：

```text
频道消息 / 评论对象 / 评论上下文
  ↓
输入过滤规则
  ↓
不通过：不处理，不进入 AI
  ↓
AI 生成多条评论候选
  ↓
逐条输出校验
  ↓
不通过：转换后再校验 / 丢弃该条
  ↓
通过：发送评论
  ↓
失败重试 / 记录结果
```

推荐默认策略：

```text
AI 回复：转换一次，仍失败则丢弃
AI 评论：转换一次，仍失败则丢弃
```

### 17.4 规则版本

每次修改规则都生成新版本。

任务绑定具体规则版本，或显式选择跟随当前发布版本。

这样可以保证：

- 运行中的任务不会被突然影响。
- 规则变更可追踪。
- 规则出问题可以回滚。
- 不同任务可以使用不同版本。

版本解析规则：

- 绑定固定版本的任务，执行项生成时直接使用绑定版本。
- 跟随当前发布版本的任务，执行项生成时解析当时的当前发布版本。
- 无论使用固定版本还是跟随当前发布版本，每个执行项生成时都必须固化 `resolved_rule_set_version_id`。
- 执行项、发送记录、命中记录和审计记录永远追溯到具体规则版本。
- 规则发布后，只影响之后新生成的执行项，不回写已经生成的执行项。

### 17.5 规则测试器

规则测试器按任务类型切换。

测试模式：

- 仅规则测试：不调用 AI，不创建执行项，不发送，只对输入样例和输出样例做规则判断。
- AI 干跑测试：可以调用 AI 生成候选内容，但不发送、不创建真实执行项，只展示校验、转换和丢弃结果。
- 历史样本回放：使用历史源消息、历史 AI 输出或历史评论场景回放规则效果，用于验证新版本是否会改变处理结果。

所有测试模式都必须标记为测试数据，不写入真实发送队列，不影响账号限额和任务进度。

监听转发测试展示：

- 是否通过输入过滤
- 命中了哪些规则
- 为什么被跳过或拦截
- 转换后内容是什么
- 会发到哪些目标群
- 每个目标群当前有多少可发送账号
- 预计使用哪种账号策略
- 是否触发限速
- 源媒体待缓存时是否进入等待队列
- 等待超时后缓存才完成时是否拒绝补发
- 旧版本缓存完成事件迟到时是否被拒绝
- 相册一张图失败时是否剔除失败图并保持剩余图片顺序
- 等待队列超量时是否直接放弃受影响素材缓存

AI 回复 / AI 评论测试展示：

- 输入是否通过过滤
- AI 候选内容列表
- 每条候选的输出校验结果
- 不通过原因
- 转换后内容
- 最终可发送内容
- 被丢弃内容及原因

规则上线前必须可以测试。

### 17.6 规则中心前端页面

规则中心前端不再展示独立“关键词库”。页面统一围绕规则集、规则版本、规则配置、规则测试和命中记录组织。

```text
规则中心
  ├─ 规则集
  ├─ 规则版本
  ├─ 规则配置
  ├─ 规则测试
  └─ 命中记录
```

页面分区：

- 规则中心首页：规则集名称、适用任务、当前版本、绑定任务数、最近发布、状态和操作。
- 规则编辑器：基础信息、输入过滤、输出校验、内容转换、路由分发、账号策略、限速策略、重试策略、发布设置。
- 规则版本页：版本号、状态、创建人、发布时间、绑定任务数、复制为草稿、发布、回滚和查看绑定任务。
- 规则测试器：按监听转发、AI 回复、AI 评论、普通发送切换测试表单和结果。
- 命中记录页：时间、任务、任务类型、规则集、规则版本、命中阶段、命中规则、处理动作、原始内容、处理后内容和结果。

命中记录分工：

- 规则中心命中记录解释输入过滤、输出校验、内容转换、路由分发和规则版本命中。
- 风控中心命中记录解释账号不可用、代理异常、限额 / 冷却、换号 / 转派、延后 / 暂停。
- 执行记录需要同时关联规则中心命中记录和风控中心命中记录。

任务创建页只保留：

```text
选择规则集
选择规则版本
```

如果不选版本，默认使用当前发布版本。

旧关键词库直接废弃，不迁入新规则集，不继续作为独立配置入口，也不作为任务可选项保留。新任务只选择规则集和规则版本；如确有需要，运营人员在新规则集中重新配置输入过滤、输出校验或路由条件。

限速与重试边界：

- 规则中心定义任务内的限速和重试偏好，例如单目标冷却、同内容去重、最大重试次数、失败后换号。
- 风控中心定义全局硬边界，例如账号小时 / 日上限、账号全局冷却、代理告警、FloodWait 处置和账号风险等级。
- 执行中心执行时必须先满足风控中心硬边界，再应用规则集限速和重试策略。
- 规则集策略不能放宽全局风控策略，只能在全局允许范围内进一步收紧或细化。

### 17.7 规则中心不使用人工审核口径

- 审核队列
- 审核通过
- 审核驳回

改为：

- 自动处理策略
- 输入过滤策略
- 输出校验策略
- 内容转换策略
- 路由分发策略
- 拦截策略
- 跳过策略
- 限速策略
- 重试策略

## 18. 风控中心功能架构

风控中心是发送安全和账号运营控制中心，负责全局发送节奏、账号评分、代理健康、策略分层、任务执行准入、异常账号处置和风控留痕。

风控中心不替代账号中心、规则中心或运营目标中心：

```text
账号中心：账号接入、登录、同步、分组和基础状态
规则中心：内容过滤、输出校验、转换、路由、账号策略、限速和重试规则
运营目标中心：目标资产、授权、账号覆盖和目标能力
风控中心：账号能不能参与运营、以什么强度参与、异常后怎么处置
```

代理软件与平台的边界：

```text
服务器代理软件：管理机场订阅、节点、分流和固定路由
TG 运营平台：只管理 socks5/http 本地代理地址、proxy_id 和账号绑定
```

代理风控采用告警驱动：

```text
无代理告警：任务和发送仍按账号分组 / 账号池使用
有代理告警：风控中心提醒处理本地代理或账号绑定
```

代理告警不是把代理变成任务选择维度，而是把代理作为账号运行环境进行健康约束。任务中心仍然按账号分组、账号池或手动账号范围创建任务；代理只作为健康徽标、发送前预检查结果、风控评分因素和异常处置入口。

代理告警状态建议分为：

```text
normal -> observing -> alerting -> acknowledged -> recovered
                       └-> disabled
                       └-> ignored
```

达到告警的典型条件包括：连续健康检查不可达、持续超时、认证失败、同代理多个账号异常聚集、同代理多个账号 FloodWait 异常增加、绑定账号数或并发连接超过建议上限。未达到告警条件时，代理异常只进入观察和评分解释，不打断日常任务创建。

代理告警通知采用分级提醒：`observing` 只在页面展示，`alerting` 进入风控中心待处理并生成站内通知，critical 级别才可选管理员 TG 通知；同一 `proxy_id` 在未恢复前需要去重，人工确认后不重复提醒，但新增失败仍记录命中。

代理健康检查由后台任务执行，检查本地端口、协议握手、认证、Telegram 连接探测和业务失败反馈。平台只检查本地 socks5/http 地址，不读取机场订阅，不解析节点列表。

### 18.1 前端页面

风控中心作为一级页面，建议放在规则中心之后、运营数据之前。

页面结构：

```text
风控中心
  ├─ 总览
  ├─ 全局策略
  ├─ 账号评分
  ├─ 代理告警
  ├─ 处置队列
  ├─ 命中记录
  └─ 策略审计
```

总览页用于值班，展示当前风控等级、静默状态、可用账号数、阻塞账号数、今日发送 / 互动量、最近 FloodWait、最近账号受限、最近转派、最近拦截和待处理处置项。

全局策略页维护发送抖动、批次间隔、静默时段、账号小时 / 日上限、账号全局冷却、默认重试、账号异常处理、API 限流处理和内容拦截处理。任务级策略只能在全局允许范围内覆盖。

账号评分页展示账号名称、登录状态、代理绑定、风控等级、健康分、当前策略、小时用量、日用量、冷却结束时间、最近风险和处置入口。

处置队列汇总待完成登录、Session 失效、代理异常、账号受限、FloodWait 频繁、疑似封禁、目标权限异常、待验证任务和连续内容拦截。

代理异常只有达到告警条件才进入处置队列；普通健康代理不打断任务创建和发送流程。

代理告警页展示 `proxy_id`、本地代理地址、绑定账号数、并发建议、最近健康检查、最近错误、告警状态、影响账号和建议动作。页面不展示机场订阅链接、机场节点名称或节点地区；节点修复由具备服务器代理维护权限的管理员在服务器代理软件里完成。

命中记录解释“为什么没发、为什么换号、为什么延后、为什么暂停”，字段包括命中阶段、账号、任务、目标、策略、原因、系统动作和结果。

### 18.2 账号评分与策略分层

账号使用不能只看“在线”，必须按评分和策略分层。

| 等级 | 分数 | 策略 |
| --- | --- | --- |
| A 健康 | 85-100 | 正常参与任务，标准节奏 |
| B 观察 | 70-84 | 可参与任务，轻微降频 |
| C 保守 | 55-69 | 只参与低频任务，降低并发 |
| D 冷却 | 30-54 | 不参与新任务，只保留恢复检查 |
| E 阻塞 | 0-29 | 暂停参与任务，需要人工处置 |

评分因素包括登录状态、Session 可用性、开发者应用健康、代理健康、代理未绑定、代理复用、最近成功执行、失败率、FloodWait、账号受限、目标权限失败、同步异常、发送密度、内容拦截关联，以及 AI 活跃群和转发监听中的账号角色一致性。

任务选号默认优先 A / B 账号；C 账号只在任务允许保守账号时参与；D / E 不参与新执行项。

### 18.3 账号使用与发送准入

账号参与发送必须通过五类检查：

```text
账号中心状态通过
  ↓
账号代理绑定和健康通过
  ↓
风控中心策略通过
  ↓
运营目标权限通过
  ↓
规则中心内容和任务规则通过
  ↓
才允许调用发送
```

任务中心创建任务时只配置账号范围，优先使用账号分组 / 账号池 / 手动账号选择；后端在生成执行项和执行前根据风控结果做最终账号决策。前端不能指定“最终一定用某个账号发送”，也不要求运营按代理端口来创建任务。

普通消息发送也必须走风控预检查：

```text
选择账号
  ↓
按账号过滤可发送目标
  ↓
输入内容
  ↓
发送前预检查
  ↓
创建发送任务
  ↓
Worker / 派发器再次校验
  ↓
调用 Telegram 网关发送
```

预检查返回需要覆盖四类决策：账号决策、代理决策、目标决策、内容决策。代理决策至少包含代理健康、告警状态、是否阻塞、原因码、建议动作和关联告警 ID。前端预检查只用于提前提示和减少无效提交，Worker / 派发器执行前必须重新校验一次。

代理相关原因码建议包括：

```text
proxy_missing
proxy_unreachable
proxy_timeout
proxy_auth_failed
proxy_alert_active
proxy_reuse_over_limit
proxy_concurrency_over_limit
proxy_floodwait_cluster
```

账号首次登录、验证码、扫码、2FA、Session 刷新、同步、监听启动、目标权限探测和发送，都必须通过账号绑定代理创建 Telegram 客户端；代理缺失或不可用时不能裸连 Telegram。

策略优先级：

```text
硬阻塞事实
  账号未登录 / Session 失效 / 代理严重告警 / 目标无权限 / 账号禁用
  ↓
全局风控硬策略
  静默期 / 全局暂停 / 账号 E 级阻塞 / 代理 disabled
  ↓
规则中心内容硬拦截
  内容禁止发送 / 链接不允许 / 输出校验失败
  ↓
限额与冷却
  账号小时日上限 / 目标冷却 / 代理并发上限
  ↓
任务级偏好
  账号池 / 权重 / 目标粘性 / 任务重试策略
  ↓
执行队列优先级
  运行中任务优先于已暂停任务 / 高优先级任务优先于低优先级任务
```

硬阻塞和全局风控硬策略不能被任务级配置覆盖；人工确认代理告警只减少提醒，不代表解除发送阻塞。

### 18.4 账号转派与处置

当执行项原账号因为限额、冷却、目标权限或账号异常不可用时，系统优先在同一任务账号池中寻找替换账号。

候选账号必须满足：

- 在线且未禁用。
- 风控等级允许参与。
- 未超过小时 / 日上限。
- 不在冷却中。
- 绑定代理健康。
- 对目标有发送、监听、评论、浏览或点赞权限。
- AI 活跃群和转发监听任务中角色和行为一致性允许。

找到替换账号时，执行项记录原账号、替换账号和转派原因；找不到替换账号时，按策略延后、跳过、暂停任务或停止任务。

需要人工处理的风险进入处置队列，动作包括：完成登录、必要时重新登录、账号自动探测、立即探测、解除群限制、目标能力重查、检查代理、切换 `proxy_id`、暂停账号、设置冷却、恢复账号、替换任务账号和禁用账号。处置项不能靠简单“标记已处理”恢复，必须触发对应检查并记录恢复证据。

### 18.5 与运营数据的关系

运营数据负责复盘和统计，风控中心负责运行控制和处置。运营数据可以展示风险趋势和历史统计，但不再作为风控策略和账号处置的主入口。

运营数据不再保留“全局风控中心”策略抽屉和账号处置入口，只展示指标、趋势、命中明细和风控中心跳转。

## 19. AI 内容中心功能架构

AI 内容中心不只是配置模型，还负责 AI 任务中的内容生产。

主要能力：

- 群上下文摘要
- AI 活跃群对话生成
- 频道评论生成
- 频道回复生成
- 转发内容改写
- 内容自动校验
- 账号角色一致性检查
- Token 和费用统计
- AI 生成记录

### 19.1 AI 生成原则

AI 输出应该结构化。

例如 AI 活跃群不是输出一段话，而是输出：

- 本轮目的
- 多条发言
- 每条发言的角色
- 每条发言的意图
- 每条发言的延迟
- 是否回复某条消息
- 风险级别
- 素材意图，例如 `material_intent`
- 是否允许规则引擎选择图片或表情包

频道评论 / 回复也应输出：

- 评论内容
- 评论风格
- 关联消息
- 使用账号角色
- 发送建议时间
- 自动校验状态

### 19.2 AI 自动校验

AI 生成内容不走人工审核，但必须进入规则中心的输出校验链路。AI 回复和 AI 评论通常一次生成多条候选内容，系统必须逐条校验、逐条转换、逐条决定发送或丢弃，不能因为一条候选失败就废弃整批结果。

输出校验包括：

- 禁止关键词检查
- 禁止链接检查
- 禁止 @ / 联系方式检查
- 重复内容检查
- 长度检查
- 引流表达检查
- 账号角色一致性检查
- 目标群规则检查
- 上下文过期检查
- 格式检查

逐条处理结果：

- 通过
- 转换后再校验
- 丢弃该条
- 重新生成一次
- 拦截并记录原因

推荐默认策略：

```text
AI 回复：转换一次，仍失败则丢弃
AI 评论：转换一次，仍失败则丢弃
```

## 20. 执行中心功能架构

执行中心负责把任务计划真正落地。

它是后台执行能力，不一定是单独页面，但需要在任务详情和执行记录里体现。

主要能力：

- 任务计划生成
- 账号分配
- 发送排队
- 风控准入
- 冷却判断
- 账号转派
- 失败重试
- 执行状态更新
- 上下文过期取消
- 任务进度汇总
- 异常恢复

### 20.1 执行前检查

每个执行项发送前都要检查：

- 任务是否仍在运行
- 当前时间是否允许执行
- 账号中心状态是否允许参与任务
- 账号是否绑定可用本地代理
- 风控中心是否允许该账号以当前强度参与
- 账号是否可访问目标
- 账号是否可发言
- 账号是否冷却结束
- 是否超过上限
- 内容是否通过规则中心自动校验
- 目标是否仍可用
- 上下文是否过期
- 是否命中去重规则

执行中心调用发送前的准入链路：

```text
加载执行项
  ↓
加载任务、账号、目标、规则版本
  ↓
账号中心状态检查
  ↓
proxy_id 和本地代理健康检查
  ↓
风控中心评分、限额、冷却和策略检查
  ↓
运营目标权限检查
  ↓
规则中心内容与任务规则检查
  ↓
读取账号 session 和开发者应用凭据
  ↓
读取账号绑定代理地址
  ↓
调用 Telegram 网关
```

### 20.2 失败处理

失败后需要明确：

- 失败原因
- 是否可重试
- 是否换账号重试
- 是否进入冷却
- 是否暂停目标
- 是否暂停任务
- 是否需要重新规划
- 是否需要进入风控处置队列
- 是否需要进入代理异常处置

失败原因要能被运营人员看懂，例如：

- 账号离线
- 账号无目标权限
- 目标群不可发言
- 账号冷却中
- 账号代理不可用
- 账号未绑定代理
- 超过每日上限
- 内容被自动拦截
- 上下文过期
- 发送失败
- 媒体不可发送
- 规则转换失败

发送结果需要回写：

- 执行项状态、远端消息 ID、失败类型和失败详情。
- 账号状态、最近活跃时间、健康分、冷却结束时间。
- 风控事件、代理异常、命中策略、转派原因、延后原因或暂停原因。
- 任务完成数、失败数、跳过数、转派数和重试数。
- 审计记录。

当前任务中心已按默认失败策略处理三类高频异常：

- 账号不可用 / 账号受限：可配置为跳过账号、暂停任务或停止任务，并记录最近一次策略命中；停止任务时会跳过同任务剩余待执行项。
- API 限流 / 慢速模式：可配置为等待重试、跳过或暂停；等待重试会从错误信息中解析等待秒数并延后执行项。
- 内容拦截：可配置为跳过消息、改写重试或暂停任务；改写重试会先对命中的关键词、群禁词和非白名单链接做确定性改写，改写后延后重试，无法产生可用改写时才标记为待改写。

### 20.3 千账号执行层升级

面向 1000 个账号时，执行中心不能只依赖单 worker 顺序 drain。它需要升级为“原子领取 + 多 worker + 有界并发 + 多维限流”的执行模型。

执行项领取应采用短事务：

```text
BEGIN
  SELECT due pending actions
  JOIN running tasks
  APPLY task priority / type weight / per-task quota
  APPLY account pool fairness
  FOR UPDATE SKIP LOCKED
  LIMIT claim_limit

  在 claim 阶段完成最终账号选择 / 转派 / 延后判断

  UPDATE selected actions
  SET status = 'executing',
      lease_owner = current_worker,
      lease_expires_at = now() + lease_seconds
COMMIT

并发执行已领取 actions
逐条短事务回写结果
```

执行中心扩容后的职责分层：

- Planner 只生成 action，不调用 TG API。
- Dispatcher 只执行已领取 action，不重复扫描任务规划。
- Listener 独立采集事件，不拖慢发送执行。
- Recovery 处理 lease 过期、worker 失联和执行超时。
- Metrics 汇总 pending、executing、最老 pending 等待时间、TG API 耗时、FloodWait、SlowMode、代理异常和 worker heartbeat。
- `execution_attempts` 在调用 Telegram Gateway 前写入 `before_call_at`，进入 Gateway 调用边界前写入 `gateway_call_started_at`；Recovery 只能根据已进入 Gateway 调用边界的未完成 attempt 推断 `unknown_after_send`，不得直接重发。

运行时必须限制：

- 平台全局 TG API 并发。
- 单 worker 并发。
- 单账号并发和冷却。
- 单代理并发和熔断。
- 单目标群慢速模式和群级冷却。
- 单任务并发和节奏。
- 媒体下载 / 重传并发。
- 限流统一使用 Redis token bucket；token 获取必须在 action 进入 `executing` 前完成，推荐使用带 TTL 的 reservation；Redis 不可用时默认 fail-closed。
- 同一账号默认只能被一个 worker 同时使用，账号 in-flight lock 负责互斥，token bucket 负责速率。
- 数据库连接池按 API、Planner、Dispatcher、Listener、Recovery、Metrics 和预留人工连接统一计算。
- 运行明细默认保留最近 5 个自然日，第 6 天先汇总第 1 天总数，再滚动删除第 1 天全部运行明细；删除后长期只保留汇总总数和清理审计。

该升级的详细设计、索引建议、前端观测项、部署参数和压测验收见 `docs/capacity-and-dispatch-upgrade-plan.md`。

### P1：千账号容量与 Dispatcher 执行层升级

目标：

```text
把当前顺序 drain 的执行层升级为可横向扩容的多 worker 执行架构。
```

改造内容：

- 新增 Action 原子领取机制，使用数据库行锁和 `SKIP LOCKED` 防止多 worker 重复执行。
- Planner 同时落地 `plan_batch_key` 和 `action_dedupe_key`，避免批次重跑或同窗口多事件造成重复 / 误拦。
- Planner 增加全局和单任务 pending 积压保护，防止 Dispatcher 被限流或 Redis fail-closed 后继续无限生成 action。
- 将任务规划和 action 执行拆成 Planner 与 Dispatcher 两类职责。
- Dispatcher claim 按任务状态、任务优先级、任务类型权重、每任务 claim 配额、账号池公平性、防饥饿 aging 和 action 到期时间领取。
- Dispatcher claim 改为两段式短事务：DB 预领取写 `claiming`，事务外获取 Redis token、账号 in-flight lock、代理和目标配额，再用条件更新进入 `executing`。
- claim 前先按当前 worker 账号分片筛选候选账号池，账号转派只在本分片内完成。
- Redis token bucket 在 action 进入 `executing` 前获取；未拿到 token 的 action 保持 `pending` 并延后 `scheduled_at`。
- Redis token bucket 使用带 TTL 的 reservation 或接受保守 token 浪费，不能因 DB 更新失败绕过 DB 状态直接调用 TG。
- 增加账号 in-flight lock，保证同一账号默认不会被多个 worker 同时使用。
- 增加 `execution_attempts` 或等价执行尝试记录，在进入 Gateway 调用边界前写入 `gateway_call_started_at`，并由 Recovery 推断 `unknown_after_send`。
- Dispatcher 内部增加有界并发，默认从小并发开始逐步压测。
- 执行前统一经过账号、代理、目标、规则和风控准入链路。
- 增加基于 Redis token bucket 的账号级、代理级、目标级、任务级、任务类型级、媒体发送和全局 TG API 限流；Redis 不可用时默认 fail-closed。
- 补齐数据库连接池与 worker 并发公式。
- 将监听采集从发送执行 drain 中拆出，形成独立 Listener Worker。
- Listener 增加 source claim、事件水位、唯一事件键、回补窗口、bot 消息过滤、media group 聚合、编辑 / 删除事件口径。
- 完善 Recovery Worker，对 lease 过期、worker 心跳失联和执行超时做可解释恢复。
- 增加 Metrics Worker 或运行快照，记录 pending 数、executing 数、最老 pending 等待时间、FloodWait、SlowMode、代理异常和 worker heartbeat。
- action、listener event、execution attempt 和 daily stats 增加唯一约束，保证 Planner、Listener、执行尝试和汇总幂等。
- 运行明细默认保留最近 5 个自然日，第 6 天先汇总第 1 天总数，再滚动删除第 1 天全部运行明细；长期只保留汇总总数和清理审计，不再保留窗口外单条异常明细。
- 补充面向 1000 账号、20-30 个持续任务的压测脚本和容量报告。

验收标准：

- 同时运行 2-4 个 Dispatcher Worker 时，同一条 action 不会重复执行。
- 单 worker 内部可以并发执行，但同一账号不会被并发滥用。
- 已调用 TG 但本地结果未知的 action 进入 `unknown_after_send`，不会被自动重复发送。
- Redis 不可用时 Dispatcher 不会 fail-open 继续发送。
- worker 异常退出后，lease 过期 action 能被恢复并进入既有失败 / 重试策略。
- 监听压力增大时，不会拖慢普通发送 action 的执行。
- Listener 重复采集不会产生重复事件，短暂采集失败后可按水位回补。
- 任务中心和运营概览能看到队列积压、最老等待时间、worker 心跳和主要失败原因。
- 压测报告能给出推荐 worker 数、并发数、claim limit、PostgreSQL 连接池和单机容量边界。

## 27. 最终新版架构总结

新版 TG 运营管理平台应该重构成：

> 以账号中心、运营目标中心、任务中心、监听中心、规则中心、风控中心、AI 内容中心、执行中心、数据中心和审计中心为核心的 Telegram 运营后台。

三类核心任务分别是：

```text
频道互动任务：
父任务 → 频道消息 → 动作子任务 → 账号执行项

AI 活跃群任务：
父任务 → 活跃循环 Cycle → 发言 Turn → 自动校验 → 账号发送

转发监听群任务：
父任务 → 源群事件 → 规则过滤 / 转换 / 路由 → 转发批次 → 目标群发送
```

关键变化是：

```text
旧模式：
任务直接操作账号和目标，执行记录混在一起。

新版：
账号中心提供资源。
账号绑定固定本地代理出口。
目标中心确认范围。
规则中心定义规则。
风控中心决定账号能不能用、怎么用、出问题后怎么处理。
监听中心捕获事件。
任务中心负责编排。
AI 中心生成内容。
执行中心负责发送。
数据与审计中心负责追踪。
```

最终运营链路变成：

```text
账号
  ↓
代理
  ↓
目标
  ↓
规则
  ↓
风控
  ↓
后台账号权限
  ↓
任务
  ↓
监听 / AI / 执行
  ↓
数据
  ↓
审计
```

最重要的新版原则是：

1. 频道任务按消息拆成子任务。
2. 频道和源群监听都要聚合，不能按任务重复监听。
3. AI 活跃群按 Cycle / Turn 循环执行。
4. 转发监听群按源事件和系统级规则自动执行。
5. AI 活跃群和转发监听群都不需要人工审核后发送。
6. 账号发送必须经过账号状态、代理健康、风控策略、目标权限和规则内容检查。
7. 后台账号权限只限制菜单入口和操作按钮，不做业务数据隔离。
8. 所有自动生成、过滤、转换、发送、跳过、失败、重试、转派和处置都必须可追踪、可解释、可审计。
