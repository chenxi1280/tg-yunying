## 1. 项目定位

TG 运营管理平台面向 Telegram 运营团队，用一个后台统一管理 TG 账号、运营目标、消息发送、频道互动、AI 活跃群、转发监听、规则、风控、归档、运营数据和审计。

平台核心目标：

```text
账号接入
  -> 资产同步
  -> 目标确认
  -> 目标画像
  -> 规则配置
  -> 风控策略
  -> 任务创建
  -> 监听 / AI / Worker 执行
  -> 任务详情追踪
  -> 数据复盘
  -> 审计留痕
```

新版系统不再以旧 Campaign、旧 AI 草稿审核、多租户 SaaS、卡密或订阅套餐作为主线。历史模型可以保留为兼容层，但不能进入新版运营主流程。

系统边界：

- 只能使用已接入并授权的 TG 账号。
- 任务创建时可以选择已有运营目标，也可以粘贴群聊 / 频道入口，由系统自动解析并创建或复用运营目标。
- 只能监听已授权源群、群聊、频道或讨论组。
- 所有发送、监听、过滤、转换、AI 生成、失败、重试都必须留痕。
- 后台支持多个内部账号共同使用，权限做到菜单、按钮和写接口级别。
- 高风险动作依靠二次确认、原因填写和审计追踪控制。

产品原则：

```text
可配置
可自动执行
可追踪
可暂停
可重试
可解释
可风控
可审计
```

AI 活跃群和转发监听群不走人工审核主线。系统通过规则过滤、自动校验、风控、限速、上下文检查、失败重试和审计来保证任务可控。

## 2. 当前实现状态

本节按当前代码口径总结，不等同于最终目标。

### 2.1 已落地能力

- 前端主导航已覆盖：运营中心、TG 账号管理、运营目标、消息发送、素材中心、任务中心、监听中心、规则中心、风控中心、归档中心、运营数据、系统设置、审计记录；目标口径需新增“目标画像”一级菜单，承载全站唯一 AI 画像。
- 后端主路由已覆盖账号、账号池、开发者应用、AI 配置、运营目标、消息发送、任务中心、监听与规则汇总、运营数据、归档和审计。
- 旧 Campaign、旧 operation-task、旧 review 路由已从默认主路径隔离，仅在显式兼容开关打开时进入旧口径。
- 任务中心主任务类型为 5 类：AI 活跃群、转发监听群、频道浏览、频道点赞、频道评论/回复；资料初始化批次属于只读系统任务投影，不计入可创建的主任务类型。
- 任务中心支持创建、创建并启动、启动、暂停、继续、停止、重试、重置、编辑配置、删除、详情查看和执行记录追踪。
- 频道任务默认按 `dynamic_new` 持续采集新消息，支持容量检查、每条频道消息执行明细、账号执行项和容量不足提示。
- 频道任务启动前会统一检查候选账号对目标频道的关注状态；任务创建时可以直接粘贴 `@username`、公开频道链接、`https://t.me/+...` 或 `joinchat` 邀请链接，系统自动创建或复用运营目标；已关注账号进入主互动，未关注账号先按抖动节奏关注频道，成功后才进入浏览、点赞、评论或回复。
- AI 活跃群已按 Cycle / Turn 记录发言计划，支持账号角色映射、近期账号记忆、长期账号画像、AI 生成记录、内容过滤、基础重复过滤、上下文采集和账号冷却；仍需升级为接话 / 暖场 / 沉默三态质量模型。
- 转发监听群已接入源群上下文采集、规则版本、过滤、转换、路由、去重、目标群发送项和转发批次追踪。
- 监听中心已有聚合页面和 `/api/listeners/summary`，按频道 / 群展示关联任务、监听账号、事件积压、最后事件和错误。
- 规则中心已有规则集版本管理、规则测试、规则冲突提示、规则命中统计和转发归因。
- 运营数据已有 `/api/operation-metrics/summary` 真实汇总接口。
- 审计记录支持详情抽屉和 CSV 导出。
- 素材中心已从系统设置拆为一级菜单，复用现有素材上传、批量上传、编辑、列表和缓存健康能力；系统设置只保留提示词与素材运行配置。
- 前端操作手册已作为文档面同步最近更新功能，覆盖运营中心日常入口、素材中心、运营方案模板、任务创建动态向导、账号资产与可用性、数据汇总延迟、导航升级、账号安全加固、批量资料初始化、任务内目标输入和频道任务关注前置。
- 执行层已有 Task / Action 模型、Action claim、执行租约、worker heartbeat、监听运行层、账号容量检查、Redis token bucket 基础和 planner / dispatcher / listener / recovery / metrics 分角色 drain 能力。

### 2.2 仍需收敛的问题

- `groups` 与 `operation_targets` 仍处在双轨过渡期。用户侧入口应优先使用运营目标，执行层仍可保留群资产作为 TG 授权、账号权限和上下文采集承载。
- 任务创建体验仍需继续收敛为“创建任务时选择已有目标或粘贴新目标，后端自动 upsert 运营目标并补齐关注 / 加入前置”；编辑任务不再承载新目标输入。
- 群聊加入前置还需要和频道关注前置统一，AI 活跃群、转发监听群和频道互动任务都应走同一套账号-目标准入模型，并区分源群只需可监听、目标群必须可发送。
- 入群验证需要支持 AI 自动处理文本问题、简单算数题和固定问答，失败时标记人工处理或验证失败。
- 任务创建页仍暴露部分执行参数，例如抖动、静默最多账号、静默每轮发言、爬坡比例、上下文过期。应改为“默认可运行、解释可理解、高级可覆盖”的创建体验。
- AI 活跃群仍需从“定时生成多条聊天”收敛为“接真人话为主、低频暖场为辅”的质量模型，补齐事实锚点、语义去重、幻觉拦截、低置信沉默和质量留痕。
- 目标画像需要从运营目标详情中抽离为全站唯一画像中心。AI 活跃群和频道评论 / 回复共用同一画像；运营目标只作为学习来源和执行目标，不能让用户误解成每个群 / 频道各有一份画像。
- 规则中心仍需继续取消独立关键词库口径，将关键词、敏感词、白名单词、路由词统一收进规则集条件。
- 后台账号与权限仍需补完整闭环：后台账号登录、用户管理、菜单和按钮权限、后端写接口校验、权限变更审计。
- 单账号代理、代理健康检查、代理告警状态机和风控中心处置闭环仍是增强项。
- 手机号不脱敏展示已进入本轮闭环：所有涉及账号或联系人的页面、联系人、归档、风控、账号安全、审计记录和导出日志都必须优先使用完整 `phone_number`；`phone_masked` 只作为历史数据缺失完整手机号时的兼容兜底。
- 面向 1000 账号和 20-30 个持续任务，当前已具备 role 化 drain 能力；仍需确认生产部署是否已拆成多个独立 worker 进程，并继续补齐并发配额、跨进程账号锁、容量面板、token 预留 / 退款、运行快照和压测口径。

### 2.3 2026-05-20 更新记录

本日更新重点围绕账号资料初始化和频道任务启动前置条件，已落入代码和测试：

- 账号资料初始化：批量资料预览改为一次 AI 请求生成整批账号资料，避免逐账号调用导致慢和超时；前端资料预览请求支持按账号数量放宽超时时间。
- 账号资料初始化：新增命名风格提示 `custom_prompt`，默认引导生成真实 TG 用户昵称，而不是正式姓名或营销号名称；本地兜底昵称池也改为更自然的网名风格。
- 账号资料初始化：AI 供应商选择改为优先使用健康默认供应商，默认不可用时回落到首个健康供应商；AI 超时、mock 或无健康供应商时仍生成本地兜底预览，并在预览里展示原因。
- 账号资料初始化：资料动作的预检不再强制同步登录设备安全快照；缺失或不可用头像来源不阻塞昵称、简介和 username 执行，只跳过头像并展示警告。
- TG 账号管理：批量入口拆成“资料初始化”“设置二步密码”“清理登录设备”，降低把资料初始化和安全加固混在同一动作里的误操作风险。
- 频道互动任务：频道浏览、点赞、评论和回复启动前先检查关注状态；已关注账号先进入主互动，未关注账号进入准入前置阶段，关注成功后再追加进入后续互动容量。AI 活跃群和转发目标群还必须确认账号已加入且可发言；转发源群只要求已加入 / 可读取。现有代码仍以 `ensure_channel_membership` 承载频道历史动作，后续需要升级为通用 `ensure_target_membership`。
- 频道互动任务：新增 PostgreSQL 级回归测试覆盖关注前置、全部关注失败阻断主互动、运行时守卫和旧任务守卫；后续实现和测试继续按“先检查关注状态、未关注先关注、成功后互动”的 PRD 口径校正。
- 账号新增：PostgreSQL 下创建账号前会同步 `tg_accounts.id` 序列，避免历史导入或手工插入后序列落后导致新增账号主键冲突。
- 前端操作手册：新增“最近更新功能”说明，并同步到 TG 账号管理、运营目标、任务中心、风控中心和上线前检查口径。
- 文档口径：手机号不脱敏展示列为本轮已定口径；代码里仍存在的 `phone_masked` 是历史兼容兜底，PRD 和 ops 文档不能再把脱敏手机号作为主展示字段。

### 2.5 2026-05-27 线上 AI 活跃群排障记录

- 本次“AI 活跃群没有账号发送消息”不是单一验证码问题，而是任务暂停、目标能力旧诊断、准入候选截断、单账号禁言和 AI 供应商读超时共同造成的可见性问题。
- 目标群诊断必须以当前 Telegram 事实为准。群内没有对应验证机器人或最新验证消息时，页面不能继续展示“方丈机器人图形验证码未通过”作为当前原因。
- 普通 supergroup 成员权限探测要正确处理 Telethon 权限对象缺少 `send_messages` 字段的情况；缺字段表示需要结合默认权限和实际参与状态判断，不能直接当成 `send_messages=false`。
- 入群 / 关注准入要覆盖任务账号配置选中的全部在线账号；`max_concurrent`、发送容量、账号冷却和健康权重只控制主互动并发与选择，不控制准入候选数量。
- 任务中心和运营中心必须显著区分“已暂停”“启动校验中”“准入补齐中”“等待 AI”“等待上下文”“等待冷却 / 下一轮”和“发送中”。只展示 `running` 会掩盖真实卡点，只展示 `paused` 小标签会让运营误以为任务仍会继续。
- AI 供应商超时、空响应或无健康供应商要独立展示为 AI 运行不可用，不得被合并到账号限制、群验证码或目标权限异常。

## 3. 系统总架构

新版系统按功能中心组织。面向运营人员的主入口是运营中心；素材中心独立承载表情包、头像包和媒体素材；任务中心退到执行详情 / 调度控制台；系统设置只维护平台底座能力。

1. 运营中心
2. TG 账号中心
3. 运营目标中心
4. 消息发送中心
5. 素材中心
6. 任务中心
7. 监听中心
8. 规则中心
9. 风控中心
10. 数据、归档与审计中心
11. 系统设置中心
12. 后台账号与权限能力（系统设置子模块）
13. 执行中心

这里的“能力”表示产品功能边界，不等同于一级导航。素材中心必须是一级菜单；AI 供应商、提示词、素材运行配置和后台账号权限归入系统设置 Tab。运营中心、任务中心、规则中心、账号资料初始化和消息发送只引用素材中心资产。

模块关系：

```text
系统设置
  提供 TG 开发者应用、AI 供应商、提示词、素材运行配置、权限和基础运行配置。

素材中心
  提供表情包库、头像包、图片、文件、组合消息、批量上传、分组标签、缓存健康和引用关系。

TG 账号中心
  提供可登录、可同步、可绑定代理、可发送、可监听的账号资源。

运营目标中心
  把账号同步到的群、频道、联系人整理成可运营对象。

规则中心
  提供过滤、转换、路由、账号策略、限速、重试策略和输出校验。

风控中心
  负责统一账号健康分解释、发送准入、代理资源、代理健康、目标能力、限流、冷却、处置和告警。

运营中心
  以运营目标为默认视角，展示今日工作台、运营方案、目标异常、任务失败聚合和建议动作。

任务中心
  把业务意图或运营方案拆成 Task 和 Action，记录执行明细、失败事实、准入子任务和调度控制。

执行中心
  负责 Planner、Dispatcher、Listener、Recovery、Metrics 等后台执行角色。

运营数据 / 归档 / 审计
  提供统计、追溯、归档和审计闭环。
```

三层边界：

| 中心 | 定位 | 不承载 |
| --- | --- | --- |
| 系统配置 | 平台底座能力，管理员维护 | 群活跃策略、频道互动策略、任务节奏、具体失败处理 |
| 运营中心 | 日常工作台 + 运营方案 + 异常处理入口，运营人员默认入口 | action 原始明细、底层供应商配置 |
| 任务中心 | 执行详情 / 失败事实源 / 调度控制台 | 长期运营方案、默认运营工作台 |

任务中心负责记录失败，运营中心负责暴露失败、归因失败、引导处理失败。运营中心默认按运营目标聚合异常，点开目标后查看关联任务失败，并优先打开上下文弹窗 / 抽屉处理；复杂流程才深链跳转到任务详情、账号处理、目标处理或规则处理。

运营中心上下文处理分三层：

| 层级 | 适用场景 | 交互 |
| --- | --- | --- |
| 轻处理 | 查看异常原因、账号不可用原因、规则命中、风控限制说明、确认处理 / 忽略 / 标记已处理、刷新当前 issue 摘要 | 在运营中心弹窗内完成，关闭后保留目标展开、筛选、分页和滚动位置 |
| 中等处理 | 重试失败项确认、同步资产触发、同步安全状态触发、目标准入失败处理、目标能力调整、规则版本选择、方案影响预览、任务暂停 / 恢复确认 | 打开右侧处理抽屉，只调用对应模块已定义的小闭环接口；抽屉顶部展示目标、任务、账号、失败类型和 issue 来源 |
| 重处理 | 复杂登录、批量资料初始化、设置二步密码、清理登录设备、完整规则集编辑、完整运营方案编辑、大量 action / attempt 明细 | 深链跳转到对应页面，并携带 `return_to`、`source_issue_id`、目标 ID、任务 ID、默认 Tab 和筛选条件 |

深链返回运营中心时必须恢复原目标、原 issue、筛选、分页、展开行和滚动位置，避免运营人员处理完一个问题后重新查找目标。

运营中心的方案模板是业务入口，不是系统配置：

- 群活跃方案生成 `group_ai_chat` 任务，重点配置接话 / 暖场、话题方向、账号角色、AI 黑话和质量策略。
- 转发监听方案生成 `group_relay` 任务，重点配置源群、目标群、过滤、转换、路由、去重和来源排除。
- 频道互动方案生成 `channel_view`、`channel_like`、`channel_comment` 任务，默认面向新消息流，指定消息只作为快捷入口。
- 账号使用策略只生成账号范围、冷却、容量、换号和恢复建议；账号安全、资料初始化、2FA 和设备清理仍跳转账号中心执行，不替代账号中心批次明细。

方案保存只更新方案配置；点击“生成任务草稿”或“生成并启动任务”才创建任务。方案调整运行中任务前必须展示影响预览，确认后再更新任务配置。

## 5. 系统设置 - 后台账号与权限

后台不再假设所有用户都是同级 admin。权限目标是控制菜单、按钮和写接口，不做数据隔离。

角色建议：

| 角色 | 职责 |
| --- | --- |
| 平台管理员 | 全部配置、账号接入、任务管理、风控处置、用户权限 |
| 运营主管 | 运营中心、目标、消息、任务、规则、数据和审计查看，部分风控处置 |
| 运营人员 | 创建和维护任务、查看执行结果、处理日常异常 |
| 账号添加专员 | 只负责 TG 账号接入、登录、恢复和基础信息同步 |
| 只读观察员 | 查看概览、任务、数据和审计，不执行写操作 |

权限粒度：

- 菜单权限：控制导航入口。
- 按钮权限：控制创建、编辑、启动、暂停、删除、导出、切换账号等操作。
- 写接口权限：后端必须二次校验，不能只靠前端隐藏按钮。
- 危险动作：停止任务、删除任务、重置 / 重排计划、手动 drain、切换监听账号、恢复账号、修改风控策略、查看验证码和导出敏感数据必须填写原因并写审计。

账号添加专员只能访问：

- TG 开发者应用查看或选择。
- 账号登录 / 二维码 / 验证码 / 2FA 流程。
- 账号同步状态。
- 账号接入失败原因。

账号添加专员不能访问：

- 任务中心。
- 消息发送。
- 规则中心。
- 风控策略。
- 运营数据。
- 审计导出。

账号接入值班排障口径：

- 看到“操作失败 token expired”时，先按管理后台登录态过期处理，不判断为 TG 账号登录失败。
- 核对后端访问日志中的 `/api/tg-accounts`、`/login/start`、`/login/verify` 状态码，以及账号表 / 登录流水的最终状态；如果账号已经 `online`，说明 Telegram 登录链路已完成。
- 管理后台认证 401 必须触发前端清理本地 token、关闭当前业务弹窗并回到后台登录；页面不得把 `token expired` 混入账号失败原因或登录流水。

## 6. 系统初始化中心

系统初始化中心只放运行基础配置，不承载任务节奏和风控策略。

包括：

- TG 开发者应用：`api_id`、`api_hash`、应用名称、使用状态、账号绑定情况。

- 系统运行配置：数据库、Redis、worker、媒体目录、环境开关。

不包括：

- 全局发送节奏。
- 全局静默时段。
- 默认失败处理策略。
- 账号健康分、风险等级和扣分原因。
- 处置队列。
- 代理新增、编辑、禁用和账号绑定；这些归风控中心。
- 命中记录。

这些统一归入风控中心。

## 7. TG 账号中心

账号中心负责账号资产和账号维护，不负责发送、不负责运营方案、不负责风控处置工作台。

账号中心负责：

- 账号接入、登录恢复、验证码 / 扫码 / 2FA 登录续办。
- 完整手机号、昵称、username、头像、账号分组、开发者应用和代理绑定管理。
- 资料、联系人、群、频道同步结果展示。
- 资料初始化、登录设备清理、二步密码托管和同步安全状态。
- 账号健康分、账号状态、可用性摘要和不可用原因展示。

账号中心不负责：

- 联系人发送、人员发送、素材选择和消息内容编辑；这些统一在消息发送页。
- AI 活跃群、转发监听、频道互动等运营方案；这些统一在运营中心。
- 风控策略、处置队列、限制解除和命中记录；这些统一在风控中心。
- 在运营中心铺受限账号清单。运营中心只展示目标 / 任务受到账号问题影响后的摘要和跳转。

### 7.1 账号状态

建议状态模型：

| 状态 | 含义 | 任务是否可用 |
| --- | --- | --- |
| 在线 | session 可用，最近连接正常 | 需要再看容量、目标权限和风控 |
| 待登录 | 已创建但未开始登录 | 不可用 |
| 等待验证码 / 等待扫码 / 等待 2FA | 登录流程待继续 | 不可用 |
| 需重新登录 | session 失效或 2FA 失败 | 不可用 |
| 受限 | 账号被 TG 限制发送或互动 | 不可用或只读 |
| 疑似封禁 / 已封禁 | 多次失败或明确不可用 | 不可用 |
| 异常 | 代理、权限、同步或登录异常 | 按可用性和风控判断 |
| 禁用 | 人工停用 | 不可用 |

### 7.2 账号详情

账号详情需要展示：

- 基础资料：昵称、用户名、完整手机号、头像、简介、账号状态、账号健康分。账号健康分是全平台唯一账号分数，账号中心和风控中心必须同源同值；展示层优先使用 `phone_number`，`phone_masked` 仅作为历史数据缺失完整手机号时的兼容兜底。
- 登录 / 验证：session 状态、当前登录流、验证码、二维码、2FA、最近登录时间。
- 同步资产：资料、联系人、群、频道最近同步结果。联系人只读展示，不在本页发消息。
- 分组与运行环境：账号分组、开发者应用、代理绑定、代理健康和最近错误。
- 可用性摘要：可发送、可监听、可加入、可评论、容量、不可用原因和下次可重试时间。
- 账号安全：平台可信设备、外部设备、2FA、安全快照和最近加固结果。
- 维护批次：资料初始化、设置 2FA、清理登录设备批次和失败重试。
- 执行记录：该账号最近参与的发送、评论、浏览、点赞、监听记录，只读下钻；实际发送和任务控制跳转到消息发送或任务中心。
- 审计记录：移动分组、资料修改、重新登录、敏感查看、禁用等操作记录。

### 7.3 账号分组与账号池

账号分组用于账号资源分类、批量维护和任务 / 发送范围选择，不是权限隔离，也不是发送页面。

账号中心内允许：

- 新建、重命名、禁用账号分组。
- 查看组内账号、批量移动账号、批量资料初始化、批量设置 2FA、批量清理设备。
- 为任务中心、消息发送、风控策略和报表提供账号范围。

账号中心内禁止：

- 不在进入账号分组后展示联系人发送、人员发送、消息编辑或素材选择。
- 不在账号分组里配置运营方案、AI 接话策略、转发规则或任务节奏。
- 不用账号分组代替权限。权限仍由后台账号权限和审计控制。

任务创建时账号范围可选择：

- 全部可用账号。
- 指定账号分组。
- 手动选择账号。
- 按目标推荐账号。

执行时必须再次经过风控中心和账号容量检查。前端预检只用于减少无效提交，不能替代 worker 执行前校验。

消息发送页也可以选择账号分组作为发送账号范围，但发送内容、联系人选择、素材选择、预检和发送记录都留在消息发送页。

### 7.4 代理绑定

代理不是日常任务选择主维度。没有代理告警时，任务仍按账号分组 / 账号池选择账号。代理作为账号运行环境进入风控：

- 代理不可达。
- 代理认证失败。
- 同代理账号异常聚集。
- 同代理账号 FloodWait 异常增加。
- 绑定账号数或并发连接超过建议上限。

代理告警达到阈值后进入风控中心处置队列。

### 7.5 资料初始化与账号安全动作拆分

TG 账号管理的批量入口必须按动作拆开，不能再用一个笼统的“安全加固”承载所有操作：

| 批量入口 | 默认动作 | 是否可混入其他动作 | 说明 |
| --- | --- | --- | --- |
| 资料初始化 | `update_profile`、`update_username`、`update_avatar` | 不混入安全动作 | 用于头像、昵称、简介、username 初始化；预览阶段不刷新登录设备安全状态 |
| 设置二步密码 | `set_two_fa` | 不混入资料或设备动作 | 只为未设置二步验证的账号生成并托管二步密码；已设置账号提示跳过 |
| 清理登录设备 | `cleanup_devices` | 不混入资料或 2FA 动作 | 只清理非本平台可信登录设备，保留平台当前 Session；新登录不足 24 小时按 TG 限制等待 |
| 同步安全状态 | 安全事实同步 | 不创建批次 | 逐账号读取平台可信设备、外部设备数量、2FA 状态和新登录限制，供后续判断使用 |

资料初始化是资料能力，不是安全加固能力。其预检只读取已有安全快照和账号在线状态，不应为了生成昵称、头像或 username 而触发实时设备扫描。设置二步密码和清理登录设备才进入账号安全批次，并写入安全快照、批次项、失败原因和审计记录。

账号批量动作入口采用动作优先：

- 运营人员可以先点击“资料初始化 / 设置二步密码 / 清理登录设备”，再在抽屉第一步选择账号。
- 抽屉内支持按账号组、状态、账号健康分、在线状态、2FA 状态、资料完整度、安全状态和最近批次状态筛选。
- 支持搜索手机号、昵称、username、账号 ID，支持跨页累计勾选和区间选择；当前账号列表已有勾选时只作为预填，进入抽屉后仍可继续增删。
- 三类动作确认时使用二次确认弹窗，展示动作名称、账号数量、可执行 / 跳过 / 不可执行数量、主要风险提示和操作原因；不再要求输入固定“确认加固”文案。
- “同步安全状态”不是安全加固动作，不清理设备、不设置二步密码、不修改资料，也不创建安全批次。

资料初始化的 AI 命名规则：

- 多账号预览必须一次请求 AI 生成整批账号资料，不允许按账号逐个请求模型。
- 前端和后端超时时间按账号数量伸缩；小批量保持快速，大批量允许模型完整返回。
- 默认命名风格是自然、随机、生活化的 TG 昵称 / 网名，例如“锅巴洋芋”“蕉太狼”“早睡失败”，不生成正式中文姓名或营销号名称。
- 界面提供命名风格提示 `custom_prompt`，运营人员可以直接输入本次昵称风格。
- 预览行需要展示生成来源和异常原因，例如 AI 成功、AI 超时、本地兜底、头像来源缺失、username 不可用或账号离线。
- 运营人员可以在确认前逐行编辑昵称、username、简介和头像策略；确认后按批次执行，批次详情需要能追踪每个账号的资料动作结果。
- 确认创建资料初始化批次后，后台 worker 执行状态必须同步进入任务中心，以“资料初始化批次”系统任务展示批次状态、账号进度、成功 / 跳过 / 失败数量、最近失败原因和可刷新详情；账号中心仍保留批次入口，任务中心负责执行可见性和追踪。
- AI 超时、mock、无健康供应商或返回不足时，使用本地随机网名兜底，并在预览行展示 warning。
- 勾选设置头像但没有可用头像来源时，只跳过头像并提示，不阻塞昵称、简介和 username 预览与执行。
- 随机头像包必须来自素材中心头像包；运营人员选择“随机头像包”后不需要再手填素材 ID 或平台路径。后端只允许使用已完成 TG 缓存的头像素材执行头像更新；素材仍在缓存中、FloodWait、失败或不可恢复时，对应账号头像步骤进入等待、跳过或失败状态，并在任务中心展示缓存进度和原因。
- 更新头像成功后必须回写账号头像对象和预览 URL，账号列表、账号详情、资料初始化结果和后续资料同步都必须能回显新头像；不能只显示批次成功而账号页面仍是旧头像或空头像。

## 8. 运营目标中心

运营目标是系统沉淀群、频道、讨论组、联系人和任务执行关系的底层业务对象。新版主流程不要求运营人员先到运营目标页创建目标；任务中心可以在创建任务时自动解析输入、创建或复用运营目标。运营目标页不再提供“新建群聊 / 新建频道目标”作为任务准备入口，主要负责管理、查看、同步、复盘和准入处置。

目标类型：

| 类型 | 能力 |
| --- | --- |
| 群 | 发送、监听、AI 活跃、转发源、转发目标、归档 |
| 频道 | 浏览、点赞、评论、回复、监听新消息、归档 |
| 讨论组 | 评论/回复承载、上下文采集 |
| 联系人 / 私聊 | 消息发送，默认不进入自动任务 |

目标详情需要展示：

- 目标基础信息。
- 授权状态。
- 可发送 / 可监听 / 可归档 / 可创建任务能力。
- 关联账号和账号能力。
- 账号关注 / 加入状态：已关注、未关注、关注中、关注失败、不可加入。
- 任务准入批次：由任务创建或任务启动触发的关注 / 加入前置结果、成功数、失败数和最近失败原因；重试与重新准入处理后续统一沉到运营目标页。
- 风险状态、命中原因和跳转风控中心入口；不在运营目标详情直接编辑风控策略。
- 是否正在作为全站目标画像学习来源；不在运营目标详情做画像版本、重建、清空或样本治理。
- 最近消息和上下文。
- 已绑定任务和运行状态。
- 数据统计和失败原因。

目标中心负责把 TG 资产整理成运营对象；执行层仍可使用群资产和频道消息表承载底层授权与上下文。目标中心不是任务创建前的必经步骤。

目标详情里的最近消息和账号覆盖可以服务于画像学习来源展示，但画像治理统一进入“目标画像”一级菜单。任务中心发送频控只保留全局风控与任务级每小时上限两类入口，任务自己的节奏和数量仍在任务中心编辑，不再读取运营目标页隐藏的每日上限或群冷却。

### 8.1 目标画像中心

目标画像是全站唯一 AI 学习资产，所有 AI 活跃群、频道评论和频道评论回复共同使用。它不属于某个运营目标；运营目标只是可选学习来源和任务执行目标。

目标画像中心需要展示：

- 当前画像：名称、版本、启用状态、最近重建时间、最近 AI 使用时间。
- 使用范围：固定展示 AI 活跃群、频道评论、频道评论回复。
- 学习来源：哪些群聊、频道评论区或讨论组被选为来源，哪些账号负责监听采集。
- 学习状态：自动同步新消息是否开启、最近同步时间、最近向上拉取时间、候选样本数、采纳 / 降权 / 剔除样本数。
- 样本质量规则：身份过滤、文本过滤、广告模板过滤、质量评分阈值、场景权重和禁学模式。
- 样本治理：按来源、发送人、时间和状态筛选，支持采纳、降权、剔除并填写原因。
- 历史学习：支持从当前水位向上拉取更多历史消息，展示起止水位、拉取数量、入库样本数和失败原因。
- 版本治理：重建、查看版本、恢复版本、清空画像，危险动作必须填写原因并写审计。

日常配置路径：

```text
打开目标画像
  -> 选择学习来源
  -> 选择每个来源的监听账号
  -> 配置样本质量过滤规则
  -> 同步新消息或向上拉取历史
  -> 查看候选样本
  -> 采纳 / 降权 / 剔除样本
  -> 重新计算候选样本
  -> 重建画像
  -> 确认生效版本和使用范围
```

学习来源规则：

- 群聊来源用于学习自然接话、短句节奏、语气、追问方式、常见话题和禁学内容。
- 频道评论来源用于学习真实读者短评、回复方式、对频道原文细节的引用习惯和讨论区语境。
- 来源选择支持按群 / 频道 / 讨论组、可监听状态、监听账号、最近消息时间和任务类型筛选。系统可以推荐正在运行 AI 活群或频道评论任务的目标，但不能自动勾选。
- 没有监听账号覆盖的来源允许保存，但必须标记为“不可自动同步”；历史拉取和自动同步失败必须展示失败原因，不能静默删除来源。
- 自动同步不是静默吸收全部真人消息；系统先形成候选样本，再过滤机器人、托管账号、自身账号、按钮提示、模板广告、重复文案和低质量内容。
- 样本质量规则由运营人员配置，规则变更只影响候选样本计算；必须显式重新计算候选并重建画像后，才影响 AI 活跃群和频道评论。
- 学习来源选择和监听账号选择都在目标画像中心完成；运营目标详情只显示“正在作为学习来源 / 未作为学习来源”和跳转入口。
- 旧目标级画像数据放弃处理，不迁移、不合并、不兼容。新画像首次为空版本，运营人员配置来源并拉取样本后再形成有效画像。

### 8.2 任务内目标解析与账号准入

任务创建不能要求运营人员先创建目标、等待账号加入完成、再返回任务中心。正确流程是任务优先：

```text
创建任务
  -> 创建时选择已有目标，或粘贴群聊 / 频道入口
  -> 后端解析 target_input
  -> 自动创建或复用 operation_targets
  -> 按任务账号范围检查账号-目标准入关系
  -> 已满足账号立即进入主任务动作
  -> 未关注 / 未加入账号进入准入子任务
  -> 准入成功账号追加进入主任务容量
  -> 失败账号记录原因并从主互动中剔除
```

任务创建时允许录入，编辑任务时不再展示或提交新目标入口：

- 公开频道或公开群 `@username`。
- 公开链接。
- `https://t.me/+...` 或 `joinchat` 邀请链接。
- 已知 peer id。peer id 只能用于识别已同步目标，不能保证主动加入。

任务内准入规则：

- 运营人员在任务里选择“全部可用账号 / 指定账号分组 / 手动选择账号”作为候选账号范围。
- 对未关注频道、未加入群聊或已加入但不满足本任务发送 / 监听能力的账号，系统生成统一准入前置动作。
- 频道任务的 `ready` 表示已关注；转发监听源群的 `ready` 表示已加入 / 可读取；AI 活跃群和转发目标群的 `ready` 必须同时满足已加入且可发言，`can_send=True` 才能进入主互动。
- 准入候选必须来自任务账号配置选中的全部在线账号；`max_concurrent`、每轮发言数、发送容量、账号冷却和健康权重只影响主互动选择与节奏，不能截断准入动作生成。
- 权限探测必须区分缺字段、账号不在群、群默认禁言、单账号禁言和实际发送 API 失败。普通 supergroup 成员缺少 `send_messages` 字段时不能直接写成 `can_send=False`。
- 准入前置在任务详情中作为可见子任务展示，包含执行状态、预计进度、预计完成时间、账号级结果和失败原因。
- 准入前置不阻塞已经满足条件的账号执行主任务；已满足账号先进入主互动，后续加入 / 关注成功的账号再补充进入本任务可用容量。
- 准入成功后写入账号-目标关系，本任务继续使用，后续任务也可复用。
- 失败账号记录失败原因，不进入浏览、点赞、评论、回复、群发言、监听或转发等主互动；已加入但不可发言的账号在 AI 活跃群和转发目标群中仍视为未满足，需要重新准入到可发言状态。
- 机器人 / 图形验证码诊断只在当前群成员、最近消息或验证 action 仍能证明时展示；证据消失后，账号详情、任务详情和运营异常读模型必须刷新为最新目标能力。
- 如果目标只有 peer id 且没有可加入链接，系统只能标记为“待同步 / 需已有账号发现”，不能承诺主动加入。

任务预检不应把“未关注 / 未加入”当成阻塞；它应该告诉运营人员：

- 已满足本任务准入账号数：频道已关注、转发源群已加入 / 可读取、AI 活跃群和转发目标群已加入且可发言。
- 未关注、未加入或已加入但不可发言，且可自动准备账号数。
- 不可加入或受限账号数。
- 预计需要多少关注 / 加入前置动作。
- 如果立即启动，已满足账号会先进入主互动，未满足账号会先进入准入子任务。

如果已经存在满足准入条件的账号，预检和详情必须明确“主任务可先执行”。此时准入子任务只负责补齐剩余账号，不应把已满足账号的浏览、点赞、评论、回复、群发言或监听动作整体挂起。只有 0 个账号满足且 0 个账号准入成功时，主互动才不能开始。

### 8.3 入群验证处理

群聊加入可能遇到入群问题、简单算数题、固定问答、人工审批或验证码。平台处理口径：

- 文本问题、简单算数题、固定问答由 AI 自动尝试回答。
- 每个账号对同一个目标最多自动尝试一次，避免反复触发风控。
- AI 无法判断、图片验证码、需要人工审批或 TG 拒绝加入时，标记为 `manual_required`、`challenge_failed` 或 `failed`。
- 验证问题、AI 答案、结果和失败原因必须写入准入 action result 和任务统计。
- 验证失败账号不进入主互动，但不阻塞其他成功账号继续任务。

## 9. 消息发送中心

消息发送中心是手动发送和小批量发送入口，不承担运营方案管理，也不替代任务中心执行详情。它适合运营人员明确选择账号、联系人或运营目标后发送一批消息。

消息发送负责：

- 选择发送账号、账号分组或手动账号范围。
- 选择目标：运营目标、账号联系人或手动对象。
- 输入文本、选择素材中心资产、保存草稿。
- 提交前执行账号可用性、目标能力、规则和风控预检。
- 创建发送任务、批量发送任务，并查看发送记录。
- 对发送记录执行取消、重试、派发和失败原因查看。

消息发送不负责：

- 不维护素材资产；素材上传、批量上传、分组、头像包和表情包库回到素材中心。
- 不维护长期运营方案；AI 活跃群、频道互动和转发监听方案回到运营中心。
- 不展开大量执行明细；需要追溯 action / attempt 时深链任务中心或发送详情。

发送失败也必须进入运营异常闭环。首期可以保留 `message_tasks` / `message_task_attempts` 作为手动发送事实源；Metrics / Recovery 必须把失败按目标、账号和失败类型上卷为 `operation_issue`，并在来源表里记录 `source_type=message_task`。如果后续统一迁移到 Task / Action，也要保持旧发送记录可追溯。

## 10. 规则中心

规则中心维护系统级规则集，任务绑定已发布规则版本。

规则集承载：

- 输入过滤：关键词、敏感词、白名单词、链接白名单、来源条件。
- 输出校验：生成内容是否命中禁词、是否过长、是否像模板噪声、是否包含运营人员内部话术。
- 内容转换：轻量改写、去来源、保留来源、格式化、素材处理。
- 路由规则：按源群、关键词、目标类型、权重分配目标。
- 账号策略：固定账号、轮询、权重、目标账号映射。
- 限速策略：任务级、目标级、账号级、任务类型级。
- 重试策略：最大重试、退避、失败后跳过或暂停。

规则中心原则：

- 任务只能绑定已发布版本。
- 草稿版本不能被运行任务绑定。
- 关键词不再作为独立产品中心出现，它只是规则条件的一种。
- 规则测试器必须能预览过滤、转换、路由、账号策略和限速结果。
- 规则命中需要能回溯到规则集版本、任务、目标、账号和执行项。

## 11. 风控中心

风控中心负责运行硬边界和处置闭环。

风控职责：

- 全局发送节奏。
- 默认失败处理策略。
- 账号小时 / 日上限。
- 账号全局冷却。
- 目标群 / 频道发送窗口。
- 群日上限和群冷却。
- 代理健康和代理告警。
- FloodWait、SlowMode、账号受限、权限不足、目标权限、任务配置和内容拦截处置。
- 账号健康分、风险等级、扣分原因和处置队列。
- 命中记录和策略审计。

代理告警支持标记处理中、临时忽略和恢复。临时忽略必须填写原因和到期时间，只暂停运营提醒，不删除告警事实、命中记录或策略审计。

账号健康分只表达账号自身和账号运行环境是否适合继续参与任务。目标、权限、任务配置和内容规则失败不扣账号健康分，只作为命中记录、处置队列或目标能力异常展示；只有账号本体、账号运行环境和账号级限流才允许扣账号健康分。

策略审计只展示与风控运行策略直接相关的留痕：全局风控策略、代理资源、代理告警处置、账号代理绑定和批量绑定。普通任务审计仍回到审计记录页；风控中心内的策略审计必须保留操作人、动作、对象、原因 / 变更详情、时间和 `trace_id`。

执行准入链路：

```text
Action
  -> 账号可用性
  -> 代理健康
  -> 目标权限
  -> 目标级风控
  -> 规则中心过滤 / 转换 / 路由
  -> Redis 多维限流
  -> Telegram Gateway
```

失败处置和健康分口径：

| 失败类型 | 影响对象 | 是否扣账号健康分 | 默认处置 |
| --- | --- | --- | --- |
| FloodWait | 账号 | 是 | 写入账号冷却，延后 action |
| 账号受限 / 疑似封禁 | 账号 | 是 | 暂停账号派单，进入处置队列 |
| Session 失效 / 登录不可用 | 账号 | 是 | 转派或跳过，跳转账号中心重新登录 |
| 代理不可达 / 认证失败 | 代理 / 账号运行环境 | 是 | 进入代理告警，必要时阻塞绑定账号 |
| SlowMode | 目标 | 否 | 写入目标冷却，延后 action |
| 无评论权限 / 群限制发言 | 目标能力 | 否 | 记录目标权限异常，跳转运营目标或账号详情重查目标能力 |
| 目标无权限 / 未授权 | 目标 | 否 | 拦截并提示重新授权 |
| task_deleted / 任务配置错误 | 任务 | 否 | 记录任务失败事实，跳转任务详情处理 |
| 内容拦截 | 内容 / 规则 | 否 | 跳过、改写重试或暂停任务 |
| 未知错误 / 待分类事件 | 任务 / 系统运行 | 默认否 | 展示原始错误、trace_id 和待分类状态，不直接扣账号健康分 |
| unknown_after_send | 执行项 | 否，除非归因为账号侧失败 | 进入人工确认或补偿查询，不自动重发 |

风控中心总览卡片必须可点击下钻。可用账号、降频账号和阻塞账号进入账号健康分 Tab 并带筛选条件；待处理处置项进入处置队列；最近 FloodWait 和代理告警进入命中记录或代理资源。账号健康分行内的“账号中心处理”必须带 `account_id`、默认 Tab、`return_to=risk-control` 和来源原因跳转到账号中心，按钮列不得被截断。

## 12. 素材中心与 AI 内容边界

素材中心是独立一级运营资源，不放在系统设置里。系统设置只保留素材运行配置，例如缓存账号、缓存会话、上传大小限制、临时文件 TTL 和缓存队列参数。素材中心只维护可复用素材资产、版本、缓存健康和引用关系，素材库从“URL / 文本配置”增强为可复用基础能力：

- 实时上传素材。
- 批量上传表情包、头像包、图片和文件。
- 支持直接上传 `.zip` 压缩包批量导入图片素材，包内只接受 `.png`、`.jpg`、`.jpeg`，单张图片不超过 500KB。
- zip 导入时先选择导入类型：表情包分组、头像包或普通图片分组；默认可用压缩包文件名作为分组 / 素材包名称，导入前允许修改。
- zip 只是批量导入容器，不作为可发送文件素材；包内非图片、隐藏目录、`__MACOSX`、超过 500KB、重复文件或 MIME 校验失败的文件必须跳过并在导入结果中展示原因。
- 导入结果必须展示包名、解析总数、成功数、失败数、跳过数、重复数、超过 500KB 数、生成的分组 / 素材包和逐文件失败原因。
- 维护素材类型、大小、用途和标签。
- 维护素材分组、头像素材包、表情包分组和引用关系。
- 缓存 Telegram 媒体引用。
- 支持消息发送、转发监听和 AI 活跃群共用。
- 支持账号资料初始化从头像包随机或按分组选择头像。
- 支持源媒体下载再上传，不假设不同账号可直接复用 Telegram 文件引用。

AI 内容能力不属于素材中心日常管理。AI 供应商、提示词和黑话配置归系统设置；AI 活跃群接话 / 暖场策略归运营方案；输出质量校验归规则中心和风控中心。素材中心只提供 AI 可引用的图片、表情包、头像包、文件和组合消息资产。

AI 内容能力负责：

- 群上下文摘要。
- 读取全站唯一目标画像，为 AI 活跃群、频道评论和频道评论回复提供共同的风格底色。
- AI 活跃群多账号对话生成。
- AI 活跃群接话 / 暖场模式判断。
- AI 回复事实锚点、语义去重、幻觉风险和沉默原因记录。
- 频道评论生成。
- 回复生成。
- 转发改写。
- 内容自动校验。
- 账号画像和账号语气记忆。

AI 输出必须经过规则中心和风控中心校验，不能把内部提示词、后台术语、审核词或操作员话术发送到公开群。全站目标画像只提供语气、句式、话题倾向和读者口吻，不提供具体事实。AI 活跃群不能为了凑活跃度凭空编造“上次体验”“位置确认”“回访”“准点”等没有上下文或素材锚点的事实。频道 AI 评论同样不能用“参考价值、值得讨论、支持一下”等泛化模板补量；候选评论必须贴频道原文的具体词、数字、物品、场景或问题，语义重复或模板化时直接跳过并留痕。

## 13. 任务中心

任务中心统一使用 Task / Action 模型，并允许承载少量系统执行任务的只读投影。

```text
Task
  表示一个长期或短期运营任务。

Action
  表示某次具体执行动作，例如发送、浏览、点赞、评论、回复。

System Task Projection
  表示由其他业务批次驱动、但需要在任务中心展示执行状态的后台任务，例如账号资料初始化批次。
```

任务通用状态：

| 状态 | 含义 |
| --- | --- |
| draft | 草稿 |
| running | 运行中 |
| paused | 暂停 |
| completed | 完成 |
| failed | 失败 |
| stopped | 人工停止 |
| deleted | 删除归档 |

等待执行由 `actions.pending` 表达，不作为 Task 主状态。部分成功、部分失败是运行摘要或准入子任务状态，不进入 `tasks.status` 主枚举。

任务中心必须同时展示主状态和派生运行阶段。主状态回答调度是否允许继续，派生阶段回答任务当前为什么没有发、正在等什么、谁需要处理。

| 主状态 / 条件 | 派生阶段 | 页面展示 |
| --- | --- | --- |
| `paused` | 已暂停 | 高强调状态条：不会继续规划或执行；展示暂停来源、暂停时间、最后错误、继续入口 |
| `running` + 首轮校验未完成 | 启动校验中 | 展示目标、账号、规则、AI、风控校验进度 |
| `running` + 准入 pending / running | 准入补齐中 | 展示已满足、待准备、成功、失败、预计完成；说明已满足账号可先执行 |
| `running` + AI timeout / no healthy provider | 等待 AI | 展示供应商、模型、错误类型和下次重试 |
| `running` + 上下文不足或质量跳过 | 等待上下文 | 展示最近上下文、沉默原因和下一触发条件 |
| `running` + slowmode / cooldown / next_run_at future | 等待冷却 / 下一轮 | 展示冷却对象、剩余时间和下次运行 |
| `running` + due actions executing | 发送中 | 展示执行中 action 数、账号、目标和最早租约 |

资料初始化、设置二步密码、清理登录设备仍由账号中心发起并写入账号维护批次表；任务中心不得把它们做成普通运营任务创建入口。但资料初始化批次一旦确认执行，必须在任务中心列表和详情中可见，用批次 ID 作为稳定任务 ID，展示后台 worker 进度、头像素材缓存进度、账号级执行结果和失败原因。

系统任务投影的实现口径：

- 资料初始化批次的任务 ID 固定为 `account_security_batch:{batch_id}`。
- 任务类型使用 `account_profile_init`，任务名称使用“资料初始化批次 #{batch_id}”。
- 任务中心列表展示进度和失败事实；详情展示 `profile_batch`，包含账号项、资料 / username / 头像分步状态、头像缓存状态、更新后的头像预览和失败原因。
- 系统任务投影只读，不展示启动、暂停、停止、重置、删除等普通任务控制按钮。
- 头像素材缓存未 ready 时，头像步骤保持等待或按规则跳过，不能把批次整体误标为成功。

任务操作：

- 启动：进入运行状态，开始规划 Action。
- 暂停：暂停新规划和新执行，保留状态。
- 继续：从暂停恢复。
- 停止：结束任务，跳过未执行项。
- 重试：重试失败项。
- 重置：清空未完成计划，重新规划；属于调度控制动作，需要 `tasks.dispatch_control`、二次确认和操作原因。
- 删除：隐藏或软删除任务，保留审计。

任务详情必须补充“准入前置”区域，用于展示任务内加入群聊 / 关注频道的子任务情况：

| 字段 | 含义 |
| --- | --- |
| 子任务类型 | `target_membership`，覆盖关注频道和加入群聊 |
| 子任务状态 | `not_required`、`pending`、`running`、`partial_success`、`blocked`、`completed`、`failed` |
| 账号容量 | 已满足、待准备、准备中、成功、失败、不可准备 |
| 预计进度 | 按准入 action 总数、已完成数、退避等待和 FloodWait 估算百分比 |
| 预计完成 | 展示预计剩余时间或预计完成时间；无法估算时展示原因 |
| 当前阶段 | 排队中、加入 / 关注中、等待 FloodWait、等待 AI 回答验证、等待人工处理 |
| 账号明细 | 每个账号的准入状态、挑战问题、AI 答案、是否可发言和失败原因；任务详情只展示，不在此处处理重试 |

准入子任务是主任务的前置补齐视图，不是全局串行锁。任务详情需要同时展示两条容量：

- 已满足账号：立即进入主互动并产生主 action。
- 补齐中账号：在准入子任务里按抖动和风控执行；成功后追加进入后续主 action，失败则只影响该账号。

任务列表可以展示简化摘要，例如“主任务执行中，准入 3/10，预计 8 分钟补齐”。任务详情则展示完整进度、账号级结果和失败原因。

### 13.1 创建体验

任务创建页应采用“快速创建 + 高级设置”。

快速创建只要求：

- 任务类型。
- 目标：可选择已有运营目标，也可直接粘贴群聊 / 频道 `@username`、公开链接或邀请链接。
- 账号范围。
- 任务结束时间。
- 业务意图，例如话题方向、评论方向、转发处理方式。
- 创建前预检。

高级设置承载：

- 账号参与策略。
- 24 小时活跃曲线。
- AI 生成与规则集选择。
- 内容过滤与改写。
- 失败处理和重试。
- 审计和告警。

默认配置必须能直接运行，不能要求运营人员先理解抖动、爬坡、静默比例、冷却和退避。

任务创建向导固定为 5 步：

1. 基础信息：任务类型、任务名称、结束时间、来源方案。
2. 目标来源：选择已有运营目标，或粘贴群 / 频道入口并由后端解析创建或复用目标。
3. 任务配置：按任务类型展示字段，AI 活跃群、转发监听、频道浏览、频道点赞、频道评论 / 回复各自独立。
4. 账号与节奏：账号范围、24 小时活跃曲线、低频 / 高峰阈值、最大并发、冷却和失败策略。
5. 预检确认：目标解析、账号可用性、准入前置、目标能力、预计 action、规则版本、风控命中和阻塞项。

频道任务默认使用 `dynamic_new` 新消息范围；`specific` 指定消息只作为从运营目标详情进入任务创建的快捷入口，不能回到手工登记单条频道消息的旧主流程。

### 13.2 24 小时活跃曲线

任务级节奏从多个零散字段收敛为一条 24 小时活跃曲线。每小时一个强度点，取值 0-100。

强度含义：

| 强度 | 含义 |
| --- | --- |
| 0 | 休眠，只保留监听和状态检查 |
| 1-20 | 低频 / 静默 |
| 21-60 | 常规 |
| 61-100 | 高峰 |

曲线只表达计划强度，不直接承诺发送量。实际动作量还必须经过账号容量、目标能力、规则和风控检查。

AI 活跃群和频道评论 / 回复统一采用“小时总预算为硬上限，轮次 / 单条目标 / 参与比例只做分配”的数量模型。任务每小时最大发送量和账号小时上限是真正的发送上限；AI 活跃群每轮计划发言数、频道评论每条消息累计目标、参与账号比例和抖动只决定本小时预算如何拆到轮次、消息、账号和时间。抖动不得突破小时上限，也不得通过参与比例反向增加本轮发言总量。

小时计划量建议公式：

```text
小时计划量
  = 任务类型默认基准量
  * 曲线强度比例
  * 可用账号系数
  * 风控系数
  * 目标能力系数
```

### 13.3 预检确认

创建前必须展示：

- 可用账号数。
- 目标能力。
- 预计动作量。
- AI 活跃群 / 频道评论的动态推荐值：任务每小时最大发送量、AI 活跃群每轮计划发言数、频道评论每条消息累计目标、每号每小时评论上限。
- 当前配置值和推荐值差异；如果用户已手动修改数量字段，账号范围变化后只提示新推荐，不静默覆盖。
- 容量缺口。
- 容量口径：目标每条、候选账号、有效账号、最大并发和缺口；`max_concurrent` 只控制同时执行数量，不截断频道浏览 / 点赞 / 评论的本轮账号池。
- 规则版本。
- 风控命中。
- 可能导致任务无法执行的阻塞项。

预检只是提示和减少无效提交，worker 执行前必须重新校验。

任务预检必须区分目标解析和账号准入：

- 目标是否可解析、可创建或可复用。
- 账号范围内有哪些账号已满足本任务准入：频道已关注、源群已加入 / 可读取、AI 活跃群和转发目标群已加入且可发言。
- 哪些账号未关注、未加入或已加入但不可发言，且可自动准备。
- 哪些账号无法加入、受限、离线或缺少加入入口。
- 预计需要多少准入前置动作。

如果任务目标存在且有可准备账号，即使当前 0 个账号已满足准入，也允许保存和启动任务。启动后先进入准入前置阶段；准入成功账号进入主互动，准入失败账号不进入主互动。如果已有账号满足准入条件，这些账号不需要等待其他账号加入 / 关注 / 重新取得可发言能力，主任务应先执行；如果 0 个账号满足且 0 个账号准入成功，主互动阶段不能开始。

群聊和频道在产品口径上使用同一套“账号-目标关系”模型：任务创建时选择账号范围，系统在预检和执行前置阶段补齐关注 / 加入 / 可发言关系。区别只在主互动能力上：频道任务进入浏览、点赞、评论或回复；转发源群进入监听；AI 活跃群和转发目标群进入发言或转发发送。

## 14. 任务类型设计

### 14.1 频道互动任务

包括：

- 频道浏览。
- 频道点赞。
- 频道评论 / 回复。

核心设计：

- 运营人员可以在任务创建时通过 `@username`、公开频道链接、`https://t.me/+...` / `joinchat` 邀请链接或已知 peer id 指定频道目标；其中 peer id 更适合作为已同步目标识别，无法保证单独完成关注。
- 启动频道互动任务前，系统先检查候选账号对目标频道的关注状态；未关注账号自动生成 `ensure_target_membership` 前置阶段，覆盖本任务账号范围内所有需要补齐关注的候选账号。历史 `ensure_channel_membership` action 必须继续兼容展示和执行。
- 已关注账号直接标记为已满足前置条件；未关注账号按抖动、限速和风控节奏执行关注；关注失败账号记录失败原因，任务详情只展示失败明细，后续重试或重新准入统一沉到运营目标页。
- 主互动阶段只能使用已确认关注成功或原本已关注的账号。
- 已关注账号的浏览、点赞、评论或回复不等待未关注账号完成关注；未关注账号在准入子任务成功后再进入后续互动容量。
- 如果历史任务或同步已经沉淀过账号-目标关系，任务启动应复用已有关系，只为新选择且未关注的账号补齐前置。
- 默认持续监听新消息。
- 每条频道消息生成独立执行明细。
- 去重维度至少包含任务、频道消息、动作类型、账号。
- 评论和回复当前可由同一个 `channel_comment` 类型承载，payload 区分 `comment_mode`、`reply_to_message_id` 和 `reply_target_label`。
- 目标量必须经过账号容量和目标能力检查。
- 频道评论 / 回复的 `预计每条评论 / 回复` 是每条频道消息累计目标。Planner 每轮只补足已发送、待执行和执行中 action 之外的差额；多条频道消息同时命中时，先按任务每小时预算分配，再按单条缺口补计划。
- 评论任务创建页默认按可评论账号数动态推荐任务每小时最大发送量、每条消息累计目标和每号每小时评论上限。用户手动改过的字段只提示推荐差异，编辑已有任务必须点击应用或保存后才生效。
- 同一频道消息下优先不同账号评论；账号不足且目标量较高时，只允许跨时间窗口复用账号。回复模式和混合模式的回复计入同一条频道消息的评论 / 回复目标，不绕过小时上限。
- 关注频道前置阶段不能集中瞬时执行，必须使用随机账号顺序、批次间隔、单账号退避和 FloodWait 延后。
- 如果部分账号关注失败，默认使用成功关注账号继续执行主互动阶段，并在任务详情中展示容量缺口和失败账号明细；如果 0 个账号成功关注，则任务进入异常或等待处理状态。

### 14.2 AI 活跃群

AI 活跃群用于在授权群中进行多账号自然发言。产品定位不是定时批量生成“体验好评”，而是在真实群聊上下文里接话、追问、附和、补充和低频暖场。

核心设计：

- 按 Cycle / Turn 规划。
- 任务创建时可以直接粘贴群聊 `@username`、公开链接或邀请链接；系统自动创建或复用群聊运营目标。
- 未加入目标群或已加入但不可发言的候选账号先进入群聊准入前置，达到已加入且可发言后才进入 AI 发言主动作。
- 入群验证由 AI 尝试处理文本问题、简单算数题和固定问答，失败账号不进入发言。
- 准入准备覆盖全部选中在线账号，不受 `max_concurrent` 截断；`max_concurrent` 只限制本轮可同时进入主互动的账号数量。
- 群内验证码、机器人拦截、目标不可发言和单账号禁言必须分别落到不同 failure_type；没有当前 Telegram 证据时不能沿用旧“图形验证码未通过”文案。
- AI 生成超时、空响应或无健康供应商时，任务派生阶段进入“等待 AI”，不创建假发言 action，也不把原因归到账号或群权限。
- 支持账号角色、人设、近期账号记忆和长期账号画像。
- 每轮生成读取全站唯一目标画像。画像只影响表达风格、常见话题和追问方式；当前任务的目标群真人上下文仍是事实来源。
- Prompt 拼装必须分层传入目标群真人上下文、任务配置、全站目标画像、账号角色 / 记忆、规则与禁学约束；画像不能覆盖实时事实，也不能把学习样本里的具体人名、价格、地点当成当前群事实。
- 生成前汇总真人上下文、上一轮 AI 内容、近期 AI 已说话题和主线方向。
- 每轮先判断模式：有可接真人消息时进入接话模式；长期无人说话且任务允许时进入低频暖场模式；上下文不足、重复风险高或事实锚点不足时进入沉默模式。
- AI 活跃群创建页默认按可发言账号数动态推荐任务每小时最大发送量和每轮计划发言数；编辑已有任务只展示推荐差异，必须由用户应用或保存后才改变运行配置。
- `每轮计划发言数` 只决定本轮最多计划多少个 Turn，不是单账号额度，也不承担风控硬上限；它不再有 10 条这类固定上限。
- `参与账号比例` 是多轮滚动窗口内的账号覆盖目标。它不表示每一轮固定让对应比例账号参与，也不能反向抬高本轮发言数；本轮 Turn 数低时，只能在后续轮次继续补覆盖。
- 同轮默认优先一号一条。只有本轮 Turn 数超过可用账号数或明确允许重复时，才允许同号在同轮复用；跨轮复用受冷却轮数、账号小时上限和全局风控约束。
- 接话模式只能围绕最近 3-8 条真人消息展开，优先短句、追问、附和、吐槽和补充，不允许脱离当前话题自说自话。
- 暖场模式只能低频触发，最多少量账号抛轻量话题或延续任务主题，不允许编造具体经历、服务过程、位置、回访、准点、穿着等未被上下文或素材支持的事实。
- 每条候选消息必须带事实锚点：来自哪条真人消息、哪个话题、哪个素材或哪个账号画像。没有锚点的具体事实候选应被丢弃或改写成泛化追问 / 附和。
- 画像不可用或样本不足时，不允许用模板话补量；有实时上下文时可围绕实时上下文生成，实时上下文也不足时进入沉默并记录原因。
- 一次生成多账号对话时，账号角色必须分工：起哄型、追问型、补充型、降温型或观察型，避免多个托管账号用同一套句式连续复读。
- 发送前做内容过滤、语义重复过滤、幻觉风险检查、上下文过期检查和账号冷却。
- 不走人工审核。

去重和质量闸门：

- 重复判断不能只比对文字，应识别同一意图反复出现，例如“照片准 / 没照骗 / 真人没差”“态度稳 / 不催 / 不敷衍”“位置提前发 / 没绕路”等要按同一语义簇限频。
- 同一轮内不允许多个账号表达同一个结论；近 N 轮内同一语义簇达到阈值后，本轮应改为追问、换角度或沉默。
- 如果最近真人聊天正在讨论人名、价格、榜单、评价或具体问题，AI 必须优先围绕该对象接话，不能跳回模板化“上次那个老师”。
- 质量结果要写入 generation 和 action payload / result，至少包括 `chat_mode`、`anchor_message_ids`、`semantic_cluster`、`duplicate_risk`、`hallucination_risk`、`skip_reason` 和本轮生成来源。

频道评论 / 回复复用同一套质量方向，并读取同一份全站目标画像。频道原文、被回复评论和讨论区上下文是事实锚点，画像只影响读者口吻、短评长度、提问方式、追问方式和讨论倾向。Prompt 拼装必须分层传入频道原文事实、被回复评论事实、全站画像读者口吻、任务配置、规则与禁学约束。AI 候选必须像真实读者短评，优先接频道消息里的具体细节；泛泛表扬、收藏、值得讨论、支持一下、继续展开等模板壳句不用于补量。若本轮候选全部重复或模板化，Planner 不创建假评论 action，并在任务上记录质量跳过原因。

图片和表情包策略：

- 可从素材库选择。
- 不把图片作为所有发言的默认必选项。
- 媒体发送必须经过素材缓存和目标能力检查。

### 14.3 转发监听群

转发监听群用于监听源群事件，按规则过滤、转换、路由并发送到目标群。

核心设计：

- 源群可以有多个。
- 目标群可以有多个。
- 源群准入只要求监听账号已加入 / 可读取；目标群准入必须要求发送账号已加入且可发言，已加入但不可发言的账号不能生成目标发送 action。
- 规则集版本决定过滤、转换、路由、账号策略和限速。
- 源事件必须有唯一键，避免重复转发。
- 多目标转发时，一个源事件发到 3 个目标群，计为 3 条目标群发送量。
- 目标群发送量由 24 小时曲线和风控共同决定。
- 不走人工审核。

来源过滤配置：

- 屏蔽机器人消息：默认开启。旧任务未显式配置时也按开启处理，避免机器人消息继续触发转发。
- 不转发群主和管理员消息：默认关闭。开启后，源群群主和管理员发送的消息只记录来源事件，不生成目标群转发发送项。
- 来源不转发名单：默认空。运营人员可以从最近来源发言人中选择，也可以手动粘贴 `@username` / `sender_peer_id` / 昵称。
- 名单保存优先使用稳定的 `sender_peer_id`，其次使用 `@username`，昵称只作为无法识别稳定 ID 时的兜底；昵称匹配存在同名误伤风险，界面和方案中必须提示。
- 来源过滤属于转发任务配置，不是监听中心的固定全局规则；不同转发任务可以对同一个源群采用不同过滤策略。

## 15. 监听中心

监听中心负责源事件和上下文采集，不负责业务发送。

监听对象：

- 群。
- 频道。
- 讨论区。
- 评论树。

监听中心展示：

- 监听对象。
- 监听账号。
- 关联任务。
- 订阅事件类型。
- 待分发统计。
- 去重键统计。

- 最近错误。

- 采集失败和发送失败分开统计。

## 16. 执行中心

执行中心是面向 1000 账号规模的核心。

### 16.1 当前与目标执行架构

后台执行已具备五类 worker 的 role 化 drain 能力；`all` 角色只用于开发和兼容，不作为生产推荐。生产是否已稳定拆成多个独立 worker 进程，需要结合部署配置、运行中的容器 / 进程、心跳和指标继续确认。

生产 worker 容器健康检查必须绑定 worker 自身心跳，不能 curl backend API 端口。每个 role 循环启动时写入对应 `worker_heartbeats.process_type`，compose healthcheck 使用 `python -m app.worker_health --role $WORKER_ROLE` 校验最近心跳。

| Role | 职责 |
| --- | --- |
| Planner Worker | 扫描 running task，生成 pending action |
| Dispatcher Worker | 原子领取 pending action，执行 TG API，回写结果 |
| Listener Worker | 采集源事件和上下文，维护监听水位 |
| Recovery Worker | 恢复 claim 超时、lease 超时、worker 失联和结果未知 |
| Metrics Worker | 记录队列、延迟、限流、worker、账号和代理运行快照 |

### 16.2 Worker role 契约

| Role | 输入 | 允许写入 | 禁止事项 |
| --- | --- | --- | --- |
| planner | running tasks、规则、账号池、上下文 | actions、tasks.next_run_at、任务 stats | 禁止调用 TG Gateway |
| dispatcher | due pending actions | action result、execution attempts、账号运行状态、任务 stats | 禁止生成新业务 action |
| listener | listener source、源目标、监听账号 | 上下文消息、listener source state、唤醒任务 | 禁止发送业务消息 |
| recovery | claiming / executing / heartbeat 超时数据 | action 恢复状态、attempt 结果未知、任务错误摘要 | 禁止调用 TG |
| metrics | action、task、worker、账号、代理、Redis 指标 | runtime snapshots、聚合指标 | 禁止改变业务状态 |
| legacy | 旧 message task / campaign / operation task | 旧兼容表 | 禁止进入新版 Task Center 主流程 |

### 16.3 Action 状态机

主路径：

```text
pending
  -> claiming
  -> executing
  -> success / failed / skipped
```

恢复路径：

- `claiming -> pending`：claim 超时、运行资源不足、账号分片不匹配、Redis 限流、账号 in-flight 冲突。
- `executing -> unknown_after_send`：已进入 Telegram Gateway 调用边界，但本地结果未知。
- `executing -> failed`：没有进入 Gateway 调用边界，或明确失败。

`unknown_after_send` 不能自动重复发送，必须进入人工确认、补偿查询或明确失败处理。

### 16.4 Dispatcher claim

Dispatcher claim 必须两段式短事务：

```text
阶段 1：DB 短事务预领取
  SELECT due pending actions
  JOIN running tasks
  APPLY task priority / type weight / per-task quota / account shard
  FOR UPDATE SKIP LOCKED
  SET status = claiming, claim_owner, claim_token, claim_expires_at

阶段 2：事务外获取运行资源
  Redis token bucket
  account in-flight lock
  proxy / target / media quota

阶段 3：DB 短事务确认执行
  SET status = executing, lease_owner, lease_expires_at
```

Telegram API 调用期间不持有数据库事务。

### 16.5 限流与账号互斥

多 worker 限流使用 Redis token bucket，覆盖：

- 全局 TG API。
- 任务。
- 任务类型。
- 账号。
- 代理。
- 目标。
- 媒体发送。

Redis token bucket 可以先采用保守扣减语义。完整 reservation / confirm / refund 在 Dispatcher 并发稳定后再增强。

同一账号默认只能被一个 worker 同时使用。账号 in-flight 与 token bucket 分开处理：

- token bucket 管速率。
- in-flight lock 管同一账号是否正在执行。

DB 唯一索引可以作为最后兜底，但不应作为主要调度机制。

### 16.6 账号分片

账号分片必须在 claim 前生效。

原则：

- 账号转派只能在当前 shard 内完成。
- 当前 shard 无可用账号时 action 延后并记录原因。
- 不建议运行中频繁修改 `ACCOUNT_SHARD_TOTAL`。
- worker 故障接管需要明确是保持 shard 积压，还是由 standby worker 接管。

### 16.7 数据库连接池与 Telethon client

Dispatcher 并发必须受 PostgreSQL 连接池预算约束。

需要显式配置：

- `DB_POOL_SIZE`
- `DB_MAX_OVERFLOW`
- `DB_POOL_TIMEOUT`
- `DB_POOL_RECYCLE`

数据库压力治理先不采用分库分表。当前阶段以短事务、汇总读模型和冷热数据边界为主：

- 热写事实表只记录不可丢失的执行事实，例如 `tasks`、`actions`、`execution_attempts`、监听事件、上下文消息和网关结果。
- 运营中心首页、目标工作台、异常列表和数据卡片不直接扫描大体量明细表，默认读取目标级、任务级和日期级汇总读模型。
- 任务中心详情、Action 明细和执行尝试详情才允许按 `task_id`、`action_id`、`target_id`、时间范围下钻热写事实表。
- Metrics / Recovery / 清理任务负责把 action 结果、执行尝试、监听状态和账号状态增量汇总到运营读模型。
- PostgreSQL 唯一索引和必要查询索引用于兜底一致性和下钻查询，不把明细表全量聚合当作运营中心常规查询路径。
- 后续只有在单库连接池、索引、分区、保留周期和汇总读模型都无法支撑时，才评估分区表、读副本或分库分表。

Gateway 契约数据结构已收敛到 `backend/app/integrations/telegram/contracts.py`，Telegram 网关适配器集中在 `backend/app/integrations/telegram/` 包内，旧 `backend/app/gateways.py` 已删除。Telethon client 生命周期已从网关适配器中抽到 `backend/app/telethon_lifecycle.py`，由独立模块管理后台 event loop、client cache、idle 释放、LRU 上限和应用停止时的 disconnect。生产参数由环境变量控制：

- `TELETHON_CLIENT_CACHE_SIZE`：单进程最多缓存的 Telethon client 数。
- `TELETHON_CLIENT_IDLE_SECONDS`：client 空闲多久后释放。
- `TELETHON_CLIENT_CONNECT_TIMEOUT_SECONDS`：连接超时。
- `TELETHON_OPERATION_TIMEOUT_SECONDS`：同步业务入口等待异步 TG 调用的上限。

生命周期规则：

- 每个 worker 只应加载自身 shard 内账号；Dispatcher / Listener 默认不共享进程，也就不共享 client cache。
- client cache key 首期按 `api_id + session` 维度管理；session 失效、账号受限、代理切换时应调用 lifecycle invalidate 或由后续风控处置清理。
- FastAPI shutdown 会调用 Telethon lifecycle shutdown，尽量断开已缓存 client。
- 单 worker 的 client 数必须小于 cache size，并结合文件描述符、内存、代理出口和 TG 延迟做真实压测。

### 16.8 Metrics 快照

运营数据实时查询不能替代 Metrics Worker。Metrics 需要周期性记录运行快照。

首期指标：

- `actions.pending.count`
- `actions.claiming.count`
- `actions.executing.count`
- `actions.oldest_pending_age_seconds`
- `actions.claimed_per_minute`
- `actions.success_per_minute`
- `actions.failed_per_minute`
- `gateway.flood_wait.count`
- `gateway.slowmode.count`
- `worker.active.count`
- `worker.stale.count`
- `redis.token_limited.count`
- `db.pool_wait_ms.p95`

页面必须能回答：

- 当前是否积压。
- 积压在哪个 role。
- 哪个 worker 掉线。
- 哪个账号、代理、目标造成限流最多。

### 16.9 端到端执行与展示链路

PRD 必须把流程写到可验收粒度，不能只写模块名。当前主链路固定为：

```text
预检
  -> 创建任务 / upsert 目标
  -> 启动任务
  -> Planner 规划准入动作和主动作
  -> Dispatcher claim / 获取资源 / 调用 TG Gateway
  -> 回写 actions 和 execution_attempts
  -> Metrics / Recovery 更新汇总读模型和 operation_issue
  -> 运营中心读汇总
  -> 任务中心按 task_id / action_id 下钻明细
```

页面读写边界：

- 运营中心只读目标、任务、账号和异常汇总，默认按目标展示。
- 任务列表只读任务摘要和任务级汇总，不加载 action 明细。
- 任务详情先展示任务配置和准入前置，再分页加载 action；attempt 只有展开 action 时加载。
- 原始失败事实只在任务中心和审计 / 归档中追溯；运营中心展示可处理异常和建议动作。
- 任务列表、任务详情顶部和运营中心关联任务摘要必须使用同一派生阶段计算；不能一个页面显示运行中，另一个页面显示暂停或验证码旧错误。
- Recovery / Metrics 更新汇总读模型时要对比当前目标能力、账号能力、AI 健康和任务主状态；旧 `last_error` 与当前事实不一致时降级为历史记录，不再作为当前异常展示。
- `paused`、`stopped`、`failed` 属于硬停止类展示；`running` 下的等待 AI、等待上下文、等待冷却和准入补齐属于运行中子阶段，不能混成任务失败。

## 17. 归档、运营数据与审计

### 17.1 归档中心

归档中心负责群、频道和消息上下文的长期保存和检索。

能力：

- 群 / 频道归档。
- 消息检索。
- 成员快照。
- 归档任务状态。
- 与运营目标关联。

### 17.2 运营数据

运营数据用于复盘，不承载策略编辑。

数据维度：

- 账号数据：发送数、成功数、失败数、FloodWait、受限次数。
- 目标数据：发送量、互动量、失败量、容量不足次数。
- 任务数据：计划数、执行数、成功率、失败原因、积压。
- 规则数据：命中次数、转化、目标归因、账号归因。
- 素材数据：素材指纹、源事件、目标、成功率、失败和跳过。
- worker 数据：活跃、过期、处理速率、最老 pending。

运营数据必须优先读取汇总读模型，不直接把任务执行明细表当报表库使用。

建议汇总读模型：

| 读模型 | 用途 | 更新来源 |
| --- | --- | --- |
| `target_runtime_summary` | 运营中心目标工作台、目标异常、目标效果卡片 | action 结果、监听状态、账号状态、规则命中 |
| `task_runtime_summary` | 任务列表、任务健康、任务失败趋势 | tasks、actions、execution_attempts |
| `account_runtime_summary` | 账号可用性、容量、账号失败、FloodWait、安全批次待重试和受限趋势 | action 结果、账号状态、安全快照、账号安全批次、风控预检结果 |
| `operation_issue` | 运营中心按目标展示异常和建议动作 | 失败事实、规则拦截、AI 质量跳过、账号 / 目标能力异常 |
| `daily_runtime_stats` | 日维度复盘、长期趋势和清理后的留存数据 | 每日汇总任务 |

汇总读模型允许短延迟，默认接受分钟级更新。运营中心的实时感来自增量刷新和最近更新时间提示，而不是每次打开页面都重新扫描 `actions` 和 `execution_attempts`。

运营方案使用独立配置和运行记录，不混入系统设置：

| 对象 | 用途 |
| --- | --- |
| `operation_plan_templates` | 保存群活跃、频道互动、转发监听和账号使用策略 |
| `operation_plan_targets` | 记录方案覆盖的目标范围和目标预览结果 |
| `operation_plan_task_links` | 记录方案生成或调整过哪些任务 |
| `operation_plan_generation_runs` | 记录生成预览、生成任务、应用关联任务的结果、阻塞原因和 trace_id |

### 17.3 运营异常

运营异常是任务失败、账号异常、目标能力异常、规则拦截、AI 质量拦截和风控限制上卷到运营中心后的处理对象。第一版不要求分库分表，但不应让运营中心每次打开都实时扫描 `actions` 和 `execution_attempts`。默认做法是由 Metrics / Recovery / 清理任务增量维护 `operation_issue` 或等价汇总读模型，运营中心读取目标级异常；轻量处理在上下文弹窗 / 抽屉内完成，需要原始 action / attempt 明细时再深链到任务中心并保留返回上下文。

建议字段：

| 字段 | 含义 |
| --- | --- |
| `target_id` | 异常影响的运营目标 |
| `issue_type` | 账号、目标权限、准入、AI 质量、规则、风控、TG 限制、监听、容量等类型 |
| `severity` | info / warning / critical |
| `representative_task_id` | 代表性任务，用于快速定位 |
| `representative_action_id` | 代表性失败 action |
| `affected_task_count` | 受影响任务数 |
| `affected_account_count` | 受影响账号数 |
| `failure_type` | 失败码，例如 `COMMENT_UNAVAILABLE`、`PEER_INVALID`、`context_expired` |
| `failure_reason` | 运营人员可读原因 |
| `suggested_action` | 建议动作，例如重新登录、重新准入、检查讨论组、暂停任务、查看规则、调整方案 |
| `handling_mode` | modal / drawer / deep_link |
| `return_to` | 深链返回运营中心原目标、原 issue 和筛选位置 |
| `claimed_by` / `claimed_at` | 多人协作时的处理负责人 |
| `status` | open / acknowledged / resolved / ignored |

运营中心默认按目标展示异常，例如“太郎日记频道：频道评论失败 8 条，影响 2 个任务，主要原因 `COMMENT_UNAVAILABLE`”。点开目标异常后再展示关联任务失败、账号影响和操作入口。

`operation_issue` 主表只保留代表来源和计数，完整来源进入 `operation_issue_sources`，影响账号进入 `operation_issue_accounts`，避免高频失败把数组塞进单行。

任务中心仍保留完整失败事实：action payload、result、attempt、堆栈、原始错误、重试记录和调度状态。运营中心不能直接替代任务中心明细；它只承载运营人员需要处理的聚合异常。

### 17.4 数据保留与冷热边界

运行明细默认保留最近 5 个自然日。第 6 天先把第 1 天明细汇总到日统计，再滚动删除第 1 天明细。

冷热边界：

| 数据层 | 数据范围 | 查询入口 | 保留策略 |
| --- | --- | --- | --- |
| 热写事实 | `actions`、`execution_attempts`、监听事件、AI 上下文、网关结果 | 任务中心详情、失败下钻、Recovery | 近 5 天或按容量配置滚动保留 |
| 热读汇总 | 目标、任务、账号、失败类型、运营异常汇总 | 运营中心、运营数据、任务列表卡片 | 长期保留，允许重算 |
| 冷归档 | 历史明细、归档消息、成员快照、清理审计 | 归档中心、审计、人工追溯 | 按归档策略保存，不参与首页查询 |

运营中心默认只查热读汇总；任务中心详情按 ID 和时间范围查热写事实；归档中心处理冷数据检索。任何跨全量明细的统计必须先进入汇总任务或离线导出流程，不能挂在普通页面请求上。

长期保留：

- 任务维度汇总。
- 账号维度汇总。
- 目标维度汇总。
- 任务类型维度汇总。
- 日期和状态维度汇总。
- 清理审计。

`unknown_after_send` 和人工未处理项不应被普通保留周期清理，除非已经人工确认或归档。

### 17.5 审计记录

审计记录必须覆盖：

- 账号登录、恢复、禁用、代理切换。
- 目标授权、能力变更、账号覆盖编辑。
- 任务创建、编辑、启动、暂停、停止、删除、重试、重置；停止、删除、重置必须记录操作原因。
- 规则发布、回滚、复制、测试。
- 风控策略修改和处置。
- 数据导出。
- 权限变更。
- 手动 drain 或其他危险操作。

## 18. 前端页面规划

页面应以“可操作、可解释、可定位”为目标。PRD 中的“前端页面设计”是后续统一重构的验收基准；总设计文档只保留页面定位、主布局和跨页面关系。

前端统一采用：

- 左侧主导航 + 顶部平台状态 + 页面标题区 + 筛选区 + 摘要区 + 主内容区 + 抽屉 / 弹窗详情。
- 列表页默认读汇总或分页接口；详情页按对象 ID 下钻；运营中心、运营数据和任务列表不能为了展示摘要扫描执行明细大表。
- 每个页面必须覆盖 loading、empty、error、permission denied、partial success 和汇总延迟状态。
- 每个可见按钮必须有后端接口、前端本地行为或明确的只读说明；危险动作必须二次确认和审计。

- 任务中心展示完整执行细节、失败事实和调度控制，但不是运营人员发现失败的唯一入口。

## 19. 压测与容量报告

1000 账号能力必须通过压测固化，静态评估不能等同线上承载证明。

当前已新增可重复的 mock gateway 容量模型：

- 脚本：`backend/scripts/run_capacity_benchmark.py`
- JSON 输出：`reports/capacity/latest.json`
- Markdown 报告：`docs/capacity-report-100-300-1000.md`

该报告用于固化容量参数和验收口径，仍不能替代真实 PostgreSQL / Redis / TG API 延迟压测。

压测场景：

- 100 账号 / 5 任务。
- 300 账号 / 10 任务。
- 1000 账号 / 20-30 任务。
- TG Gateway mock 快速返回。
- TG Gateway mock 慢返回。
- FloodWait / SlowMode / 账号受限注入。
- Redis 不可用。
- DB 连接池紧张。
- worker 异常退出和重启。

容量报告必须输出：

- 推荐 worker 数。
- 推荐 `DISPATCHER_CONCURRENCY`。
- 推荐 `ACTION_CLAIM_LIMIT`。
- 推荐 `DB_POOL_SIZE / DB_MAX_OVERFLOW`。
- 每分钟处理量。
- oldest pending P95。
- `unknown_after_send` 数。
- 重复发送数，必须为 0。
- 单机容量边界。

## 20. 实施优先级

完整拆分、验收项和阻塞关系以 `docs/tg-ops-platform-prd-refactor-checklist.md` 为准。本文只保留总设计层面的落地顺序。

### P0：基线和口径收敛

- 保持新版主线，不让旧 Campaign、旧 review、旧多租户订阅口径回流。
- 菜单、权限、按钮、手册和接口文案统一到 PRD 目标口径；`/dashboard` 文案统一为运营中心，路由保持兼容。
- 系统设置不承载发送节奏、风控策略、运营方案、目标异常处理或素材日常资产管理。
- 前端操作手册必须同步运营中心、素材中心、运营方案模板、任务创建动态向导、账号资产与可用性、汇总延迟和导航升级。

### P1：汇总读模型与运营异常

- 建立目标、任务、账号三类运行摘要和运营异常模型。
- 任务执行、监听、恢复和指标 worker 增量写入摘要，不让运营中心首页实时扫描 action / attempt 明细。
- 运营中心默认读取目标摘要和 open issue；任务中心按任务、action、attempt 下钻事实。
- 汇总数据必须展示最近更新时间、stale 标记和刷新入口。
- 运营中心建议动作必须返回处理方式：弹窗、抽屉或深链；深链必须携带 `return_to`，返回后恢复原目标和原 issue。

### P2：账号资产与可用性中心

- 账号中心从粗账号列表升级为账号资产与维护中心，展示完整手机号、分组、登录状态、同步资产、资料 / 安全状态、可发送、可监听、可加入、可评论、可修改资料、可读取验证码和剩余容量。
- 账号分组只作为资源分类和选择范围，不承载联系人发送、人员发送、消息编辑、素材选择或运营方案。
- 账号安全、资料初始化、2FA、登录设备、代理、验证码和目标准入失败都要沉淀为可解释原因。
- 任务创建预检读取账号可用性摘要，展示已满足、可准备、不可准备三类账号容量。
- 手机号默认展示完整 `phone_number`；历史缺失时才使用 `phone_masked` 兼容兜底。

### P3：目标画像中心

- 目标画像作为一级菜单，不放在运营目标详情里；全站只有一份生效画像。
- 学习来源从运营目标选择，支持监听账号选择、自动同步、向上拉取历史、样本质量规则、样本治理和版本治理。
- AI 活跃群、频道评论和频道回复读取同一画像版本；画像只提供风格和读者口吻，不提供具体事实。
- 旧目标级画像数据放弃处理，不迁移、不合并、不兼容。

### P4：运营方案中心

- 运营方案 / 策略模板位于运营中心下半部分，不进入系统设置，不让任务中心承载策略管理。
- 补齐模板、目标覆盖、生成任务、生成记录和关联任务链接。
- 方案支持生成任务草稿、生成并启动、暂停、恢复、复制和应用到关联任务。
- 调整运行中任务前必须展示影响预览、二次确认并写审计。

### P5：运营中心重构

- 运营中心上半部分是目标工作台，默认按目标展示异常、失败任务、影响账号、最近失败和建议动作。
- 运营中心下半部分是运营方案 / 策略模板，展示方案状态、目标覆盖、关联任务和生成入口。
- 任务中心失败必须上卷到运营中心；点开目标后能看到关联任务失败、代表 action 和建议处理路径。
- 建议处理路径默认打开上下文弹窗 / 抽屉；只有复杂登录、批量账号维护、完整规则编辑、完整方案编辑和大量执行明细才深链跳转，并带返回上下文。
- 运营中心只做摘要、异常和行动入口，不替代任务中心执行详情。

### P6：任务中心收敛

- 任务中心列表读取任务级汇总，详情继续作为执行事实源。
- 任务详情补 action attempt 独立下钻，避免大任务一次性加载全部明细。
- 任务创建收敛为快速创建 + 高级设置折叠，5 类任务按类型动态展示字段、准入预览和确认页。
- 生产执行层继续确认 planner / dispatcher / listener / recovery / metrics 多 role worker、并发配额、Redis token bucket、token 预留 / 退款、跨进程账号 in-flight、容量面板和压测结论。

### P7：素材中心、系统设置、手册和最终验收

- 素材中心独立承载表情包库、头像包、图片 / 文件 / 组合消息素材、批量上传、zip 压缩包导入、分组标签、缓存健康和引用关系。
- 系统设置只维护 TG 开发者应用、AI 供应商、提示词、素材运行配置、后台账号和权限等平台底座能力。
- 权限矩阵、按钮显隐、后端写接口校验、敏感动作原因和审计记录一起验收。
- 回归验收覆盖后端 pytest、前端 build、权限越权、汇总延迟、任务失败上卷、任务详情下钻和旧路由兼容。
- 前端操作手册作为验收入口之一，必须与真实菜单、按钮、弹窗、任务类型、异常速查、权限审计要求和异常提示同步。

## 21. 验收标准

系统设计验收：

- 用户侧主流程只围绕账号、运营目标、目标画像、任务、规则、风控和数据展开。
- 运营人员日常入口是运营中心；运营中心默认按目标展示异常，点开后看到关联任务失败和建议动作。
- 运营中心默认读取目标、任务、账号和运营异常汇总读模型，不把 `actions` 和 `execution_attempts` 明细表作为首页实时查询来源。
- 系统配置只维护平台底座能力，不承载具体运营方案、任务节奏或目标异常处理。
- 任务中心保留完整执行明细、失败事实和调度控制，不作为普通运营人员发现问题的唯一入口。
- 数据库压力治理先采用汇总读模型、短事务、索引、保留周期和冷热数据边界，不把分库分表作为当前阶段前置方案。
- 旧 Campaign、旧 review、旧订阅套餐等不进入新版主流程。
- 每个任务都能追踪到目标、账号、规则版本、执行项、结果和审计。
- 公开群消息不泄露后台配置、内部提示词、审核词或运营人员话术。
- AI 活跃群公开发言不能脱离真人上下文空转，不能重复同一语义簇，不能编造无锚点事实；低置信时必须跳过并记录原因。
- AI 活跃群准入准备覆盖任务选中的全部在线账号；主互动并发配置不能导致未入群账号长期不生成准入动作。
- 任务暂停、启动校验、准入补齐、等待 AI、等待上下文、等待冷却 / 下一轮和发送中必须在任务中心和运营中心清晰可见。
- 目标权限、单账号禁言、未入群、验证码 / 机器人拦截、AI 运行不可用和旧诊断必须拆分为不同展示原因。

执行层验收：

- 多 Dispatcher Worker 同时运行时，同一 action 不重复执行。
- 单 worker 内可并发执行，但同一账号不被并发滥用。
- Redis 不可用时 Dispatcher 不 fail-open。
- 已调用 TG 但本地结果未知的 action 进入 `unknown_after_send`，不自动重复发送。
- Listener 压力增大时不拖慢发送 action。
- 页面能看到队列积压、最老等待时间、worker 心跳和主要失败原因。
- 热写事实、热读汇总和冷归档边界清晰；运营中心读汇总，任务中心按 ID 下钻明细，归档中心处理历史检索。
- 压测报告能给出推荐 worker 数、并发数、claim limit、PostgreSQL 连接池和单机容量边界。

## 22. 总结

TG 运营管理平台的新版主线是：

```text
账号资源
  -> 运营目标
  -> 规则与风控
  -> 任务中心
  -> 执行中心
  -> 数据、归档、审计
```

当前系统已经具备可用的任务中心、监听中心、规则中心、风控基础、运营数据，以及 role 化 drain 的执行层。下一阶段重点不是继续叠加功能，而是确认生产多 worker 进程、并发配额、token 预留 / 退款、横向扩容、可观测、可恢复、账号锁、容量面板和压测口径，同时把任务创建体验从工程参数表单收敛为运营人员可理解的业务流程。
